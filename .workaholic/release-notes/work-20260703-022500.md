# Fix the passphrase prompt to key on the controlling terminal etc

## Summary

v0.0.15 fixes the v0.0.14 first-run regression where piping a credential on stdin disabled the vault passphrase prompt, and replaces the loopback-listener Google consent with gmail-ftp's terminal paste-back flow — so `qfs account add google` now works over plain SSH with no port-forward.

## Key Changes

- The vault passphrase prompt now reads the controlling terminal (`/dev/tty`), so `cat credentials.json | qfs app add google` prompts correctly on a terminal
- `qfs account add google` runs a paste-back browser consent: open the printed URL in your local browser, approve, paste the `http://localhost/...` redirect back — no listener, works over SSH
- Press `c` at the consent prompt to copy the URL to your local clipboard via OSC 52 (works across SSH and tmux)
- The pasted redirect's CSRF `state` is verified before any token exchange; one consent grants the Gmail + Drive + Analytics scope union

## Changes

### Changed
- Passphrase prompt gates on the controlling terminal instead of stdin, restoring every pipe-a-secret command on a terminal
- Google account authorization is now the paste-back browser consent; the loopback-listener flow is removed and the OAuth `redirect_uri` is the portless `http://localhost`
- README, guides, and the Gmail cookbook describe the paste-back steps
