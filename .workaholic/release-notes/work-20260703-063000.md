# Unlock the vault at scan time for cloud reads etc

## Summary

v0.0.17 delivers the two fixes the live onboarding loop surfaced: cloud reads on a terminal now prompt for the vault passphrase exactly when a query reads a cloud mount, and the consent prompt's copy/open choice acts on a single keypress. The full round-trip — one consent serving both `/mail` and `/drive` — is now live-verified.

## Key Changes

- A cloud read on a terminal without `QFS_PASSPHRASE` prompts for the passphrase at scan time (once per process) instead of failing with a misleading "no usable account" for an account sealed behind the locked vault
- Headless with no passphrase now gets an honest locked-store hint naming the export recipe; previews and local reads still never prompt
- Press `c` at the consent prompt to copy — no Enter needed, and the key never appears on the prompt line (`o` to open a browser likewise)

## Changes

### Changed
- Cloud mount binding is deferred to scan time with a one-time terminal passphrase prompt; locked-vault and missing-account errors are now distinct
- The consent `c`/`o` choice reads a single unechoed keypress via a raw-mode terminal read, restored immediately after
