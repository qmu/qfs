# v0.0.21: Self-heal a loose credential store to 0600 on open (fix the v0.0.20 upgrade friction)

## Summary

v0.0.21 fixes the v0.0.20 upgrade friction: a credential DB or secrets blob created world/group-readable
under a pre-v0.0.20 umask is now **self-healed** — tightened to `0600` in place on open — instead of
refusing to open, which bricked the CLI until the user manually `chmod`'d each store.

## Key Changes

- A loose-but-**owned** credential store/blob is tightened to `0600` on open and used, rather than
  rejected — a pre-v0.0.20 `644` store heals on the next command. A file that cannot be tightened
  (foreign owner → `EPERM`, or a mode-ignoring filesystem) is still refused with the `chmod 600`
  remedy. The guard only ever tightens, never loosens.
- The embedded-DB migration runner is now concurrency-safe: the pending-migration apply uses an
  `IMMEDIATE` write-locking transaction that re-checks under the lock, with a `busy_timeout`, so
  concurrent opens of one DB no longer race the `schema_version` check-then-insert.

## Changes

### Changed

- Auto-tighten a loose-but-owned credential store/blob to `0600` on open (self-heal) instead of
  refusing (`store/fs_perms.rs`, `secrets/local.rs`).
- Concurrency-safe migration runner (`IMMEDIATE` re-check + `busy_timeout`), fixing a flaky
  `UNIQUE constraint` crash under concurrent DB opens.
