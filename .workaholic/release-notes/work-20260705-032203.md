# Implement the blueprint §13 self-hosting-integrations feature etc

## Summary

This release delivers the entire §13 self-hosting-integrations feature: an ordinary qfs script can now declare a driver, install it to `/sys/drivers`, and connect it as a live, host-confined wire mount — no compiled Rust. It also converges the `/sql` run/describe registries onto `qfs connect` and fixes a CI-blocking test race.

## Key Changes

- Full §13 self-hosting integrations: the declaration surface (`CREATE DRIVER/TYPE/VIEW/MAP` → `/sys/drivers`), an evaluator that turns those rows into a live wire mount, and a public conformance check, proven by converting the compiled Slack driver into its own `slack.qfs` script twin.
- A security-critical three-layer host-confinement boundary pins each declared driver to its own host at the single `send_one` dispatch chokepoint, so a generated script cannot read one service and silently write to another.
- `/sql` run and describe now converge on the canonical `qfs connect` binding: a `qfs connect /sql/<conn>` connection wires both the runtime driver and the describe mount, and `describe /sql/<conn>/<table>` reflects the live catalog.
- Fixed a CI-blocking race in the env-mutating tests (one non-poisoning lock + a fresh `XDG_CONFIG_HOME` per test).

## Changes

### Added
- Declared-driver declaration surface (`CREATE DRIVER/TYPE/VIEW/MAP` desugaring to `/sys/drivers`, migration 14).
- Declared-driver evaluator: two-source registry, live wire read/write, and three-layer host confinement.
- Conformance API (declared type vs delivered columns) and the first compiled→script conversion, `slack.qfs`.

### Changed
- `/sql` connection registries converged onto the canonical `qfs connect` (`path_binding`) source.
- Env-mutating tests serialized on one lock with a fresh config home.
