# Fire CREATE JOB rows with a real-clock sweeper; declared FOLLOW/ENCODE file transfer

## Summary

qfs v0.0.59 closes the three implementation gaps that blocked the capability-tryout mission's
owner-attended live rounds. The daemon now actually fires `CREATE JOB` rows — a 30-second
real-clock sweeper drives due jobs through the live committer (policy gate, irreversible guard,
real applier) with durable `last_run` tracking and an auditable read-only
`/server/jobs/<name>/runs` collection. Declared drivers gain two generic file-transfer
primitives — `FOLLOW <field>` for credential-free cross-host downloads and `ENCODE multipart`
for form-data uploads — which `chatwork.qfs` uses to declare file download and upload in the
query language itself. Every declared request now carries a versioned `User-Agent` header.

## Key Changes

- `qfs serve` fires due `CREATE JOB` rows on a 30s real-clock sweeper; `last_run` survives
  restarts; each firing appends to a read-only `/server/jobs/<name>/runs` collection (capped at
  50 runs, removed with the job row)
- Declared `FOLLOW <field>` pipe stage downloads a row's cross-host URL without forwarding the
  declared host's credential (the token never leaves the declared host)
- Declared `ENCODE multipart` between target and VALUES desugars onto the existing effect
  pipeline, backed by a generic `multipart/form-data` encoder — no service-specific code
- `chatwork.qfs` now declares the file blob view (FOLLOW download) and the multipart upload map
- Wire paths accept a `?query=` suffix in declared path templates
- Every declared driver request sends `User-Agent: qfs/<version>` (GitHub and similar services
  refuse agent-less requests)
- A new durable mission records the CREATE AGENT semantics design (agents as first-class
  principals) — design only, no code on this branch
