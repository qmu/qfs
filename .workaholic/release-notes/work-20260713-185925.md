# Give /fs and /drive single-file reads their content-column parity etc

## Summary

v0.0.61 clears the two contained blob-driver content-schema follow-ups from the v0.0.60 defect
campaign: `/fs` and `/drive` single-file reads now advertise and materialise a `content` column at
plan time, so `… |> select content |> transform` type-checks against them. It also proves the
server scheduler live end-to-end.

## Key Changes

- `/fs` gained the single-file `content` read and `describe` parity `/local` received in v0.0.60.
- The gdrive file-content and folder-listing read schemas were unified, so `/drive/<file> |> select content` type-checks at plan time (the round-5 struct bypass is no longer needed).
- Live round 9 proved the daemon scheduler: `qfs serve` fired a 1-minute JOB, `affected 1`, and history survived a restart.

## Changes

### Changed
- `/fs` (`driver-fs`) advertises a nullable `content` column and materialises a single file's bytes; directory listings carry a null `content`.
- gdrive (`/drive`) converges its file-content and folder-listing reads on one unified schema; a directly-addressed file now reports its canonical Drive metadata plus `content`.
