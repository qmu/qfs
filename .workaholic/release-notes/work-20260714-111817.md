# ls /type is SHOW TYPES: the shell face lands, and the effect selector goes uniform

## Summary

qfs v0.0.70 completes the shell face of the typed-path space. The docs already promised a shell the binary could not run — `ls /type`, `describe` in the REPL, `cd` into a catalog. This release makes those true, closes a mail `mv` trap that silently sent messages, and finishes the effect-selector channel underneath.

## Key Changes

- `ls /type` now **is** SHOW TYPES: `/type` mounts as a read-only catalog, and `describe` works inside the interactive shell — where it can address the current directory, which the one-shot form cannot.
- `cd` now enters what it should: definition catalogs (`/transform`, `/type`), a SQL connection's catalog, and the mail label tree. Entering a row-set is still refused, now with an error that says why.
- `mv` on a mail path is refused instead of silently sending. It previously lowered to copy+delete — a new message to a third party, and the original trashed. It now names the honest spelling: `UPDATE … SET labels`.
- A filtered `UPDATE`/`REMOVE` honours the `WHERE` you actually wrote. Filters no longer ride the write payload, so a non-key filter is respected instead of ignored, and `SET id = 2 WHERE id = 1` works for the first time.

## Changes

### Added
- `of <type>`: a mid-pipe type assertion that checks a relation against a declared type at any stage and never coerces.
- `/type` catalog mount and a `describe` REPL builtin.

### Changed
- `ls` reads each path by its entry kind, fixing `unknown column` errors on `/mail/inbox` and `/transform`.
- `cd` admits a node when its children are locations, a fact each driver now states per path.
- `cp` picks its verb from the destination (blob upserts, everything else inserts); `mv` is same-kind-only.
- Copying data into a definition catalog is now a clear category error rather than an obscure schema mismatch.
