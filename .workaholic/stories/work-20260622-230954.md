# Branch Story: work-20260622-230954

## 1. Overview

This branch turns the repository from the legacy Go **gmail-ftp** tool into **qfs** — a single Rust
binary that exposes every external service (mail, files, databases, GitHub, Slack, git, object
storage, Google Analytics) through **one uniform, filesystem-shaped, pipe-SQL query language**, with
a preview-then-commit safety model designed so an AI agent can drive every service from one small
grammar.

The core was built by a three-agent **trip** (`cfs-foundation-e0`: Planner / Architect /
Constructor) that designed, implemented, reviewed, and QA'd all 41 RFD-0001 tickets through full
three-gate cycles (**1240 tests green**). The remainder of the branch is the post-trip work that
took that foundation from "code complete" to "a shipped product with docs and a release."

Base: `main` (gmail-ftp v1.1). **213 commits.**

## 2. The journey

1. **The trip (t01–t41).** Language core (lexer, grammar, types, name resolution, evaluator,
   stdlib), effect-plan + PREVIEW/COMMIT runtime, federation + pushdown + local combine engine,
   codec registry, the driver contract, twelve drivers (local fs, SQL, HTTP, Google OAuth/Gmail/
   Drive/Analytics, S3/R2, Cloudflare, GitHub, Slack, git), the credential store, the CLI
   (interactive shell + one-shot), the E7 server (runtime, bindings DDL, HTTP endpoints, scheduler,
   watchtower, default-deny POLICY engine), deployment host adapters, the security threat model, an
   offline test harness with golden/anti-drift infrastructure, the embedded AI skill, and generated
   docs + distribution. Accepted as **TRIP COMPLETE**.

2. **Backlog tidy-up.** Archived all 41 completed tickets out of `todo/` into the branch archive,
   each stamped with its acceptance commit.

3. **Rename + restructure.** Renamed the project `cfs → qfs` everywhere (binary, 37 crate names,
   `CFS_*` env vars, the `.cfs` DSL extension, docs, RFD/ADRs, trip history) and moved the whole
   Cargo workspace under `packages/qfs/` (a `packages/<project>/` monorepo layout). Erased the
   gmail-ftp project in full — Go code, the plugin, and its release.

4. **User documentation + site.** Replaced the engineering-note docs with a real user guide and a
   **Cookbook** (mail, databases, files, cross-service joins, code, automation) — every example
   verified against the actual parser/planner. Added a **VitePress** site at repo-root `/docs` with
   a custom `qfs` syntax highlighter, served via **Docker** for local dev. Rewired the anti-drift
   doc generator to emit at the repo root while staying in sync.

5. **A working release.** Found and fixed a release pipeline that could never have produced
   binaries (Linux-only runner building macOS targets; `sha256sum` absent on macOS; a mis-named
   checksum sidecar; a wasm "artifact" that was never built). Rebuilt `release.yml` as a proper
   per-runner matrix, made `xtask dist` target-selectable, and cut **v0.0.1** — all four native
   tarballs (Linux musl + macOS, both arches) published and checksum-verified. The real `install.sh`
   was run end-to-end against the live release.

6. **Agent enablement.** Added a Claude Code plugin (skill + manifests, mirroring the sibling
   gdrive-ftp repo) so an agent in this repo learns the qfs query syntax/semantics and can execute
   `qfs` commands safely.

## 3. Changes (by area)

- **Language & runtime:** closed-core pipe-SQL grammar (frozen keyword set), effect-plan with
  PREVIEW (default) / COMMIT, irreversible-effect gating, transactions/idempotency, audit ledger.

- **Drivers (12):** local fs, SQL, HTTP/REST, Google OAuth + Gmail + Drive + Analytics, S3/R2,

- **Product:** packages/ monorepo layout, user docs + Cookbook + VitePress/Docker, working release

## 5. Concerns / honest caveats

- **Live-service commits are unverified end-to-end.** By deliberate trip scope, the suite is
  hermetic (no live credentials/network): driver `describe`, `preview`/plan, and the closed-core
  semantics are thoroughly tested, but actually *committing* against real Gmail/SQL/GitHub/etc. has
  not been exercised. The trip routed this to a future "live smoke" pass. This is the principal
  risk to weigh before relying on writes against production services.
- **Cloudflare Workers / wasm is parked.** `qfs-host` compiles wasm-clean, but no deployable wasm
  cdylib exists yet, so releases ship native binaries only (documented; not on the install path).
- **The `curl …/main/…/install.sh` one-liner** needs `install.sh` on `main` — satisfied by this
  merge. The release artifacts themselves are already live.

## 6. Successful development patterns

- **Closed core + open registries** kept a new backend from ever touching the grammar — twelve
  drivers, zero new keywords.
- **Anti-drift generated docs** (reference rendered from the binary's own registries, checked in CI)
  meant the reference can't lie about what the binary does.
- **Verify-against-the-binary** for every documented example — the Cookbook and the agent skill were
  validated by running statements through the real parser/planner, not by hand-waving.
- **Preview-by-default + irreversible gates** as the load-bearing safety property for both humans
  and agents.

## 7. Release readiness

**Safe to ship as the qfs foundation.** It is internally consistent and fully green (1240 tests, CI,
anti-drift, fmt/clippy), the release pipeline works and has published v0.0.1, and the docs/install
story is real. Merging makes qfs the repository's product, replacing gmail-ftp. The one caveat a
reviewer should hold consciously is that **writes against live services are not yet smoke-tested**;
reads/describe/preview and the whole closed-core surface are.
