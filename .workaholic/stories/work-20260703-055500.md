## 1. Overview

A hotfix on top of v0.0.15: the first live paste-back Google consent on the released binary failed at the final step with `userinfo status 401`, because the consent scope union carried only the four API scopes and no OIDC identity scope. The consent now also requests `openid email`, so the profile-email lookup that keys the account succeeds. Version bumped to 0.0.16.

**Highlights:**

1. The Google consent auth URL now requests the OIDC identity pair (`openid email`) alongside the four least-privilege API scopes, fixing the live `userinfo status 401` at the end of the paste-back flow
2. The scope-union unit test now pins the exact six-scope union (four API scopes + openid/email) and forbids the broad `profile` and full-mailbox scopes

## 2. Motivation

The owner ran the first real paste-back consent round-trip on released v0.0.15: `state` verification and the code exchange succeeded (the pasted redirect's `scope=` parameter showed exactly the four granted API scopes), but the flow died at the last step — `profile lookup failed: userinfo status 401`. `authorize` keys the account by the userinfo profile email, and Google's OIDC userinfo endpoint rejects an access token that carries only API scopes. This was a latent gap since the original loopback flow: live consent had never actually run (it sat behind the retired `QFS_GOOGLE_CONSENT` opt-in, and live verification used the stdin token import), so the missing identity scope was only exposed by the first real round-trip.

## 3. Changes

The fix is one scope-union change plus its test: request `openid` and `email` in the consent auth URL so the minted access token can call userinfo. `include_granted_scopes=true` accumulates the new scopes onto the owner's existing grant, so re-running `qfs account add google` completes the interrupted onboarding.

### 3-1. Consent scope union must include the OIDC identity pair (openid email) ([435eb6a](https://github.com/qmu/qfs/commit/435eb6a))

Added `openid` and `email` to `all_google_scopes()` in `crates/qfs/src/google.rs` — the union stays least-privilege (no `profile`, no `https://mail.google.com/`) — and updated the scope-union test to assert the exact six-scope set. Version bumped 0.0.15 → 0.0.16.

## 4. Outcome

- The paste-back consent can now complete end-to-end: the access token minted from the code exchange carries the identity scopes the userinfo profile-email lookup requires
- The scope union is test-pinned at exactly six scopes: gmail.modify, gmail.compose, drive, analytics.readonly, openid, email
- All workspace tests (141 suites), clippy `-D warnings`, fmt, gen-docs and gen-skills checks green

## 5. Historical Analysis

The v0.0.15 branch shipped the paste-back consent with the state check, exchange, and persistence machinery fully hermetically tested — but the userinfo step's scope requirement is a property of Google's live endpoint that no mock could catch. It follows the same lesson as the v0.0.14 passphrase-gate regression: first-user, owner-attended runs on the released binary are the only way to expose live-endpoint contract gaps, which is exactly why the deployment contract keeps the owner-attended round-trip as the acceptance proof.

## 7. Successful Development Patterns
