# Branch Story: work-20260705-015204

## 1. Overview

A one-ticket v0.0.21 hotfix for upgrade friction introduced by the v0.0.20 owner-only `0600`
credential-store hardening: a credential DB or secrets blob created world/group-readable under a
pre-v0.0.20 umask is now **self-healed** ‚Äî tightened to `0600` in place on open ‚Äî instead of refusing
to open (which bricked the CLI until the user manually `chmod`'d each store).

**Highlights:**

1. A loose-but-**owned** credential store/blob is tightened to `0600` on open and used, rather than
   rejected ‚Äî a pre-v0.0.20 `644` store heals on the next command.
2. A file that cannot be tightened (foreign owner ‚Üí `EPERM`, or a mode-ignoring filesystem) is still
   refused with the manual `chmod 600` remedy. The guard only ever tightens, never loosens.

## 2. Motivation

The owner upgraded to v0.0.20 and hit `credential DB ~/.config/qfs/project.db is
group/other-accessible (mode 644); refusing to use it` on their own host ‚Äî the intended fail-closed
posture, but it bricks every existing install until a manual `chmod 600` per store. Owner chose to
self-heal (option A): tightening one's own credential DB from `644` to `600` is strictly safe and is
the exact remedy the error already printed, so do it automatically.

## 3. Changes

### 3-1. Auto-tighten a loose-but-owned credential store to 0600 on open ([d88851e](https://github.com/qmu/qfs/commit/d88851e))

`verify_owner_only` in both `store/src/fs_perms.rs` (the Project/System DBs) and `secrets/src/local.rs`
(the LocalStore credential blob) now tightens a loose-but-owned file to `0600` via `set_permissions`
and continues, re-verifying the tighten landed. A foreign-owned file fails the chmod with `EPERM` ‚Äî
the correct refusal, without an explicit uid syscall. Reject-behavior tests became tighten tests.

### 3-2. Make the migration runner concurrency-safe to unblock flaky CI

The self-heal PR's CI failed deterministically on the pre-existing env-test flakiness: parallel
`qfs`-crate tests race `XDG_CONFIG_HOME`, open the same `system.db`, and both run the migration ‚Üí
`UNIQUE constraint failed: schema_version.version` (which then poisons the shared `ENV_LOCK` and
cascades). The migration apply now uses an `IMMEDIATE` (write-locking) transaction that re-checks the
version under the lock, plus a `busy_timeout` in `Db::open`, so concurrent opens of one DB serialize
instead of racing the check-then-insert. A new `concurrent_migrations_on_the_same_db_do_not_race`
test (8 threads, one file DB) proves it. This deterministically killed the CI crash (the flaky env
tests now pass parallel). The primary env-test-hygiene fix stays as ticket `20260705022000`.

## 4. Outcome

The v0.0.20 upgrade no longer bricks an existing install; the credential-store permission guard
self-heals a safe tightening while still refusing what it cannot (or should not) tighten. Version
bumped to 0.0.21.

## 6. Concerns

### qfs-crate env-var tests bleed `XDG_CONFIG_HOME` under parallel `cargo test` (crash fixed; hygiene remains)

- **Severity:** low
- **Description:** The `qfs`-crate `init`/`oauth`/`store` tests mutate process-global env and, under
  some parallel interleavings, open a shared `system.db`. The **crash** this caused (a migration
  `UNIQUE constraint` that poisoned the shared `ENV_LOCK` and cascaded) is fixed here by the
  concurrency-safe migration runner (¬ß3-2). The underlying env bleed remains ‚Äî the tests just no
  longer crash on it.
- **How to Fix:** Ticket `20260705022000` (filed this branch): route the env-mutating tests through a
  single `ENV_LOCK` with a fresh per-test tempdir (never `remove_var` XDG to the shared `$HOME`), and
  a non-poisoning lock so one failure can't cascade.

## 7. Successful Development Patterns

- **Relying on `set_permissions` EPERM instead of an explicit uid check** ‚Äî a foreign-owned file fails
  the chmod naturally, so the self-heal needs no `libc`/uid syscall to distinguish "ours" from "not
  ours"; re-verifying the mode after chmod additionally catches a filesystem that ignores mode.
- **Running `cargo fmt --check` unpiped** ‚Äî the prior branch's CI rustfmt failure came from piping the
  check through `tail` (masking the non-zero exit); this branch verified fmt unpiped and read the exit
  code, catching drift locally.
