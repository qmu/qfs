## 1. What this branch delivers

### An operator FAQ skill (`qfs-faq`)
- **`docs/cookbook/faq.md` → generated `qfs-faq/SKILL.md`** (`fc6d789`): a how-do-I / troubleshooting
  reference in operator vocabulary — connection & account setup, a **second or different-organization
  Google account**, the `403 org_internal` "access blocked" wall (a Google OAuth consent-screen
  config, not a qfs bug, with ranked fixes), the describe→preview→commit safety loop, common errors,
  the stable exit-code table, and a per-service routing index. Authored the same way as every other
  qfs skill (a cookbook article carrying `skill_name`/`skill_description`, rendered by the existing
  `gen-skills` pipeline — no generator change). Registered in `marketplace.json`, all four plugin
  `version` fields bumped `0.2.0 → 0.3.0` (a new taught surface = minor), and the article is linked
  from the docs cookbook index + sidebar (reachability).

### The FAQ's shell answers can't silently drift (`qfs-faq` anti-drift)
- **`crates/cmd/tests/faq_cli_surface.rs`** (`474f583`): the FAQ answers most questions with **shell
  commands** (`qfs connect --driver … --account …`), which the `cookbook_skills.rs` recipe ratchet
  never sees — it only parses ```` ```qfs ```` statements. This test extracts every
  `qfs <subcommand…> [--flags]` from the FAQ's ```` ```sh ```` fences and asserts each subcommand path
  and long flag exists in the real clap `Command` tree (via a new `qfs_cmd::clap_command()` reflection
  entrypoint). A renamed/removed flag turns CI red until the FAQ is corrected. Proven both ways: green
  on the true FAQ, red on an injected `--account → --acct` rename.

### git → `path_binding` convergence (`134e2b8`)
- `sql` converged onto the `path_binding` registry in a prior branch (`20260705000500`), but `git`
  still rode only the older declared-connection seam (`connections_config::declared_for("git")` +
  `QFS_GIT_*` env) — a persisted `qfs connect /git/<repo>` binding was **invisible** to run, commit,
  and describe. This wires `path_binding_git_connections()` into `has_connections()` + `git_driver()`
  (registered LAST so a persisted binding wins a name clash with the env/`connections.qfs` shims) and
  adds the matching `qfs describe` mount. Because `shell.rs`/`commit.rs` already call those two
  functions, run and commit converged for free; describe got the one new gated registration. Hermetic
  test mirrors the sql convergence test (project-DB `path_binding` fixture).
