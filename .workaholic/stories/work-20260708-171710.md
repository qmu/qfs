# Branch story — `work-20260708-171710`

## 1. Overview

A seven-ticket drive branch that hardens qfs's **write surface across three services** and squares
the docs to the objective-documentation policy. The through-line: every user-facing write path is
made honest — it does what its describe/preview promises, refuses what it cannot safely do, and is
taught truthfully in the cookbook. The heaviest work is a four-ticket sweep of the **Gmail compose
surface** (send-existing-draft, thread reply, attach/detach on every form) plus a **Drive
overwrite guard**, **Slack channel/DM file listings**, wiring the **interactive shell to live cloud
surfaces**, and a **docs objectivity** pass. Binary `0.0.32 → 0.0.36`; qfs plugin `0.6.0 → 0.6.2`.

## 2. Motivation

qfs's read surface was already broad; the gaps were on the **write** side, where a preview that
claims one effect and a commit that does another is the worst failure mode. A 2026-07-08 review of
the Gmail surface found three: sending an existing draft by id silently fell into a byteless
create-then-send; there was no way to reply *into a thread*; and attachments worked only on a draft
`INSERT`/`UPSERT`, not on a one-shot send or a reply. In parallel, a Drive `INSERT` could silently
overwrite an existing file, Slack file listings were mailbox-wide rather than channel/DM-scoped, the
interactive shell was a local-only sandbox that diverged from `qfs run`, and the generated docs
carried subjective/question-style headings the objective-documentation policy disallows.

## 3. Changes

### 3-1. Make docs headings and wording objective ([99794d8](https://github.com/qmu/qfs/commit/99794d8))
Reworked the generated docs (`language.md`/`drivers.md`/`server.md`) and cookbook prose to drop
subjective and question-style headings, fixing offending text at the `docs.rs` Rust source (the three
docs are byte-compared to `docs.rs` by `docs::tests`) and regenerating. `housekeeping`, `Changed`.

### 3-2. Mount cloud surfaces in the interactive shell ([e4f441d](https://github.com/qmu/qfs/commit/e4f441d))
Wired the interactive shell's plan/read side to the live cloud registry so it reaches connected
services (Gmail/Drive/Slack/…) at parity with `qfs run`, behind the unchanged preview→typed-COMMIT
gate. The commit apply side already used the live registry; only the plan/read side needed wiring.
Grew out of the `fix-drive-cp-local-byte-upload` ticket. `bugfix`, `Changed`.

### 3-3. Add channel- and DM-scoped Slack file listings ([32536be](https://github.com/qmu/qfs/commit/32536be), [6f92a20](https://github.com/qmu/qfs/commit/6f92a20))
Added channel-scoped and DM-scoped file listings (newest-first) over `/slack`, reusing the existing
`NodeKind::Files` schema/decode and the message-log channel/IM resolution seam — no new node kind, no
client change. Also fixed DM addressing to be **user-id throughout** (`conversations.open` needs a
user id, so a display name never opens a DM) and corrected the docs wording. `enhancement`, `Changed`.

### 3-4. Guard Drive INSERT against silent overwrite ([4f3fa50](https://github.com/qmu/qfs/commit/4f3fa50))
Made Drive `INSERT` **create-only**: it fails closed at COMMIT with the existing id rather than
silently overwriting. The safety is delivered without breaking qfs's preview-touches-nothing
invariant — the create-only probe is one extra leaf lookup under the already-resolved parent, not a
preview-time network read. Carries a design brief ([7f52a9e](https://github.com/qmu/qfs/commit/7f52a9e)). `enhancement`, `Changed`.

### 3-5. Fix sending an existing Gmail draft by id ([2131a57](https://github.com/qmu/qfs/commit/2131a57))
`/mail/drafts/<id> |> call mail.send` now sends *that* draft instead of falling into a byteless
create-then-send that failed "draft has no `to`". A CALL effect node carries only its target path and
literal args (never upstream rows), so the draft id had to be lowered from the **path segment**. Added
a `Driver::plan_call` seam so a driver can refuse a CALL at plan time and keep PREVIEW/COMMIT in
agreement. `bugfix`, `Changed`.

### 3-6. Add Gmail `mail.reply` thread-reply drafts ([197c851](https://github.com/qmu/qfs/commit/197c851))
New **reversible** `mail.reply` procedure addressed at a parent message
(`<parent> |> call mail.reply(body => …[, to, cc, subject])`): it drafts into the parent's thread,
setting the Gmail `threadId` (server-side) **and** `In-Reply-To`/`References` from the parent's
`Message-Id` (client-side), defaulting `to` to the parent's `From` and `subject` to `Re: <subject>`.
Sending reuses the shipped `/mail/drafts/<id> |> call mail.send`, which now threads — no second
threaded-send path. A parent with no resolvable `Message-Id` fails closed. `enhancement`, `Added`.

### 3-7. Add Gmail attach/detach on every send/reply form ([41e0ce4](https://github.com/qmu/qfs/commit/41e0ce4))
Added an `attachments` param (`Array(Struct{filename, mime, bytes})`) to the `send` and `reply`
procs, so `call mail.send(…, attachments => […])` and a reply attach files — attachments now work on
**every** form (INSERT/UPSERT already did). Made **detach = UPSERT full-replace** explicit and
tested (omit one to drop it, omit all to clear; no new keyword), and proved a *received* file
re-attaches by sourcing its bytes from the attachment byte-read (not the metadata-only listing). Also
cleaned up gen-docs's nested-param-type rendering (`docs.rs` `render_col_type`). `enhancement`, `Added`.

## 4. Outcome

All seven tickets landed with hermetic proof and the anti-drift ratchets green. The Gmail driver test
suite grew to **64 tests** (mock-only, no live Gmail): the compose surface is now proven end-to-end
through the interpreter — reply threads and defaults, attach on every form, detach-by-omission,
detach-then-reattach round-trip, received-bytes re-attach, send-by-id without re-create, and
fail-closed paths. Drive `INSERT` is create-only; Slack file listings are channel/DM-scoped; the
shell reaches live cloud behind the gate; the docs read objectively. Users can now **reply into a
thread and attach/detach freely across every compose form**, the headline gap the branch set out to
close.

## 5. Historical Analysis

This branch continues the compose-surface arc: it builds directly on the t92 attachment foundation
(`Array/Struct/Bytes` literals + the `attachments` column) and the `/mail/drafts` write-parity work,
and the reply/attach tickets explicitly reuse the just-shipped send-by-id fix (3-5) rather than
adding parallel paths. The `plan_call` seam introduced in 3-5 became the pattern the reply guard
(3-6) reused — preview/commit agreement enforced at plan time is now the driver idiom for CALL
refusal. The Slack and Drive tickets extend their respective drivers along seams that already existed
(message-log channel resolution; the resolved-parent leaf lookup), keeping the blast radius small.

## 6. Concerns

Most deferred items are **settled design choices**, not open risks. The one cross-ticket concern —
"attach/detach must cover the reply form" (raised in 3-6) — is **resolved** by 3-7 on this same
branch. Remaining low-severity follow-ups:

### Live-cloud smoke tests deferred to the owner, out-of-band
- **Severity:** low
- **Description:** Several tickets (Gmail send/reply, Drive `cp`, Slack file share, shell cloud read)
  are proven only at the hermetic mock boundary; the final live confirmation against real
  Gmail/Drive/Slack was intentionally not run (this host has LIVE cloud accounts — probing them
  mutates real data). Server-side filters (Slack file-share scoping) and threaded delivery are the
  parts a live run would confirm.
- **How to Fix:** Owner runs the one-shot live smokes out-of-band with explicit approval (reply to a
  throwaway thread; attach/detach a draft; a Drive `cp`; a channel file listing). Standing acceptance.

### Draft discard (`drafts.delete`) is a named follow-up
- **Severity:** low
- **Description:** A `/mail/drafts/<id>` node advertises `Select` + `mail.send` only; discarding a
  draft by id (`drafts.delete`) is not wired, to avoid shipping a message-id trash that breaks on a
  draft id (see [2131a57](https://github.com/qmu/qfs/commit/2131a57)).
- **How to Fix:** Add a draft-scoped discard procedure/verb in a follow-up ticket.

### Slack DM addressing is by user id, not display name
- **Severity:** low
- **Description:** `/slack/<ws>/dms/<user>` is user-id throughout; a display name never opens a DM
  (see [6f92a20](https://github.com/qmu/qfs/commit/6f92a20)). Share-target arrays
  (channels/ims/groups per file) are also not exposed as owned columns — the channel-scoped request is
  the objective surface by design.
- **How to Fix:** A `users.list` display-name→user-id resolution would be a consistent enhancement
  across DM messages and files; out of scope here.

### Detach is UPSERT full-replace (no per-attachment identity)
- **Severity:** low
- **Description:** Detach is by re-UPSERTing the desired attachment set; there is no stable
  per-attachment id or `DETACH` keyword (settled design, see [41e0ce4](https://github.com/qmu/qfs/commit/41e0ce4)).
- **How to Fix:** Revisit only if a stable attachment id is later needed.

## 7. Successful Development Patterns

- **plan_call preview/commit agreement.** A driver refusing a CALL at plan time (not apply time)
  keeps PREVIEW honest, since PREVIEW never decodes the effect. Introduced for send-by-id, reused by
  the reply guard.
- **Reuse over reinvention.** The reply draft is a normal reversible draft plus thread linkage;
  sending it reuses the shipped send-by-id path; the reply/attach forms ride the same `attachments`
  column the INSERT decoder already reads (declaring the param is all that was needed, since proc
  params are only name-checked). Slack/Drive tickets extend existing seams, adding no new node kinds.
- **Fix the source, regenerate.** The three docs are byte-compared to `docs.rs`, and cookbook prose
  feeds `SKILL.md` verbatim, so offending text is fixed at the Rust source and the docs/skills
  regenerated — the anti-drift ratchets enforce it.
- **Invariant-preserving safety.** The Drive overwrite guard delivers create-only safety without a
  preview-time network read, respecting preview-touches-nothing.
