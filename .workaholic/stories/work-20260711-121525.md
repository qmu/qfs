## 1. Overview

This branch drives the **capability-tryout mission** — proving qfs's file-handling and transformation
reach — through thirteen dependency-ordered tickets, from wiring the transform epic's deferred live
half all the way to the mission's largest language change: model-chosen `switch` routing in the pipe,
proven in an owner-attended live round against a real model, real Drive, and real Gmail. Along the way
it lands cross-service transfers in both directions (Gmail→Drive, Drive→Gmail-reply), real Slack bytes
upload, the Chatwork and GitHub declared-driver rewrite proofs (API-key and account-auth variants),
real PDF document input for Extraction transforms, pinned transform-chain semantics, restored
in-server CREATE JOB firing, a machine-enforced command-execution lock, and a re-measured dependency
posture. The live round both passed its gate and did its job as a defect net: it surfaced a latent
panic that had silently blocked every live transform commit (fixed on-branch with three other
defects), and produced two precisely-scoped driver bug tickets for the next drive.

**Highlights:**

1. Wired three live `ModelProvider` impls (Anthropic Messages, OpenAI Chat Completions, Google Gemini) behind the transform seam, then extended them with real per-provider PDF/image document input for Extraction mode — the one-statement PDF→text→Drive pipeline commits hermetically.
2. Landed model-chosen routing end to end: blueprint §18 design brief first, then `|> switch <col> { 'a' => <arm>, else => <arm> }` — the model picks among pre-declared, operator-consented arms and can never invent an effect. The owner-attended live round COMMITTED with exact per-arm counts.
3. Completed the cross-service file story: one-statement Gmail-attachment → Drive-folder transfer (fixing a general projection-alias bug), a Gmail reply threaded with a Drive-sourced attachment over a new `/mail/<label>/<msg>/replies` append-log, and real Slack bytes upload over the external-upload flow.
4. Proved the declared-driver rewrite thesis twice: `chatwork.qfs` (API-key form, zero compiled code) and `github_account.qfs` behind a new generic `AUTH ACCOUNT '<provider>'` clause that reuses a connected account's credential at wire time.
5. Hardened the platform: the "no command-execution risk" audit claim became an enforced allowlist lock (`exec_inventory.rs`); in-server `CREATE JOB` firing was restored with ruled missed-fire/overlap/UTC semantics; blueprint §11 gained a dated dependency snapshot (30 shipped direct deps, removable-today ≈ 0, `uuid` newly logged).
6. Fixed three live-round defects on-branch: the driver-http ambient-runtime panic (was silently blocking EVERY live `|> transform` commit), switch prune now bridges dependency edges (declaration order + fail-stop restored), and the System-DB v16 in-place edit self-heals via `SUPERSEDED_BODIES`.

## 2. Motivation

The transform epic (v0.0.42) shipped the whole model-calling pipeline but registered only the
fail-closed `UnconfiguredProvider`; the live providers were the recorded deferral, and the mission's
transform tickets (PDF extraction, transform chains, switch routing) were blocked on them. The
file-handling goals demanded uniform attach/detach across the three compiled messaging services,
demonstrated cross-service transfer in both directions, and reply-with-attachment; the
declared-driver goals demanded a rewrite proof for both credential shapes (API-key and
OAuth-style/account); and the platform-hardening goals demanded the command-execution posture be
enforced rather than asserted, the dependency posture re-measured with numbers, and the t65
"qfs is not a scheduler" ruling revisited (the owner reversed it: "Changed mind, we need this").
The switch ticket — "let AI choose the tool to call" — was deliberately design-brief-first by owner
direction, so the grammar landed against ratified rulings instead of improvised syntax. Each ticket
held to the hermetic-first policy: suite green gates /drive approval, live rounds stay
owner-attended; this branch is the first whose gate actually included a passed owner-attended live
round.

## 3. Changes

### 3-1. Implement three live transform model providers ([89bccc3](https://github.com/qmu/qfs/commit/89bccc3))

New `crates/qfs/src/transform_providers.rs`: a `LiveModelProvider` dispatcher routes each call to the
Anthropic/OpenAI/Google impl named by the definition's `provider` column (unknown → fail-closed
Unconfigured). Each hand-rolled REST impl builds an owned `HttpRequest` over the shared driver-http
`HttpClient` seam (MockHttpClient in tests), pins its API version, requests structured JSON output,
and folds the reply onto the declared OUTPUT schema. A missing key fails closed pre-network; 429/5xx
retries are bounded and honor Retry-After; `max_tokens` is capped for cost control. `x-goog-api-key`
was added to the single `qfs-http-core` redaction authority so the Gemini key can never reach a log.
The composition root swaps `UnconfiguredProvider` for the live dispatcher over a `ReqwestClient` and
attaches a lazy `vault:` resolver. Blueprint §15 gained the provider matrix.

### 3-2. Add Gmail-attachment to Drive-folder transfer ([6aa01d8](https://github.com/qmu/qfs/commit/6aa01d8))

One statement — `/mail/<label>/<msg>/<att> |> select filename as name, mime as mime_type, content as
bytes |> insert into /drive/<acct>/<folder>` — now transfers an attachment's bytes into a named Drive
folder, byte-identical. Two gaps were fixed: the Gmail driver's `describe` advertised the
message-listing schema for an attachment node (so the SELECT columns didn't resolve at plan time),
closed with a dedicated attachment `describe` arm; and a **general** projection bug — an aliased plain
projection lowered to the name-only pushable `Project`, which selects by source name and cannot
rename, silently dropping every renamed column — fixed by treating a rename as non-plain and routing
it through the local `ProjectExpr`. Added the first exec-level end-to-end commit test of
`materialize_pipeline_source`, plus pushdown alias-lowering regressions and a cross-service cookbook
recipe.

### 3-3. Add Slack file bytes upload (external flow) ([58722b5](https://github.com/qmu/qfs/commit/58722b5))

`SlackEffect::UploadFile` existed but was **dead** (never constructed), and the Files node didn't even
grant INSERT/UPSERT — the upload was doubly unwired. This wires `UPSERT INTO /slack/<ws>/files` (and
the `INSERT`/`cp` forms) to a real bytes upload over Slack's external-upload flow (the legacy
`files.upload` is sunset for new apps): reserve an upload URL, POST the raw bytes to the pre-signed
URL (no bearer to the third-party host), then complete with an optional channel share. The row speaks
the same `filename`/`mime`/`bytes` vocabulary as Gmail attachments and Drive blobs. Form-encoding is a
small inline percent-encoder (the driver stays dependency-light and wasm-safe). Five hermetic tests
cover the decode, the three-request wire flow, and detach irreversibility; slack.md gained the
upload/detach recipes.

### 3-4. Add Chatwork declared driver (API-key proof) ([ec0e4ec](https://github.com/qmu/qfs/commit/ec0e4ec))

New `chatwork.qfs` declared-driver asset (`AUTH HEADER 'x-chatworktoken'`, host-confined to
`/http/chatwork`, credential-free) covers rooms/messages/files listing + message post in
`CREATE DRIVER`/`TYPE`/`VIEW`/`MAP` statements with zero compiled Rust — the mission's API-key rewrite
proof, alongside the shipped Cloudflare precedent. A new `qfs-chatwork` cookbook skill teaches
connect/read/post. File **download** (a cross-host `download_url` follow) and **upload** (multipart)
are the two operations the declared model can't yet express; both are recorded in-asset as generic
single-primitive additions the evaluator needs (`FOLLOW <field>`, `ENCODE multipart`) — not
Chatwork-specific code. Satisfies two mission acceptance items at once.

### 3-5. Add Slack user-token posting cookbook guide ([0120c7c](https://github.com/qmu/qfs/commit/0120c7c))

The Slack cookbook gained "Post as yourself (user token)" and the team-wide proxy pattern — posting
as a human user is a credential-topology choice, not a code change, because `driver-slack` passes the
stored credential to `chat.postMessage` verbatim: register the `xoxp-` user token as a second account
and mount it on its own defined path beside the bot mount. Both team topologies are documented with
their trust boundaries — per-member vault (recommended; only your own AI speaks as you) versus a
central box, explicitly flagged as impersonation power concentrated in one host. A scopes table
records the live-verified gotcha that a bot token without history scope can post but not read back.
The ticket had also observed an older binary rejecting the bare positional insert (`values ('…')`) at
commit while previewing fine; a new driver-slack decode test locks that a single positional value
binds to the `text` column, so preview and commit agree. The regenerated `qfs-slack` skill carries the
recipes; plugin bumped (taught surface).

### 3-6. Lock command-execution risk to an enforced invariant ([6b4b29f](https://github.com/qmu/qfs/commit/6b4b29f))

The prose audit claim that qfs carries no command-execution risk became a mechanically enforced
invariant — the same move the transform one-seam lock made for model calls. A new governance suite,
`crates/cmd/tests/exec_inventory.rs`, enumerates every process-spawn site in the workspace against an
exact allowlist of (crate, file, spawned program) triples and fails on any new or changed site; a
companion test forbids shell-string execution (`sh -c`, shell joins) anywhere. The audited footprint
is tiny: qfs spawns only `git` (driver-git applier, repo introspection, migration guard) plus a
`Stdio::null` desktop opener. Argument-hygiene tests in `driver-git/src/applier.rs` prove the
realistic vector — git option injection through refs and oids — is neutralized: the only
query-derived positional is a ref name made inert by `qualify_ref` prefixing `refs/heads/`, and oids
are validated to 40 hex chars, so no value can present to git as a leading-dash flag. Blueprint §17
documents the inventory and the standing rule: any new spawn site requires a deliberate allowlist
edit plus blueprint update in the same PR.

### 3-7. Thread a Gmail reply with a cross-service file ([aabf68d](https://github.com/qmu/qfs/commit/aabf68d))

Since `CALL mail.reply` takes only literal arguments and cannot source attachments cross-service, a
new `/mail/<label>/<msg>/replies` append-log was added: an INSERT into it threads a reversible reply
onto the parent message, and because the INSERT body is a pipeline it flows through the existing
commit-time `materialize_pipeline_source` channel — a Drive file packed with `ARRAY_AGG(STRUCT)`
lands as the reply attachment byte-identical. The implementation reused the shipped
`GmailEffect::Reply` applier verbatim (thread-id plus `Re:` subject defaulting), so only a new path
node with describe/capability/decode arms was needed across driver-gmail. A 236-line end-to-end exec
test asserts threading fields and byte-identical attachment content. Scope was narrowed honestly with
the owner from the ticket's three pipes to the one needing new wiring: the Slack leg already ships as
file-shared-into-channel, and the Chatwork leg is blocked on the multipart primitive. The
cross-service cookbook and regenerated skill teach the recipe; plugin bumped.

### 3-8. Add AUTH ACCOUNT to the declared driver model ([73942b4](https://github.com/qmu/qfs/commit/73942b4))

The declared-driver AUTH clause previously modelled only a static header or token, which excluded
OAuth-style services from the declaration model. A new `AUTH ACCOUNT '<provider>'` grammar production
desugars to an account descriptor, and a new `AuthStrategy::Account` variant in driver-http injects
the provider account's bearer token at wire time through the same `inject_auth` seam as Bearer — the
key insight being that AUTH ACCOUNT is wire-identical to Bearer, differing only in that the
coordinate names the shared provider account rather than the driver's own SECRET, so no new injection
path exists and the API-key forms parse unchanged. The binary side builds an account-backed secrets
adapter mapping the declared coordinate to the connected account, failing closed with a structured
error when the account is absent. A read-only proof asset, `github_account.qfs`, mounts `/ghdecl`
beside the compiled `/github` driver as a side-by-side rewrite proof. Hermetic tests cover the parser
desugar, wire injection, declared-driver lift, the account adapter, fail-closed behavior, and
credential-free install; blueprint records the ruling and considered alternatives.

### 3-9. Restore in-server CREATE JOB firing, reversing t65 ([6ec5a95](https://github.com/qmu/qfs/commit/6ec5a95))

The owner reversed the settled t65 "qfs is not a scheduler" decision, so this restored in-server
`CREATE JOB` firing on top of the substrate that already existed (CREATE JOB rows,
`BindingKind::Cron`, the watchtower gate-to-commit chain). A new `qfs_watchtower::cron` module
implements the missing "when": a pure `is_due`/`interval_secs` decision over an injected `now`, and a
`fire_due` that fires each due job through the existing watchtower Committer, producing secret-free
run records. The firing decision is deliberately pure and wasm-clean — only the daemon leaf touches
`SystemTime` — and because a firing is just a watchtower commit, the policy gate, irreversible guard,
at-least-once semantics, and idempotency come for free with no new commit path. Semantics were ruled
explicitly: missed fires skip with one catch-up (no storm), overlapping runs skip, timezone is UTC
only. Blueprint §10 records the reversal along with the constraints kept from t65 (no scheduler
library, no wall-clock in the pure core). Eight injected-clock tests cover cadence, due/not-due,
missed-fire-once, overlap-skip, policy-denied-no-stamp, and irreversible-blocked.

### 3-10. Wire PDF document input for transform Extraction mode ([5b4d394](https://github.com/qmu/qfs/commit/5b4d394))

Extraction mode had shipped with a stubbed bytes-to-provider leg — a document blob was base64-inlined
as plain JSON text, so no provider actually treated it as a document. This wired real per-provider
document input in `transform_providers.rs`: an Anthropic document content block, an OpenAI file part
(data URL), or a Gemini `inlineData` part, chosen by the def's provider, with a mime sniff
distinguishing PDF/PNG/JPEG. An oversized document fails pre-network with a structured error naming
the provider's inline cap (the fail-at-PREVIEW requirement). Because the response-folding,
materialize, and Drive-write rails were already hermetic, only the request encoding plus the size
guard were needed to complete the flagship one-statement PDF→text→Drive pipeline. Tests cover the
three provider document shapes, the oversized pre-network failure, and an end-to-end fixture
asserting extracted text lands in Drive byte-identical. The cross-service cookbook gained the recipe
with an informed-egress warning naming which provider sees the document; plugin bumped.

### 3-11. Pin transform-chain semantics with tests and a recipe ([f812d54](https://github.com/qmu/qfs/commit/f812d54))

Chained transforms (`… |> transform a |> transform b`) were built to plan via the plan spine's schema
fold but had never been deliberately exercised. Investigation showed no source change was needed:
`LogicalPlan::Transform` already exposes its OUTPUT columns to the next stage's INPUT-presence check,
so a chain is just Transform-over-Transform, and forced-local stacking, the schema handoff, and the
single-commit envelope all fall out of the existing spine. The ticket made the semantics
true-by-test: lowering tests pin that a compatible chain plans as nested Transform nodes typed by the
upstream OUTPUT and that an incompatible chain fails at preview naming the unsatisfied INPUT column;
execution tests pin stage order with schema-checked handoff, that PREVIEW calls no model for either
stage, and that a mid-chain provider failure aborts the whole statement atomically. The gate ruling —
one `--commit` covers all of the chain's model calls as the statement's single effect — is now
locked. The cookbook gained a parse-checked extract-then-summarize chain recipe; plugin bumped.

### 3-12. Land the switch-routing design brief (blueprint 18) ([4243ab8](https://github.com/qmu/qfs/commit/4243ab8))

The mission's largest language change started design-first by owner choice: blueprint §18 landed
before any grammar. The brief rules the disciplined shape — a transform emits a closed enum-like
column, then a new pure `switch` stage routes rows to declared sub-pipeline arms by that column's
value — so the model chooses *among* pre-declared, operator-consented arms and can never invent an
effect (the one-seam lock holds). It fixes the surface form
`|> switch <col> { 'a' => <pipeline>, else => <pipeline> }` with `switch`/`else` as contextual
identifiers (the keyword freeze does not move), and rules per-row-batched-per-arm routing,
discriminant-typed exhaustiveness failing at PREVIEW, the previewed-union commit envelope, and the
PipeOp variant lock moving 19→20 (a deliberate MINOR). The load-bearing call it records: switch arms
may carry effects — unlike opaque let-bound lambdas — precisely because they are inline and
gate-visible, reconciling model-made choice with a fully operator-consented effect set before any
model runs.

### 3-13. Re-measure the dependency posture and update blueprint 11 ([41ed9cb](https://github.com/qmu/qfs/commit/41ed9cb))

Blueprint §11 gained a dated snapshot: 50 workspace members, a 334-crate binary dependency tree, 466
crates in the full-workspace resolution, and 30 shipped direct dependencies (a 31st, chumsky, is
spike-only; its recorded "rejected" call stands). The re-measure surfaced `uuid` as a
previously-unlogged shipped dependency (pulled in for Postgres UUID-column decode) and recorded it

added no new crate or third-party dependency. Critically, no feature trim was executed — not out of
omission but because each lever was re-checked and found already exhausted (reqwest already
no-default-features + rustls, postgres carries an explicit OID set, rusqlite requires `bundled`,
env-filter genuinely used). The honest per-lever ruling (removable-today ≈ 0; async-trait remains the
monitored exit) was recorded rather than forcing a cosmetic trim.

### 3-14. Add switch routing to the pipe language ([10b00d9](https://github.com/qmu/qfs/commit/10b00d9))

All remaining §18 steps shipped in one commit (~1,600 insertions across 25 files). Parser:
`PipeOp::Switch` AST and grammar with `switch`/`else` as contextual identifiers (keyword freeze
unmoved at 39), plus a bounded arm-boundary token scan — a depth-zero comma followed by a label and
arrow — that is exact rather than heuristic, so greedy projection-list commas never swallow the next
arm; the variant-count lock moved 19→20 deliberately. Eval lowers a terminal switch to the UNION of
every arm's effect plan — write arms reuse the existing INSERT-FROM plan shape, CALL arms the
terminal-call shape — sequenced in declaration order, with structured errors for mid-pipe switch,
missing/duplicate/non-trailing else, unknown discriminant, pure arms, and non-row-local arm ops;
resolve applies the same capability and procedure gates to arms as to top-level statements. At the
exec commit boundary the source is materialized once (the model runs once), rows are partitioned by
discriminant (unmatched falls to else), each taken arm's continuation is folded over its partition,
routed rows are embedded into the consented write node, untaken arms are pruned
(previewed-but-not-fired), and one apply runs. Structural insight: reusing existing plan shapes meant
zero new engine-side plan nodes — the entire stage lives at the effect layer. 22 hermetic tests
(7 parser + 10 eval + 5 e2e, untaken-arm prune proven against a capturing applier); language
reference, cookbook recipe "Let the model pick the tool", and skills regenerated; plugin
0.10.7→0.11.0 (MINOR — taught-surface break class).

### 3-15. Fix three defects the switch live round surfaced ([a59f914](https://github.com/qmu/qfs/commit/a59f914))

The owner-attended live routing round (the ticket's gate) surfaced four real defects; this commit
fixes three, each with its own regression lock:

1. **driver-http ambient-runtime panic** — the sync `ReqwestClient` drove its owned runtime with
   `block_on`, which panics ("cannot start a runtime from within a runtime") when called from inside
   another tokio runtime — exactly what every live `|> transform` commit does. Hermetic mocks never
   issue HTTP, so the entire live-transform commit path was latently unusable. Fixed:
   `Handle::try_current` detects the ambient runtime and drives the owned runtime on a scoped worker
   thread (a joined panic degrades to a structured transport error), locked by a loopback wire
   regression test. This also resolves the carried PR #11 runtime-within-runtime concern class.
2. **Switch prune dropped dependency edges instead of bridging** — `commit_switch` pruned untaken
   arms by dropping their dependency edges, so a later arm lost all deps and could fire before an
   earlier arm; observed live as a drafts-arm write landing from a run whose Drive arm had failed,
   breaking §18-C declaration order and fail-stop. A self-inflicted regression from 10b00d9, owned as
   such. Fixed: `prune_nodes_bridging` bridges every removed node's parents to its children, with a
   unit test on the chain shape and an e2e test asserting the applied plan's consent→arm1→arm2
   dependency chain.
3. **System-DB migration v16 in-place edit fail-closed** — v16 had been edited in place at 8f063e6
   (comments only, DDL byte-identical), so a dev DB holding the pre-edit body failed closed on
   checksum mismatch. Fixed: the old body is registered in `SUPERSEDED_BODIES` with a no-op
   reconcile, plus a committed original-body fixture and a heal test proving data survives.

The remaining two defects (Drive multi-row INSERT loss, Chatwork zero-column live read) are
driver-side and were ticketed rather than fixed. The http fix unblocks the T5–T7 live rounds, which
all commit through the previously-panicking path.

## 5. Historical Analysis

The thirteen tickets complete the capability-tryout mission's implementation arc the owner directed
(created on this branch, commit `5e01f5d`), driven in dependency order: providers first (unblocking
PDF, chains, switch), file-handling parity next, then the declared-driver proofs, platform
hardening, and finally the switch capstone. Two owner reversals shaped the branch: t65's
"not a scheduler" ruling was deliberately reversed with its good constraints kept (no scheduler
library, clock-free core), and the switch ticket was run design-brief-first so grammar landed
against ratified rulings. The pattern of probing by example kept paying: writing the target
Gmail→Drive statement surfaced two latent bugs; the transform-chain ticket turned out to need zero
source changes (the plan spine already composed) and became pure semantics-pinning; and the
owner-attended switch live round — the first live gate in the mission — caught four defects that
lived precisely at the mock boundary, including a panic that had silently blocked every live
transform commit ever. The a59f914 fixes closed the loop the hermetic-first policy predicts: mocks
prove semantics, live rounds prove the wire, and each live finding either lands with a regression
lock or becomes a precisely-scoped ticket.
