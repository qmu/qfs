## 1. Overview

This branch fixes the two driver defects the capability-tryout mission's owner-attended switch
live round surfaced (2026-07-11) and ticketed for the next drive. Both were honest-count
violations living exactly at the mock boundary: a multi-row `INSERT INTO /drive/<folder>` wrote
only the first row's file while the summary reported every source row as affected, and the
declared Chatwork driver's live read returned the right row count with every column value lost.
Both now behave honestly: the Drive applier uploads one file per row with `affected` equal to
files actually created and mid-batch failures reported as exact progress, and a declared view
whose `OF` type resolves no columns refuses loudly before any wire fetch instead of silently
projecting every value away.

**Highlights:**

1. Multi-row Drive INSERT/UPSERT decodes one upload per row; the whole batch validates before the
   first write; a mid-batch API failure surfaces as a structured `partial_apply` error naming
   exactly how far the batch got — never full success. Multi-row UPDATE/REMOVE/CALL fail closed
   instead of silently dropping extra rows.
2. The Chatwork zero-column root cause was pinned to stale data, not the read path: the live
   System DB's `type` rows predate the §5.4 object body shape, so the `OF` type resolved to zero
   columns and the typed-view shaping projected everything away. The read now fails pre-network
   with a structured error naming the fix (re-install the declaration), and re-installing
   actually heals — the type lookup prefers the newest same-name declaration.
3. The cross-service cookbook's false promise ("a missing folder fails at preview") was corrected
   to the honest split: path-shape errors at preview; folder existence is a structured
   `not_found` at commit, because preview performs no network reads.

## 2. Motivation

The switch live round (archived ticket `20260711121532`, PR #33) proved model-chosen routing end
to end but surfaced four defects; two were driver-side and ticketed rather than fixed on that
branch (`20260712005000`, `20260712005100`). Both violate the honest-count doctrine — a committed
summary must never claim work that did not happen, and a delivered row must never silently lose
its values. Fixing them unblocks the mission's remaining owner-attended live rounds: the switch
re-run (two routed files landing) and the Chatwork live re-check.

## 3. Changes

### 3-1. Upload every row of a multi-row Drive insert ([def5f40](https://github.com/qmu/qfs/commit/def5f40))

The Drive effect decode read only `rows.first()`, so a multi-row batch collapsed to one upload.
Every row reader in `driver-gdrive/src/effect.rs` now takes a row index, and a new
`from_node_rows_with` decodes one effect PER ROW for the upload-shaped kinds (`INSERT`/`UPSERT`)
while the id-addressed kinds (`UPDATE`/`REMOVE`/`CALL`) refuse a multi-row batch outright. The
applier (`applier.rs`) decodes the WHOLE batch before writing anything — payload presence,
destination resolution, and create-only probes all validate up front — then applies row by row,
returning `affected` equal to operations that actually ran; a mid-batch failure surfaces as the
new `DriveError::PartialApply` (`applied`/`total`/`row`/reason) instead of full success. A
`MemoResolver` memoizes the shared destination walk across the batch (one parent resolution
serves every row, proven by a test that queues exactly one walk page). Four hermetic tests lock
the live-round shape verbatim: 2-row insert → 2 uploads in row order with text→bytes coercion,
path-addressed batch → single memoized walk, mid-batch failure → `partial_apply` with no later
row attempted, multi-row REMOVE → fail-closed. The cookbook sentence was corrected (see 3-3 in
the diff: `docs/cookbook/cross-service.md` + regenerated `qfs-cross-service` skill). qfs
0.0.56→0.0.57; plugin 0.11.0→0.11.1 (taught-surface correction).

### 3-2. Refuse a zero-column declared OF contract loudly ([8596226](https://github.com/qmu/qfs/commit/8596226))

Root cause, verified against this host's actual System DB: the live `type` rows store the
pre-§5.4 ARRAY body (`[{"name":…}]`), which the current `type_column_names` (expecting
`{"columns":[…],"where":…}`) parses to zero columns; `view_specs` encoded that as
`of_columns: Some([])` and `shape_to_type` projected every delivered value away — right row
count, empty rows, exactly the live symptom. Hermetic installs always write the current shape,
which is why the suite never saw it. Two changes, per the no-back-compat policy (no silent
legacy parsing): `eval_view_body` now refuses an `OF` contract that resolves no columns BEFORE
any wire fetch, with a structured `invalid_path` naming the cause and the fix ("re-install the
driver declaration"); and `types_from_conn` orders newest-first so a re-installed declaration
actually WINS the lookup and heals the mount (matching the describe path's `ORDER BY id DESC`).
Five hermetic tests: the decode→typed-view seam with a real-shaped Chatwork `/rooms` bare-array
body (surplus fields included, values asserted present), the loud refusal (fetch proven never to
run), the stale array-body parse, the `Some([])` encoding, and newest-declaration-wins. The real
binary was run against this host's stale System DB: it refuses pre-network with the structured
error. qfs 0.0.57→0.0.58.

## 5. Historical Analysis

Both defects validate the mission's live-round-as-verification-gate pattern: they lived exactly
where mocks stop — the Drive mock never carried a second row, and hermetic installs never leave
stale rows behind a hard break. The Chatwork fix follows the repo's honesty-over-compat line: no
silent legacy-body parsing (the §5.4 hard break stands); instead the failure became loud,
actionable, and self-healing on re-install. The Drive fix repeats the honest-count doctrine the
read side already enforces, now on the write side, and the newest-wins type lookup aligns the
eval path with the describe path's existing ordering rather than inventing a new rule.

## 7. Successful Development Patterns

- **Reproduce against the real state, hermetically**: the Chatwork root cause was found by
  reading this host's actual System DB rows (a pure local read), then locked by tests that pin
  the STALE shape itself — not just the happy path a fresh install produces.
- **Loud, actionable, self-healing over silent compat**: instead of parsing the legacy body
  (back-compat the project forbids), the failure names its own fix and the re-install genuinely
  heals via newest-wins ordering.
- **Validate the batch before the first write**: decoding all rows up front means a malformed
  row aborts with zero partial state; only genuine API failures can leave a partial batch, and
  those report exact progress.
- **Verify the error path with the real binary when it's free**: the pre-network refusal could
  be exercised against the live stale DB with zero cloud traffic — a live-fidelity check that
  cost nothing.
