## 1. Overview

This branch fixes the **8 defects the owner-attended live rounds surfaced** (recorded on the
knowledge branch merged as PR #36). Each was a silent wrong-answer, wrong-node-write, or
never-worked path that the hermetic suite could not reach because it never drove the live committer
against real service shapes. Every fix lands with a hermetic lock; several are verified by a
negative test (they fail without the fix). `v0.0.59 → v0.0.60`; plugin `0.11.2 → 0.11.3`.

**Highlights:**

1. The v0.0.59 headline feature works: the daemon sweeper now fires a real `CREATE JOB` (it never
   had — a writer/reader format drift failed every sweep).
2. Two silent wrong-writes closed: a Drive folder `UPDATE` no longer renames the container, and a
   Slack/GitHub `WHERE` no longer returns the whole collection.
3. The taught surfaces run true: `/local |> select content |> transform` type-checks, the Gmail
   read-terminal transform commits, and reasoning models complete instead of 400-ing / returning
   empty.

## 3. Changes

Severity-ordered, one commit per ticket; each carries a hermetic regression lock.

### 3-1. HIGH — the daemon sweeper never fired a real JOB ([b76b799](https://github.com/qmu/qfs/commit/b76b799))

`/server/jobs` stores the `DO` plan as the canonical serialized `PlanSpec` (AST-JSON), and the
trigger dispatcher + `qfs job run` already rehydrate it with `PlanSpec::from_canonical`. Only
`qfs_watchtower::cron::fire_one` re-lexed the JSON as statement text → `UNEXPECTED_CHAR` every 30s
sweep, so **no daemon-fired job had ever succeeded**. Now rehydrates via `from_canonical` (no
re-parse), matching every other reader; the test helpers build the `plan` column through the real
`PlanSpec::canonical()` serializer, and `sweep_once_with_the_live_committer_applies_a_real_local_write`
drives a real-format plan through the live committer into the local-FS applier.

### 3-2. HIGH — Drive `UPDATE` on a folder path renamed the folder ([ec67ae6](https://github.com/qmu/qfs/commit/ec67ae6))

`SET name … WHERE name` collapses to a bare `SET name` on the folder path (`setwhere_row_batch`
de-dups the WHERE key when it shares the SET column), and `decode_move` resolved the path node —
renaming the container. A name-path folder rename now refuses loudly, directing to the file node or
to `/drive/id:<id>`; file renames, folder moves (parents), and id-addressed folder renames are
unaffected. REMOVE was audited and is already per-row-correct.

### 3-3. MEDIUM — Slack/GitHub `WHERE` silently dropped ([65167c6](https://github.com/qmu/qfs/commit/65167c6))

`users.list` (and GitHub list APIs) have no server-side filter param, so the driver over-returns;
the facet dropped the residual and the `where_: true` profile suppressed an engine residual, so the
predicate was honored nowhere. The Slack and GitHub facets now enforce the pushed `WHERE` at the
seam via a shared `apply_pushed_filter` (t20 over-fetch-then-filter), idempotent where a backend
already narrowed (Drive `q`, Gmail `q`).

### 3-4. MEDIUM — `/local` plan-time schema omitted `content` ([d3289df](https://github.com/qmu/qfs/commit/d3289df))

A single-file runtime read carries `content`, but `describe()` returned metadata only, so
`|> select content |> transform` failed `UnknownColumn` at plan time (the cookbook PDF-extraction
recipe was unrunnable — round 5 needed a struct bypass). `describe()` now advertises the nullable
`content` column and directory listings carry a null `content`, so plan and runtime schemas agree.
The extraction test is re-pointed at the real schema; the cookbook recipe corrected to
`|> select content as blob |> transform extract`.

### 3-5. MEDIUM — read-terminal transform over Gmail failed at commit ([f2190f0](https://github.com/qmu/qfs/commit/f2190f0))

The read-terminal commit path read the source fine, then applied the whole plan — source `Read`
node included — to ledger consent, dispatching the `Read` to the Gmail write applier ("READ is not
serviced"). Now strips source reads before the consent-ledger apply, exactly as the write-terminal
path does. Locked by a negative test (a `world_apply` that rejects READ like Gmail).

### 3-6. MEDIUM — reasoning models broke the provider layer ([0bf514d](https://github.com/qmu/qfs/commit/0bf514d))

OpenAI now sends `max_completion_tokens` (reasoning models 400 on `max_tokens`); the `effort`
ceilings leave output room after thinking (`low` 256 → 1024); an empty completion returns an
actionable "raise effort" error instead of a misleading schema-mismatch.

### 3-7. LOWER — `/drive/id:<file-id>` was `invalid_path` ([1e546bd](https://github.com/qmu/qfs/commit/1e546bd))

`parse_str` recognised the `id:` selector only at the raw string start, not after the `/drive/`
mount. The mounted form now parses to `DrivePath::ById`, so a space-named file (which can't be a
path segment) is readable / renamable / trashable by id.

### 3-8. LOWER — Drive folder `UPSERT` lacked INSERT's per-row decode ([6ed2128](https://github.com/qmu/qfs/commit/6ed2128))

`decode_upsert` now routes a name-carrying row into `<folder>/<name>` (replace by id if it exists,
else create), so a multi-row folder UPSERT writes one file per row and the INSERT-collision advice
("use UPSERT to replace") works.

### 3-9. Plugin re-version ([8a1ada4](https://github.com/qmu/qfs/commit/8a1ada4))

Bumped all four plugin version fields to `0.11.3` so installed caches pick up the corrected
`qfs-cross-service` extraction recipe.

## 6. Concerns

### (new) Server-scheduling live acceptance still owed — re-run live round 9

The sweeper fix is hermetically proven, but the mission's server-scheduling acceptance needs the
owner-attended live re-run (`qfs serve` fires a 1m JOB within 90s, history survives restart). Code
landed; live proof pending.

### (new) `/drive` single-file `select content` still diverges

The `/local` schema-widen does not extend to gdrive: its file-content read schema
(`name/mime_type/size/md5/content`) is a different column set from the folder listing, so advertising
`content` in gdrive `describe()` needs the file-vs-folder schema unified first. The round-5 struct
bypass remains available for `/drive` until then.

### (new) `driver-fs` shares the `/local` content-omission

The `QFS_FS_<NAME>` named-roots driver has the identical `FsRow` metadata-only describe schema and
should get the same `content` widen when prioritized.

### (new) Per-row Drive folder rename needs a predicate/selector channel

`decode_move` refuses a folder rename rather than renaming the matching child, because a same-column
`SET name WHERE name` is unrepresentable in the flat effect row batch. Renaming the matching child
(instead of refusing) needs the effect representation to carry the WHERE selector distinct from the
SET payload.

### (carried, updated from PR #35) Policy-less or denied job re-fires every sweep

Now REACHABLE: with the sweeper firing, a denied/policy-less job's not-stamped semantics means it
re-fires each sweep. Review the denied-job backoff once the live round exercises real denials.

## 7. Successful Development Patterns

- **Reproduce-then-fix with a negative test.** For each subtle path (the sweeper format, the Gmail
  read-terminal apply) the lock was verified to FAIL without the fix and PASS with it — proving the
  test actually guards the regression, not just the happy path.
- **Fix the class, not just the instance.** The Slack `WHERE`-drop fix also closed the identical
  GitHub twin at the same seam; the `/local` content-schema fix re-pointed the faked extraction test
  onto the real schema so the divergence can't hide again.
- **Honest scoping.** Where a full fix needed a deeper change (gdrive file/folder schema unification,
  a predicate channel for same-column SET/WHERE), the safe correct fix landed and the remainder was
  recorded as a scoped concern rather than half-done.
