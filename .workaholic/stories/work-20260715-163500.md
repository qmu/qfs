## 1. Overview

This branch prepared the repository's tree for publication. qfs is private and going public as a **fresh, squashed repository**, which means the tree stops being a snapshot of the work and becomes the *entire published artifact* — nothing outside it can be appealed to afterward, and publishing is a one-way door.

The driving ticket (`20260715151000`) named one leak visible on `main` and asserted a second lived only in history. A four-agent read sweep of the whole tree found **three more that no grep, scanner, or denylist would have surfaced**, and corrected the ticket's premise about the second.

**Highlights:**

1. **The ticket's premise was half wrong, in the dangerous direction.** It cites `be5cfbd` as history-only. That commit carried *two* leaks with different lifetimes: a RESUME checkpoint (deleted later, genuinely gone) **and** a live Cloudflare zone list written into `docs/cookbook/cloudflare.md`, which was still on `main` and had been copied into the shipped `qfs-cloudflare` skill. The squash would not have removed it. Acting on the ticket as written would have published it.
2. **A client's name was in a Rust test fixture**, captured verbatim from live Gmail as a label with the 様 honorific — in the driver source and its doc comments, plus the archived ticket that recorded the bug.
3. **Real document names from a live Drive listing** sat in four archived tickets; one names an insurance-industry deliverable which, combined with the label above, identifies the engagement.
4. The step-6 denylist is now git-ignored (`.gitignore` had no entry — creating the list of external names would itself have published them), and it **caught the report of the leaks re-leaking them**: the first draft of the ticket's findings table quoted every value into a file that publishes.
5. Steps 4 and 5 — create the fresh repository, push, verify from a clone — are **deliberately not done**. They are the owner's, because they cannot be undone.

## 2. Motivation

Publishing exposes the whole tree at once and forever. The ticket's own framing is the right one: prefer a shape where the unwanted content *cannot be present* (a fresh repo) over one that relies on removal having been thorough (a history rewrite, whose pre-rewrite commits stay retrievable by SHA until GitHub Support garbage-collects them).

But that guarantee only covers *history*. It says nothing about the tree, and the tree is what ships. The ticket knew this — step 2 asks for a **reading** sweep, with the explicit warning that "a grep only finds terms already known… the things that actually leak (a resource name, a label, a filename) are never on a list beforehand." That warning turned out to describe every real finding on this branch.

## 3. Changes

Four `Explore` agents read disjoint slices of the tree — `tickets/archive/` (298 files), the rest of `.workaholic/`, `docs/` + README + CI, and the source tree + fixtures + plugins — each asked one question: *would a stranger learn something about someone else's systems from this?*

Every removal preserved the claim the detail supported. The Gmail fixture still exercises exactly what it did — a user label, a non-ASCII display name, `id != name` — under a synthetic value; the live-verification notes still say what was verified, without saying whose data it was.

**Verified as safe rather than assumed:** the checkpoint's resource names (the real Cloudflare account id, the client's KV/D1/worker names) appear **nowhere in the tree**, term by term — so for *that* content the ticket's history-only premise does hold, and the squash removes it. `watchtower` matches 83 files but is qfs's own crate, not the client D1 of the same name. `a@qmu.jp`'s 329 occurrences are overwhelmingly ticket-frontmatter `author:` fields — the same fact as commit authorship, left alone deliberately, since rewriting them to `you@example.com` would falsify authorship records for no gain.

## 5. Historical Analysis

The repo already had real leak defenses, and they all held while missing everything that mattered:

- `packages/qfs/scripts/check-no-live-credentials.sh` runs in `release.yml` and greps credential *value prefixes* (`ghp_`, `xox*-`, `AKIA`, JWTs). It correctly finds nothing — it is structurally blind to real hostnames, labels, and filenames.
- The golden tests assert redaction against planted canaries (`xoxb-PLANTED-CANARY-should-never-serialize`). Also correct, also blind to this class.
- The zone list passed CI cleanly for a week, from `be5cfbd` (2026-07-08) to now.

The pattern: **every defense here matches shapes, and none of these leaks had a shape.** A client's name is just a word. A zone list is just a table. A filename is just a filename. They are only leaks in context — which is why the sweep had to be a read, and why the denylist added in step 6 is honestly labelled in its own header as a backstop for known terms that would have caught none of the findings that motivated it.

## 6. Concerns

- **Steps 4 and 5 remain open, and they are the irreversible ones.** The fresh repository must be created and pushed by the owner, and step 5's verification must run against a **fresh clone**, not the working tree — the working tree is not what readers get.
- **The published ticket maps the private repo.** Under the step-3 decision, `20260715151000` publishes; it names `be5cfbd`/`fdc7791` as holding client resource names in the retained private repository. The owner judged this marginal (that repo stays private, and the names are not reproduced), but it is a deliberate acceptance, not an oversight.
- **`.workaholic/` publishing is now a standing obligation, not a one-time sweep.** Every future ticket, story, and live-round note lands in the public tree. The live-connected Gmail/Drive/Slack accounts that produced three of this branch's findings are still connected.
- **The scanner gap is unclosed.** `check-no-live-credentials.sh` remains value-prefix-only. If a durable guard is wanted before going public, that script is where a hostname/domain check belongs — but see the Historical Analysis: it would not have caught this branch's findings either.
- **A pre-existing schema violation was normalized in passing:** an archived ticket carried `effort: M`, which the ticket-validation hook rejects (valid: `0.1h`–`4h` or empty). Set to empty rather than invented as `2h`, since mapping M→2h would fabricate a record.

## 7. Successful Development Patterns

- **Verify the premise, not just the steps.** The ticket was specific, confident, and wrong about which leak was where. The check that caught it cost one `git show --stat`: *does this commit's content survive in the tree?* Citing a commit as history-only is not the same as checking.
- **Read where grep cannot reach.** Four agents with one question each — not a term list — found what a week of CI, canary tests, and a credential scanner did not. The instruction that did the work was the question, not the pattern.
- **Then grep with what the reading taught you.** The checkpoint's contents were only readable *from history*; extracting its terms and re-running them over the tree is what turned "the ticket says this is gone" into "verified gone, term by term."
- **A report about a leak can re-leak it.** The findings table's first draft quoted every value into a file that publishes. The step-6 denylist — written minutes earlier, and honestly labelled as unable to catch novel leaks — caught this one, because by then the terms *were* known. That is exactly and only what a denylist is for.
