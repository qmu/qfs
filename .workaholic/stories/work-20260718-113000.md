## 1. Overview

The branch implements the qfs half of the address closure settled in the plan book on 2026-07-18: **a row is a node** — describe declares which columns select a child, and the path grammar itself learns the selection segment `@`.

The trigger was the developer driving the viewer by hand and finding that a mail INBOX would not drill inward — and that the failure was not mail-specific. The consumer's only click rule keyed on a `path` column that `append_log` rows (`id`, `thread_id`) and `blob_namespace` rows (`id`, `name`, `parents`) do not carry, so drilling worked for `/sys/*` and `/markdown/*` and essentially nothing else. The settled rulings: describe declares the key (the synthetic-path-column option rejected), **qfs owns the `@` lexer and the single lowering site**, composite keys are positional in declared key order, and secondary axes are relation segments for a later phase.

**Highlights:**

1. `ChildAddress` on `NodeDesc`/`DescribeReport` — `{"kind":"key","columns":[…]}`, `{"kind":"entry_name","column":…}`, or `{"kind":"none"}`, defaulting to **none**: "no child" is a declared, valid answer
2. `@` selection segments lex in `qfs-lang` (following the v0.0.77 quoted-segment shape; `name@version` pins untouched; `/x/@` refused)
3. One lowering site — `plan_pipeline` — turns `/x/@A` into `|> where <declared key> == A`; every read consumer (run, REPL, server, MCP, views) funnels through it
4. **Effect positions refuse unlowered selections** — the red test proved `REMOVE /db/users/@1` previously built a plan silently targeting the containment path `/db/users/1`
5. Key declarations across the drivers: SQL primary keys from the catalog, gmail `id`, slack `ts`/`id`, `/sys/paths` and markdown `path`, blob entry-name-as-containment, `/rest` `{value}` → none
6. `describe /x/@A` resolves to the row-node view in both the one-shot and the REPL

## 2. Motivation

Four faces — the pipeline, the viewer, the REPL, REST/MCP — are interfaces to the same language, and the plan book now says so. What kept them from behaving that way was that a query's result rows were not addressable: the space was not closed under the language's own operations. Closing it requires exactly two things from qfs — say which columns are a row's identity (describe), and accept that identity as a path segment (the lexer and one lowering). Everything else is a consumer reading the declaration.

The ordering was itself a settled constraint: the one-lowering rule meant qfs had to learn `@` before any consumer grew its own lowering, or the plan's no-drift rule would be violated for the whole window.
