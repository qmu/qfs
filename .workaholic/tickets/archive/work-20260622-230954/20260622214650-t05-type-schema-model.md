# Type & schema model (rows, struct/array, typed columns)

## Overview

This ticket delivers the **row/relation type model** that the whole `qfs` engine
computes over: the static description of a relation (typed, possibly-nested columns)
and the runtime values that flow through a pipeline. It implements **RFD §4 (Data &
type model)** and the schema half of **RFD §5 (driver declares schema; powers
`DESCRIBE`)**.

Everything downstream depends on it: `EXPAND <field>` explodes a `struct`/`array`
column into rows; path access `a.b.c` navigates `struct`s without flattening; typed
predicates (`= <> < > LIKE ~ IN BETWEEN`) need column types to type-check and to drive
pushdown; codecs (`DECODE`/`ENCODE`) bridge `bytes ↔ rows` and so must target this row
model. Because a `qfs` query is **pure** (RFD §3 purity invariant), the type model is
the contract that lets us type-check and `PREVIEW` a statement without touching the
World. Types are **sourced from the backend** (DB catalog, JSON/YAML inference, codec)
and must survive **heterogeneous union** (null/sparse columns across sources).

## Scope

In-scope:
- A `Schema` (ordered, named, typed columns) and a `ColumnType` sum type covering
  scalars, `Struct`, `Array`, and an open `Unknown`/`Json` escape for irregular data.
- A runtime `Value`/`Row` representation matching `ColumnType`, with explicit nulls.
- **Owned DTOs only** — no vendor/SDK types leak into the type model (RFD §9).
- Schema algebra used by the planner: `project`, `expand(field)`, path-type resolution
  `a.b.c`, and **schema unification** for `UNION`/heterogeneous sources (null/sparse).
- Typed predicate model: a `Predicate` AST node carrying resolved column types, plus
  type-checking of comparison operators against `ColumnType`.
- Type-check entry point that decorates the parsed AST (from t04) with column types and
  emits **structured** type errors (AI-consumable, RFD §5 "structured error").

Out-of-scope (deferred):
- Concrete codec implementations (`DECODE json/yaml/csv/markdown+frontmatter`) — they
  *produce* a `Schema`/`Row`, but live in the **codec registry** ticket (E3).
- `@version` temporal coordinate semantics (RFD §4) — parsing/typing of `@ref` is the
  path/addressing ticket; we only reserve a `version` field on column provenance.
- Pushdown decisions and the local combine engine (RFD §6) — E2/E3.
- Effect-plan node typing beyond "an op maps `Schema -> Schema`" — runtime ticket (E2).
- Per-driver schema declaration wiring — driver-contract ticket (E4); we define the
  trait surface only.

## Key components

New crate/module: `qfs-types` (or `core::types`), pure, no I/O, `wasm32`-safe.

```rust
pub struct Schema { pub columns: Vec<Column> }
pub struct Column { pub name: Name, pub ty: ColumnType, pub nullable: bool, pub provenance: Provenance }

pub enum ColumnType {
    Bool, Int, Float, Decimal, Text, Bytes, Timestamp, Date, Uuid,
    Struct(Schema),            // nested record; path access a.b.c walks this
    Array(Box<ColumnType>),    // EXPAND target
    Json,                      // deeply-irregular JSON stays a struct/json column (RFD §4)
    Unknown,                   // inferred-but-unresolved (sparse heterogeneous source)
}

pub struct Provenance { pub driver: Option<DriverId>, pub source_col: Option<String> }

// Runtime values mirror ColumnType; Null is explicit and orthogonal to type.
pub enum Value { Null, Bool(bool), Int(i64), Float(f64), Text(String),
                 Bytes(Vec<u8>), Timestamp(i64), Struct(Row), Array(Vec<Value>), Json(JsonValue) }
pub struct Row { pub values: Vec<Value> }  // positional, aligned to Schema.columns
```

Schema algebra (pure functions / `impl Schema`):
- `fn project(&self, names: &[Name]) -> Result<Schema, TypeError>`
- `fn resolve_path(&self, path: &[Name]) -> Result<&ColumnType, TypeError>` (struct nav)
- `fn expand(&self, field: &Name) -> Result<Schema, TypeError>` — `Array(T)` → column of
  `T` (and `Struct` fields flattened into row columns); rejects non-collection fields.
- `fn unify(a: &Schema, b: &Schema) -> Result<Schema, TypeError>` — column-wise least
  upper bound for `UNION`; missing columns become `nullable`, scalar/`Unknown` widen,
  irreconcilable types widen to `Json` rather than error (heterogeneous tolerance).

Predicate typing:
```rust
pub enum Predicate { Cmp(ColRef, CmpOp, Literal), And(Box<_>, Box<_>), Or(..), Not(..),
                     In(ColRef, Vec<Literal>), Between(ColRef, Literal, Literal), Like(ColRef, Pattern) }
pub fn typecheck_predicate(p: &Predicate, s: &Schema) -> Result<TypedPredicate, TypeError>;
```

Traits (surface only; impls in E4):
```rust
pub trait Typed { fn schema(&self) -> &Schema; }              // a relation node
pub trait SchemaSource { fn describe(&self, path: &Path) -> Result<Schema, DescribeError>; }
```

`TypeError` is a structured enum (`UnknownColumn{name, available}`, `NotAStruct`,
`NotExpandable`, `IncomparableTypes{op,lhs,rhs}`, …) carrying enough context for an AI
agent to repair the statement — matching the closed-core/structured-error policy (t04).

## Implementation steps

1. Scaffold `qfs-types` crate: no I/O deps, `#![forbid(unsafe_code)]`, `serde` on DTOs,
   confirm it compiles for `wasm32-unknown-unknown`.
2. Define `ColumnType`, `Column`, `Schema`, `Provenance`; derive `Clone, Debug, PartialEq`
   and `Serialize/Deserialize`.
3. Define `Value`/`Row`; add a `Value::type_of() -> ColumnType` and a debug assertion that
   a `Row` conforms to a `Schema` (used in tests, not the hot path).
4. Implement `resolve_path` (struct navigation) + `project` with structured errors.
5. Implement `expand(field)` for `Array` and `Struct` per RFD §4 ("same operator for mail
   attachments and JSON arrays").
6. Implement `unify` (UNION/heterogeneous): nullable promotion, scalar widening, fallback
   to `Json`; table-driven widening matrix.
7. Define `Predicate`/`TypedPredicate` and `typecheck_predicate` with the comparability
   matrix (e.g. `Int < Text` ⇒ `IncomparableTypes`); `LIKE/~` require `Text`.
8. Add the AST decoration entry point `typecheck(stmt, root_schema) -> TypedStmt` that maps
   each op `Schema -> Schema` and threads errors with source spans (from t04 AST).
9. Define `Typed`/`SchemaSource` trait surface; provide an in-memory test `SchemaSource`.
10. Golden tests + property tests (below); wire `clippy -D warnings` and `cargo deny`.

## Considerations

- **Owned DTOs / no vendor leak (RFD §9):** the type model must be expressible without any
  driver SDK type. Drivers map their catalog into `Schema` at the boundary; the engine
  never sees a vendor type. Keep `qfs-types` dependency-free beyond `serde`.
- **Heterogeneous / sparse data is the hard part.** A `UNION` of Gmail rows and SQL rows,
  or two JSON documents with different shapes, must not error. Resolution: `unify` promotes
  to `nullable`, widens scalars, and **degrades to `Json` instead of failing** — keeping the
  pipeline runnable while preserving structure where it agrees. Document the widening matrix.
- **Irregular JSON:** per RFD §4, deeply-irregular JSON stays a `Struct`/`Json` column; do
  **not** attempt full flattening. `resolve_path` works structurally; unresolved paths into
  `Json` yield `Unknown` (still queryable, late-bound), not a hard error.
- **Purity & determinism:** every function here is pure and total over its inputs — no I/O,
  no clock, no RNG. This is what lets `PREVIEW`/type-check run with **no live creds** and be
  golden-tested deterministically. `SchemaSource::describe` is the only effectful seam and is
  out-of-scope to implement here.
- **Capability/least-privilege boundary:** types feed `DESCRIBE` and predicate pushdown but
  carry **no** secrets and no capability state; provenance records *driver id only*, never
  tokens — nothing here is logged that could leak credentials (RFD §10).
- **Observability:** `TypeError` is structured (not stringly) so the server/CLI can surface
  it in audit logs and to AI agents; include `available` columns on `UnknownColumn`.
- **Coding standards:** sum types over flags (RFD §9 "model … with enums"), exhaustive
  `match` (no catch-all that hides a new variant), one module per concept
  (`schema.rs`, `value.rs`, `predicate.rs`, `unify.rs`, `error.rs`), doc every public item.

## Acceptance criteria

- `cargo build`, `cargo build --target wasm32-unknown-unknown`, and
  `cargo clippy -- -D warnings` are green for `qfs-types`.
- Unit/golden tests pass:
  - `resolve_path(["a","b","c"])` returns the nested type for a `Struct`; `NotAStruct`
    and `UnknownColumn{available}` errors are produced and snapshotted.
  - `expand` turns `Array(Struct{..})` and `Array(Int)` into the documented output schema;
    expanding a scalar column yields `NotExpandable`.
  - `unify` of two divergent schemas produces nullable/widened columns and `Json` fallback
    — covered by a golden snapshot of the resulting `Schema`.
  - `typecheck_predicate` accepts type-correct comparisons and rejects `Int < Text`,
    `LIKE` on non-`Text`, with the exact `TypeError` variant asserted.
- **Plan/type assertions:** `typecheck(stmt, root_schema)` over a fixture pipeline
  (`FROM … |> WHERE … |> EXPAND attachments |> SELECT a.b.c`) yields the expected output
  `Schema`; assertion is on the *decorated AST/schema*, requiring **no live creds**.
- Property test: `Row` generated from any `Schema` conforms to it (`type_of` round-trips);
  `unify(a,a) == a` (idempotence) and `unify` is commutative up to column order.
- All public items documented; `cargo deny` (licenses/advisories) passes; no `unsafe`.
