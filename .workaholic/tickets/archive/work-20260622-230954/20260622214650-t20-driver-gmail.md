# Driver: Gmail

## Overview

This ticket delivers the **Gmail driver** — the first real `Driver` (RFD §5) in epic E4,
and a flagship of the whole thesis (RFD §1): the Gmail mailbox is exposed as one mount under the uniform VFS + pipe-SQL DSL. It mounts the mailbox at
`/mail/...` and maps it onto the **Append/log archetype** (RFD §5): **labels = directories,
messages = files, attachments = nested entries** addressed by `id:` or `id:thread:<id>`.

It implements the driver-contract seam from t13 against a live Google backend: `describe`
powers `DESCRIBE /mail/<label>`, `capabilities(path)` gate verbs at parse time (RFD §5),
`procedures()` declares the **irreducible** `mail.send` transition, and `prelude()` ships the
pure `SEND` alias (RFD §3). It honours the **purity invariant** (RFD §3): every write —
`INSERT INTO /mail/drafts`, `UPSERT`, `REMOVE` (trash), `CALL mail.send` — **constructs a
`Plan` node and performs no I/O during planning**; only the `applier()` under `COMMIT` touches
the Gmail API. Auth (OAuth tokens, multi-account, scopes) comes from t19; this driver requests
**least-privilege scopes** `gmail.modify` + `gmail.compose` only (RFD §10), never the full
mail scope.

## Scope

In-scope:
- `/mail` mount: virtual root lists **labels** (dirs); `/mail/<label>` lists **messages**
  (files); `ls id:<msg>` / `/mail/<label>/<msg>/` lists **attachments** (nested).
- `SELECT` over a label (with `WHERE`/`LIMIT`/`ORDER BY` pushed to Gmail search `q=`);
  `get` (message → `.eml`/`.txt`, thread → `.mbox`) as blob read.
- `REMOVE` = **trash a single message** (`id:`) or a whole thread (`id:thread:`) — the
  `gmail.modify` trash op, not permanent delete.
- `INSERT INTO /mail/drafts` (create draft) and `UPSERT INTO /mail/drafts` (create-or-replace
  by draft id) — **multipart-MIME build** when `attachments` are present.
- `UPDATE /mail/<label>` for label add/remove (modify message labels).
- `CALL mail.send` — the **irreversible** procedure; prelude alias `SEND` desugars to it.
- Owned DTOs for `MailMessage`/`MailDraft`/`Attachment`; thin HTTP client (no `google-apis`
  SDK), `wasm32`-friendly request layer.

Out-of-scope (deferred):
- OAuth flow, token store, multi-account selection, refresh — **t19** (consumed here).
- `@version`/historyId incremental sync, push (`watch`) webhooks → **E7 server / trigger**
  sibling; this driver declares `VersionSupport::None` for now.
- `.eml`/`.mbox` ↔ rows codec beyond what t15 provides; we emit/consume blob bytes and lean
  on the **codec registry** (t15) for any structured decode.
- Calendar/Drive drivers (sibling E4 tickets t21/t18); only Gmail here.
- Permanent delete (`messages.delete`) — intentionally unsupported (RFD §10 blast radius);
  only trash.

## Key components

New crate `drivers/qfs-driver-gmail` implementing `Driver` (t13). No vendor SDK; a thin
`reqwest`/`http` client behind a `GmailClient` trait so it mocks in tests and builds for
`wasm32`. Owned DTOs only — Gmail JSON never leaks past the driver boundary (RFD §9).

```rust
pub struct GmailDriver { client: Arc<dyn GmailClient>, auth: Arc<dyn TokenSource> } // auth from t19

impl Driver for GmailDriver {
    fn id(&self) -> DriverId { DriverId("mail") }
    fn describe(&self, p: &VfsPath) -> Result<NodeDesc, DriverError>; // Append archetype + Schema
    fn capabilities(&self, p: &VfsPath) -> Capabilities;             // per-path (drafts vs label)
    fn procedures(&self) -> &[ProcSig];   // [mail.send { irreversible: true, requires_scopes }]
    fn prelude(&self) -> &[AliasDef];     // SEND(d) = d |> CALL mail.send
    fn version_support(&self, _: &VfsPath) -> VersionSupport { VersionSupport::None }
    fn applier(&self) -> &dyn PlanApplier; // the only impure seam (COMMIT)
}

// Owned DTOs (no google types)
pub struct MailMessage { pub id: String, pub thread_id: String, pub label_ids: Vec<String>,
                         pub date: OffsetDateTime, pub from: String, pub subject: String,
                         pub snippet: String, pub attachments: Vec<Attachment> }
pub struct MailDraft { pub id: Option<String>, pub to: Vec<String>, pub cc: Vec<String>,
                       pub subject: String, pub body: Body, pub attachments: Vec<Attachment> }
pub struct Attachment { pub filename: String, pub mime: String, pub bytes: Vec<u8> }

// MIME builder — pure bytes, no I/O
pub fn build_mime(d: &MailDraft) -> Result<Vec<u8>, MimeError>;  // multipart/mixed when attachments
```

- Path → request mapping: `MailPath::{ Root, Label(name), Message(id), Thread(id),
  Attachment(msg,att), Drafts }` parsed from `VfsPath` + `id:` addressing.
- `Capabilities` are **path-keyed**: `/mail/<label>` = `Select|Update|Remove|Ls`;
  `/mail/drafts` = `Insert|Upsert|Select|Remove`; messages are read/trash-only.
- Effect nodes produced into the t09 `Plan`: `InsertDraft`, `UpsertDraft`, `TrashMessage`,
  `ModifyLabels`, and a `Call(mail.send)` tagged `irreversible: true`.
- `ProcSig { name: "mail.send", irreversible: true, requires_scopes: [gmail.compose] }`.
- Pushdown: `PushdownProfile::Partial { where_: true, limit: true, order: false, .. }` —
  Gmail search `q=` covers many `WHERE` predicates; residual filters combine locally (RFD §6).

## Implementation steps

1. Scaffold `drivers/qfs-driver-gmail`; depend on `qfs-core` (DTOs/Plan), `qfs-driver`
   (trait/registry), and t19 `TokenSource`. Forbid google SDK deps via workspace lint.
2. Implement `GmailClient` trait (list-labels, list/search messages, get message/thread raw,
   get attachment, create/update draft, send, trash, modify labels) + a `reqwest` impl behind
   it; inject the t19 token + account into request headers.
3. Parse `VfsPath`/`id:` into `MailPath`; implement `describe` (Append archetype, message
   `Schema`: id, thread_id, date, from, subject, snippet, label_ids, attachments[struct]).
4. Implement `capabilities(path)` (path-keyed) and `procedures()`/`prelude()` (the `SEND`
   alias `AliasDef`).
5. Implement `SELECT`: translate `WHERE`/`ORDER BY`/`LIMIT` to Gmail `q=`/`maxResults`;
   **batch** the N message-detail fetches into the planner so the engine collapses N+1 (RFD §6).
6. Implement `build_mime` (RFC 5322 / `multipart/mixed`, base64url body, per-attachment parts).
7. Build effect nodes for `INSERT/UPSERT INTO /mail/drafts`, `UPDATE` (labels),
   `REMOVE` (trash single message / thread), and `CALL mail.send`; all pure → `Plan`.
8. Implement the `PlanApplier` half: apply each node via `GmailClient` under `COMMIT`,
   returning applied-effect records for the audit ledger (t12); `UpsertDraft` is retry-safe.
9. Register the driver mount in the `DriverRegistry` (RFD §3 paths registry).
10. Tests: capability golden, `DESCRIBE` snapshot, MIME golden, plan-shape tests with a mock
    `GmailClient` (no live creds); one opt-in live smoke test gated behind an env flag.

## Considerations

- **Least-privilege & secrets (RFD §10):** request **only** `gmail.modify` + `gmail.compose`
  (no full `mail.google.com`, no permanent-delete scope). Tokens come from t19, injected at
  construction; never logged, never in DTOs, never in plan output.
- **Irreversibility (RFD §3/§10):** `mail.send` and trash are where `PREVIEW` + `POLICY` earn
  their keep. `CALL mail.send` / `SEND` must surface `irreversible: true` so `PREVIEW` shows
  "1 message will be SENT (irreversible)" and policy can block it under unattended execution.
- **Idempotency / recovery (RFD §6):** `mail.send` is not naturally idempotent — guard with a
  client-side de-dupe key (draft id one-shot send: create-draft → `drafts.send` by id, so a
  retry re-sends the *same* draft id rather than a fresh message). Prefer `UPSERT` for drafts
  (retry-safe). Record every applied effect to the audit ledger (t12) for reconstruction.
- **N+1 batching (genuinely hard part):** Gmail `messages.list` returns ids only; naive detail
  fetch is N+1. Resolve by emitting detail fetches as independent plan leaves so the t10
  interpreter auto-parallelizes / batches them (the exact RFD §6 example). Do not loop-fetch
  inside `SELECT`.
- **MIME correctness (hard part):** header encoding (RFC 2047 for non-ASCII subjects),
  base64url (not standard base64) for the Gmail `raw` field, correct `Content-Type`/boundary
  for `multipart/mixed`, and CRLF line endings. Cover with golden tests against fixed inputs.
- **Owned DTOs / no vendor leak (RFD §9):** translate Gmail JSON into `MailMessage` etc. at the
  client boundary; the trait surface and `Plan` carry zero google types — enforced by a
  dep-graph test.
- **Observability:** per-leg timeouts, bounded retries, circuit breaker on the HTTP client;
  structured `DriverError` (path, verb, http status) for the audit log and AI feedback.
- **Coding standards:** driver in its own crate under `drivers/`; HTTP behind a mockable trait;
  no `async` in introspective methods (keep `describe`/`capabilities` `wasm32`-pure).

## Acceptance criteria

- `cargo build` + `cargo clippy -- -D warnings` green; the introspective half builds for
  `wasm32-unknown-unknown`.
- **No-vendor-deps** dep-graph test passes (no google-apis SDK in the crate's surface).
- **Capability golden:** `UPDATE /mail/drafts` or `INSERT INTO /mail/<label>/<msg>` is rejected
  at resolve time with a structured `CapabilityError` listing supported verbs; `REMOVE id:<msg>`
  on a message resolves.
- **`DESCRIBE` snapshot:** describing `/mail/<label>` emits Append archetype + the message
  schema columns; `procedures()` shows `mail.send` tagged `irreversible`.
- **MIME golden:** `build_mime` for a draft with a non-ASCII subject + two attachments produces
  a byte-stable `multipart/mixed`, base64url-encoded message (snapshot).
- **Plan assertions (no live creds, mock client):**
  - `FROM /mail/INBOX |> WHERE from = 'x@y' |> LIMIT 5` plans a search with `q=` pushdown and N
    independent detail-fetch leaves (no inline loop).
  - `INSERT INTO /mail/drafts VALUES (...)` plans one `InsertDraft`; the equivalent `UPSERT`
    plans a retry-safe `UpsertDraft` keyed by draft id.
  - `REMOVE id:<msg>` plans exactly one `TrashMessage` (single message, not the thread).
  - `... |> CALL mail.send` and the alias `... |> SEND` plan the **same** `Call(mail.send)` node
    tagged `irreversible`; an undeclared `CALL mail.foo` is rejected structurally.
  - `PREVIEW` of any of the above performs **no I/O** (mock client asserts zero calls).
- One opt-in, env-gated live smoke test (create draft → send to self → trash) passes when real
  creds are present, and is skipped (not failed) otherwise.
