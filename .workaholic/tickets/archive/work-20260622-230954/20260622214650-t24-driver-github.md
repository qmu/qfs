# Driver: GitHub

## Overview
Delivers the GitHub driver — the canonical **object-graph + workflow** archetype
(RFD §5 table, row 4) and the second consumer (after Slack/mail) of the universal-CRUD +
namespaced-procedure split (RFD §3). It mounts GitHub under `/github/...` as a path tree
(RFD §3 "paths" registry), maps issues/pulls/comments/reviews/runs/releases/files/branches
to nodes with per-node archetypes and schemas (RFD §5), and exposes the irreducible state
transitions GitHub has no universal verb for — `merge`, `dispatch`, `review` — as
`CALL github.*` procedures (RFD §3 "Irreducible state transitions"). It implements the
`Driver` contract from #13 and reuses the generic HTTP/REST plumbing from #18; it adds
**zero keywords** (RFD §3 closed core). Every write evaluates to a Plan, never I/O
(RFD §3 purity invariant), so PREVIEW/COMMIT and dry-run-in-CI work uniformly (RFD §10).
This is what lets an AI agent operate GitHub through the same `DESCRIBE → statement →
PREVIEW → COMMIT` loop it uses everywhere else (RFD §1).

## Scope
In scope:
- `/github/{owner}/{repo}/...` mount with namespaces: `issues`, `pulls`, `comments`,
  `reviews`, `runs`, `releases`, `files`, `branches`.
- Universal CRUD over the REST v3 API: `SELECT` (list/get + pagination), `INSERT INTO`
  (open issue/PR, post comment, create release/branch), `UPDATE` (close = `UPDATE SET
  state='closed'`, edit title/body/labels), `REMOVE` (delete comment/release/branch).
- Procedures: `CALL github.merge(method=>'squash'|'merge'|'rebase')`,
  `CALL github.dispatch(workflow=>…, ref=>…, inputs=>{…})`,
  `CALL github.review(event=>'APPROVE'|'REQUEST_CHANGES'|'COMMENT', body=>…)`.
- Capability map, owned DTOs, capability-gated parse-time rejection, pushdown of
  server-side filters (state/labels/assignee), token via credential store.

Out of scope (deferred):
- Versioned-blob FS over repo contents, `commits` relational history, mutable `refs`,
  `INSERT INTO /git/.../commits` — these belong to the **git** driver (#26). `files`/
  `branches` here are GitHub-API metadata views (read + create/delete branch ref), not a
  working tree.
- GraphQL v4 backend (REST-only for v1; revisit if pushdown demands it).
- `CREATE WEBHOOK`/inbound webhook receipt and event TRIGGERs (#E7 server).
- GitHub Apps / fine-grained installation-token flow (#E5 auth); v1 uses a PAT.

## Key components
New crate-internal module `drivers/github/` (lib crate `qfs-drivers`, per RFD §9
directory standards):
- `mod.rs` — `pub struct GitHubDriver { http: RestClient, creds: CredHandle }` implementing
  the `Driver` trait from #13:
  `fn namespace(&self) -> &Namespace`, `fn capabilities(&self, node: &PathNode) -> Capabilities`,
  `fn plan_read(&self, q: &Query) -> Result<Plan, PlanError>`,
  `fn plan_write(&self, w: &WriteOp) -> Result<Plan, PlanError>`,
  `fn plan_call(&self, c: &CallOp) -> Result<Plan, PlanError>`,
  `fn procedures(&self) -> &[ProcSig]`, `fn prelude(&self) -> &[AliasFn]`.
- `schema.rs` — static `Namespace` tree + per-node `Schema` (columns powering `DESCRIBE`)
  for the eight namespaces; `Archetype::ObjectGraph` nodes.
- `dto.rs` — owned DTOs (`IssueDto`, `PullDto`, `CommentDto`, `ReviewDto`, `RunDto`,
  `ReleaseDto`, `BranchDto`) with `serde` derive; **no octocrab/vendor types leak past the
  driver boundary** (RFD §9). `From<DtoX> for Row` converters.
- `procs.rs` — `ProcSig` declarations for `merge`/`dispatch`/`review` with typed named
  params (`method`, `workflow`, `ref`, `inputs`, `event`, `body`); each `plan_call` builds
  an `Effect::HttpCall` node (POST `.../merge`, `.../dispatches`, `.../reviews`) marked
  `irreversible: true` for merge/dispatch.
- `pushdown.rs` — translate `WHERE` predicates on listable nodes into REST query params
  (`state`, `labels`, `assignee`, `per_page`); residual predicates fall back to local eval.
- `prelude.rs` — optional pure alias `fn MERGE(d) = d |> CALL github.merge` (receiver-typed,
  in scope only for `/github` plans; ambiguity falls back to qualified `CALL`).

## Implementation steps
1. Scaffold `drivers/github/` under `qfs-drivers`; register `GitHubDriver` in the driver
   registry behind the mount prefix `github`.
2. Define `schema.rs`: the `Namespace` tree and `Schema`/columns for all eight nodes; wire
   `DESCRIBE`.
3. Define `dto.rs` owned DTOs + `From<_> for Row`; confirm no vendor types in public sigs.
4. Implement `plan_read`: list/get with cursor pagination expressed as a batched
   `Effect::HttpFetch` set (Haxl-style batching from #18) so N pages collapse in the planner.
5. Implement `pushdown.rs` predicate→query-param translation; return residual for local eval.
6. Implement `plan_write`: `INSERT INTO .../comments` (post), `INSERT INTO .../issues`
   (open), `UPDATE … SET state='closed'` (close), label/title edits, `REMOVE` deletes;
   each yields the right verb + endpoint, `RETURNING` the created/updated row.
7. Implement `procs.rs` + `plan_call` for `merge`/`dispatch`/`review`; set `irreversible`.
8. Implement `capabilities(node)` so unsupported verbs (e.g. `UPDATE` on `runs`) are
   rejected at parse time with a structured error (RFD §5).
9. Wire credential read from the encrypted store (#E5 contract); never log the token.
10. Add `prelude.rs` `MERGE` alias; register in driver prelude.
11. Golden plan tests + clippy/fmt; no live API calls in CI.

## Considerations
- **Least-privilege & secrets** (RFD §10): PAT read from the credential store at plan-apply
  time only; redact in logs/audit; the driver advertises which verbs each node supports so a
  `POLICY` can scope a handler to e.g. comment-only.
- **Idempotency/recovery** (RFD §6): `merge`/`dispatch` are at-most-once and irreversible —
  mark `irreversible: true` so PREVIEW always surfaces them and COMMIT requires explicit
  confirmation. Use optimistic concurrency on merge via the PR head SHA (`sha=>`) to avoid
  merging a stale ref; comment posts are not idempotent — document at-least-once risk, do not
  silently retry POSTs without a dedup key.
- **The genuinely hard parts**: (a) keeping the GitHub `files`/`branches` views strictly
  distinct from the git driver (#26) — here they are API-metadata reads + ref create/delete,
  not a content tree; document the boundary in module docs. (b) `dispatch` returns 204 with
  no run id — resolve by returning a plan node that, on COMMIT, polls `runs` for the freshly
  triggered run (or returns the queued status) rather than a fake id. (c) pagination as a
  pure plan: emit a bounded fan-out of fetches, not an imperative loop, so it stays
  previewable and batchable.
- **Observability** (RFD §6): per-call timeouts, bounded retries on idempotent GETs only,
  circuit breaker on 5xx/secondary-rate-limit (respect `Retry-After`); every applied effect
  recorded in the audit ledger.
- **Coding standards** (RFD §9): owned DTOs only, sum-typed effects, thin HTTP client via
  #18 — no heavy SDK; `wasm32`-safe (no native TLS deps the REST client doesn't already use).

## Acceptance criteria
- `cargo build`, `cargo build --target wasm32-unknown-unknown`, `cargo clippy -- -D warnings`,
  and `cargo fmt --check` are green.
- `DESCRIBE /github/{owner}/{repo}/issues` returns the declared columns; the same for all
  eight namespaces.
- **Plan assertions (no live creds)**: golden tests assert the exact Plan/effect DAG for:
  `SELECT … FROM /github/o/r/issues WHERE state='open' AND label='bug'` (predicate pushed to
  query params); `INSERT INTO /github/o/r/issues/123/comments VALUES (…)` (single POST node,
  `RETURNING` row); `UPDATE /github/o/r/issues/123 SET state='closed'` (PATCH, state only);
  `CALL github.merge(method=>'squash')` and `CALL github.dispatch(workflow=>'ci.yml', ref=>'main')`
  emit `irreversible` HTTP-call nodes at the right endpoints; `CALL github.review(event=>'APPROVE')`.
- Capability gating: `UPDATE /github/o/r/runs/…` is rejected **at parse time** with a
  structured capability error, not at apply time.
- A pushdown test asserts paginated `SELECT` collapses into one batched fetch set in the plan.
- No vendor/SDK type appears in any public signature of the driver module (DTO-boundary test
  or doc check); token never appears in serialized plan or log output.
