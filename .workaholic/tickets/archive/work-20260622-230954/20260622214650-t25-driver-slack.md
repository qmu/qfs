# Driver: Slack

## Overview

This ticket delivers the **Slack driver**: one of the named backends of the vision
(RFD §1) and the canonical instance of the **Append / log** archetype crossed with an
**object-graph** (RFD §5 — "a single driver may expose multiple archetypes on different
sub-paths"). It implements the `Driver` trait (t13) and the open-**paths** registry of
RFD §3, mounting Slack under `/slack/<ws>/...` so an agent reaches messages, threads,
files, DMs, and users through the *same* DSL it uses for mail or SQL — zero new keywords.

Posting a message is **`INSERT INTO /slack/<ws>/<#channel>/messages`** (the path is the
type — RFD §3 "CRUD is universal; no per-driver create verb"). Reactions are either
`INSERT INTO .../reactions` or the equivalent `CALL slack.react(...)`; pinning is the
irreducible `CALL slack.pin(...)` (RFD §3 — irreversible/irreducible transitions are
namespaced procs, not verbs). All writes evaluate to a **`Plan`** and only run under
`COMMIT` (RFD §3 purity invariant; RFD §6 PREVIEW/COMMIT). Inbound Slack **Events API**
deliveries feed the server's `CREATE TRIGGER … ON <event>` machinery (RFD §8 — "watch
services → run effect-plans"); this ticket *produces* normalized events, the server
(E7) consumes them. It reuses the generic HTTP plumbing from t18 (thin client, auth as
config, codec-decoded JSON) rather than a heavy Slack SDK (RFD §9).

## Scope

In scope:
- A `SlackDriver` implementing the `Driver` trait (t13), mounted at `/slack/<ws>/...`,
  one config block per workspace `<ws>`.
- Namespace tree with per-node archetype + schema:
  - `/slack/<ws>/<#channel>/messages` — **Append/log**; `SELECT(tail)` + `INSERT(post)`.
  - `/slack/<ws>/<#channel>/messages/<ts>/replies` (threads) — Append/log under a parent `ts`.
  - `/slack/<ws>/<#channel>/messages/<ts>/reactions` — Append/log (`INSERT`/`REMOVE`).
  - `/slack/<ws>/files` — **Blob/namespace** (`ls`/`cp`/`rm`); bytes bridge via codecs (t15).
  - `/slack/<ws>/dms/<user>/messages` — Append/log.
  - `/slack/<ws>/users` — **Relational/table** (read-mostly directory).
- Verb→Web-API mapping *internal* to the driver: `SELECT→conversations.history/replies`,
  `INSERT(message)→chat.postMessage`, `INSERT(reply)→chat.postMessage(thread_ts=)`,
  `INSERT(reaction)→reactions.add`, `REMOVE(reaction)→reactions.remove`,
  `REMOVE(message)→chat.delete`, files `cp`→`files.upload`, `rm`→`files.delete`.
- Procedures (`CALL` targets, t13 `ProcSig`): `slack.react(channel, ts, emoji)`,
  `slack.pin(channel, ts)`, `slack.unpin(...)`, `slack.update(channel, ts, text)` —
  the irreducible transitions; `slack.pin`/`slack.delete` flagged `irreversible`.
- Optional **prelude** alias (RFD §3, pure): `POST(d) = d |> CALL slack.post` desugaring
  to message INSERT, receiver-typed to Slack append nodes.
- An **inbound event normalizer**: Slack Events-API envelope → owned `SlackEvent` DTO
  the server's trigger bus consumes; URL-verification + signature-check contract.
- Cursor pagination + `@version` declaration: Slack edits are by `ts` (Snapshot at best).

Out of scope (deferred):
- OAuth install flow, bot/user-token acquisition, encrypted token store, signing-secret
  storage → **E5 / t19-style auth**. This driver consumes a resolved token + signing
  secret via the injected `CredentialResolver`; it never stores or refreshes them.
- The HTTP transport, retry/circuit-breaker, and pagination *mechanics* — reused from the
  generic REST driver (**t18** `HttpClient`/`Pagination`); this ticket configures, not reinvents.
- Trigger/webhook *registration and dispatch* (`CREATE TRIGGER`/`CREATE WEBHOOK`, the HTTP
  ingress route) → **E7 server**; here we only normalize the event payload + verify signatures.
- Pushdown of `WHERE`/`LIMIT` beyond `oldest/latest/limit` passthrough → **E3 pushdown**.
- Slack Block Kit authoring helpers / rich-layout DSL → a follow-up "Slack blocks" ticket.
- Socket Mode (WebSocket) ingestion → follow-up; this ticket targets HTTP Events API.

## Key components

New module `drivers/slack/` in the `qfs-drivers` crate. Owned DTOs only — no Slack SDK,
no `reqwest`/`Url` leakage past the boundary (RFD §9).

```rust
pub struct SlackDriver { workspaces: HashMap<String, SlackWsConfig>, http: Arc<dyn HttpClient> }

pub struct SlackWsConfig {            // owned DTO, serde-deserialized from qfs config
    pub team_id: String,
    pub token: SecretRef,             // bot/user token, resolved at COMMIT (E5)
    pub signing_secret: SecretRef,    // verifies inbound events
    pub default_codec: CodecId,       // DECODE json
}

// Append/relational/blob sub-paths resolved from a parsed Slack path:
enum SlackNode { Messages{channel: ChannelRef}, Replies{channel, parent_ts: Ts},
                 Reactions{channel, ts: Ts}, Dms{user: UserRef}, Files, Users }

impl Driver for SlackDriver {
    fn id(&self) -> DriverId;                    // "slack"
    fn describe(&self, p: &VfsPath) -> Result<NodeDesc, DriverError>;  // archetype+schema per node
    fn capabilities(&self, p: &VfsPath) -> Capabilities;              // narrows by node
    fn procedures(&self) -> &[ProcSig];          // react/pin/unpin/update/delete
    fn prelude(&self) -> &[AliasDef];            // POST
    fn version_support(&self, p: &VfsPath) -> VersionSupport; // Snapshot for messages
    fn applier(&self) -> &dyn PlanApplier;       // only impure seam (t09)
}

// Plan leaf the interpreter (E2) applies — reuses t18 HttpEffect or a typed wrapper:
struct SlackEffect { call: SlackApiCall, body: serde_json::Value,
                     idempotency_key: Option<String>, irreversible: bool }

// Inbound:
pub struct SlackEvent { pub kind: SlackEventKind, pub channel: Option<String>,
                        pub ts: Option<Ts>, pub user: Option<String>, pub raw: serde_json::Value }
pub fn parse_event(headers: &EventHeaders, body: &[u8], signing_secret: &str)
    -> Result<SlackInbound, EventError>;   // SlackInbound::UrlVerification | Event(SlackEvent)
```

- `enum SlackApiCall` — closed list of Web-API methods this driver issues (postMessage,
  history, replies, reactions.add/remove, pins.add/remove, files.upload/delete, conversations.list,
  users.list) — capability sum type (RFD §9 enums).
- `ChannelRef`/`UserRef` — `#name` or `Cxxxx`/`Uxxxx` id, resolved via a cached id lookup.
- Capability gating: messages node = `{Select, Insert, Remove}`; reactions = `{Insert, Remove}`;
  users = `{Select}`; files = `{Ls, Cp, Rm}` — unsupported verbs rejected at parse time (RFD §5).

## Implementation steps

1. Add `drivers/slack/` module; define `SlackWsConfig`, `SecretRef`, `SlackNode`, `SlackApiCall`,
   `SlackEvent` with `serde`; register in the driver/paths registry so `/slack/<ws>/...` resolves.
2. Implement Slack path parsing → `SlackNode` (channel `#name`/id, `<ts>`, `replies`, `reactions`,
   `dms/<user>`, `files`, `users`); structured `DriverError::NotFound` on malformed paths.
3. Implement `describe`/`capabilities` per node (archetype + schema columns + verb set); golden
   `DESCRIBE` snapshots.
4. Implement reads: `plan_select` for messages/replies (`conversations.history`/`.replies` with
   `oldest/latest/limit` passthrough + cursor pagination from t18) and `users`
   (`users.list`); response bytes → rows via `DECODE json` (t15).
5. Implement writes as `Plan` leaves: `INSERT(message/reply)→chat.postMessage`,
   `INSERT(reaction)→reactions.add`, `REMOVE(reaction)→reactions.remove`,
   `REMOVE(message)→chat.delete` (`irreversible`); attach a `client_msg_id`/idempotency key.
6. Implement files sub-path: `cp` into `/slack/<ws>/files`→`files.upload` (multipart),
   `rm`→`files.delete` (`irreversible`), `ls`→`files.list`.
7. Register procedures (`slack.react/pin/unpin/update/delete`) with `ProcSig` (`irreversible`,
   `requires_scopes`); add the pure `POST` prelude alias.
8. Implement `parse_event`: verify `X-Slack-Signature` (HMAC-SHA256 over `v0:ts:body`,
   constant-time compare) + timestamp freshness; handle `url_verification` challenge; normalize
   `message`/`reaction_added`/`app_mention`/`file_shared` → `SlackEvent`.
9. Wire token/signing-secret via `CredentialResolver` (resolved at commit / verify time, never
   in config structs or `Debug`).
10. Tests: golden plan tests + `MockHttpClient` integration + event-parsing fixtures (below).
11. `cargo fmt`, `cargo clippy -D warnings`, `cargo test`.

## Considerations

- **Least privilege & secrets (RFD §10):** bot token and signing secret are `SecretRef`
  indirections resolved at the edge; manual/`redacted` `Debug` so neither token nor signature
  material is logged. Capability gating lets a server `POLICY` scope a handler to read-only Slack
  (e.g. a digest job) vs. post-capable.
- **The genuinely hard part — inbound event verification + idempotency.** Slack retries event
  deliveries (at-least-once) and signs with a 5-minute-windowed HMAC. Resolve by: (a) constant-time
  signature check + timestamp-skew rejection in `parse_event` (replay defense), and (b) carrying
  Slack's `event_id` so the server trigger bus can dedupe — events are *facts*, the plan they fire
  must use idempotent/`UPSERT`-shaped effects (RFD §6). `parse_event` is **pure** (no I/O), so it is
  unit-testable from fixtures and `wasm32`-safe (Workers `WEBHOOK` ingress, RFD §8).
- **Idempotency / recovery (RFD §6):** `chat.postMessage` is *not* idempotent — never auto-retry a
  post on an ambiguous timeout; attach `client_msg_id` and surface the risk in PREVIEW. `reactions.add`
  is naturally idempotent (already-reacted is a no-op-class error we swallow). `pin`/`delete` are
  `irreversible` → only under explicit `COMMIT`, logged to the audit ledger (RFD §6/§10).
- **Channel/user id resolution:** `#name`→`Cxxxx` needs a `conversations.list` lookup. Keep this an
  internal cached resolution performed by the *applier* at commit (it is I/O), not during planning —
  PREVIEW shows the symbolic `#channel`; planning stays pure (RFD §3 purity invariant).
- **@version (RFD §4):** Slack has no true version coordinate; declare `VersionSupport::Snapshot`
  (a message `ts` identifies a point), enabling optimistic edit-by-`ts` but not history rewind.
- **Rate limits / observability (RFD §6):** Slack tiers + `Retry-After`; reuse t18 bounded retry +
  per-workspace circuit breaker, structured logs (method, channel, status; redacted token). Bound
  `conversations.history` paging.
- **Coding standards / boundaries (RFD §9):** Slack JSON and HTTP types never cross the driver
  boundary — rows out; `SlackWsConfig`/`SlackEffect`/`SlackEvent` are owned DTOs. Driver lives in
  `qfs-drivers::slack`; reuse t18 `HttpClient`/`Pagination` rather than a new client.

## Acceptance criteria

- `cargo build`, `cargo clippy -D warnings`, `cargo test` green across the workspace; wasm
  feature (`parse_event` + pure introspection) compiles for `wasm32-unknown-unknown`.
- `SlackDriver` registered; `/slack/<ws>/...` paths resolve and `DESCRIBE` returns the correct
  per-node archetype (messages/replies/reactions=Append, files=Blob, users=Relational) + schema
  + capability set (golden/snapshot test).
- **Plan assertions (no live creds, `MockHttpClient`):**
  - `INSERT INTO /slack/<ws>/#general/messages VALUES(text=>'hi')` → a `Plan` whose leaf is
    `chat.postMessage` carrying the configured token (redacted in any logged form) + a
    `client_msg_id`.
  - `INSERT INTO .../messages/<ts>/replies` → `chat.postMessage` with `thread_ts=<ts>`.
  - `INSERT INTO .../reactions` and `CALL slack.react(...)` produce **equivalent** plans
    (`reactions.add`); `CALL slack.pin(...)` produces one `Call` node tagged `irreversible`.
  - `REMOVE` of a message → `chat.delete`, `irreversible=true`.
  - A verb not declared for a node (e.g. `UPDATE` on `users`, `INSERT` on `users`) is rejected at
    **parse time** with a structured `CapabilityError` listing supported verbs.
- **Golden tests:** PREVIEW output for post / reply / react / pin matches checked-in goldens
  (method, channel, redacted token, irreversible flag).
- **Event tests (pure, fixtures):** a valid-signature `url_verification` returns the challenge; a
  valid `message`/`reaction_added` payload normalizes to `SlackEvent`; a tampered signature or
  stale timestamp is rejected; `event_id` is surfaced for dedupe.
- Pagination test: mock returns 3 cursor pages of `conversations.history` → committed `SELECT`
  yields concatenated rows; page cap enforced.
- No token or signature material appears in `Debug`/log output (redaction test).
