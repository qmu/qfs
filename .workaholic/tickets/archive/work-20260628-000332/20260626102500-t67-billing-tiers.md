# t67 — Billing (free individual / paid team)

## Overview

Delivers **billing** for the managed tier (M9, roadmap §3.4): a free individual tier and a paid team
tier, surfaced in the dashboard's admin area. The architectural point (roadmap §3.4) is that
administration is "everything is a path" — so plan state is modeled as **data** under `/sys/*` and
billing is the dashboard rendering the same engine over the same grammar, preserving
one-engine-three-faces. Plan/subscription state and the tier-gating logic are new; what exists to
build on is the `/sys/*` driver pattern (t53 `qfs-driver-sys`, itself mirroring
`crates/server/src/driver.rs` `ServerDriver`: pure introspection + a runtime applier) and the System
DB (t42). The payment-provider integration is an **external service** and is the one open product
decision to flag — this ticket models the *internal* plan state and tier gate, and defines the
provider seam without committing to a vendor.

## Exact seams

- t53 `qfs-driver-sys` (pattern: `crates/server/src/driver.rs` `ServerDriver`, `SERVER_MOUNT`, pure
  introspection + `NoopApplier`, real mutation via a runtime applier) — add `/sys/billing` (plan,
  tier, status) to the `/sys/*` surface; the read half is pure, mutation rides the runtime applier.
- `crates/core/src/ddl/server.rs` `server_node_schema(node)` (single source of truth for `/server/*`
  schemas) + `src/ddl/server/spec.rs` `StatementSpec`/`PlanSpec` — the `/sys/billing` schema is
  defined here as the single authority (the t53 `/sys` analogue), serializable spec, not live state.
- t42 System DB — plan/subscription rows (`tier`, `status`, `current_period_end`, `team_id`) live in
  the System DB via the embedded migration runner; plan state is data, queryable as a path.
- `crates/server/src/policy/enforce.rs` `evaluate` (pure, default-deny) — a paid-tier capability gate
  is expressed as policy/plan-state, not a bespoke `if paid {}`; the gate reads plan state from
  `/sys/billing` and denies team-tier features on the free tier.
- t66 team connections — the paid tier gates team-wide brokered connections; the gate reads the same
  `/sys/billing` plan state.
- The payment provider is a NEW external driver/seam (`PaymentProvider` trait, binary-leaf impl) —
  there is no payment code anywhere today; webhooks from the provider ride the existing
  `crates/watchtower` `WebhookBinding` (HMAC-SHA256 verify) to update plan state at-least-once.
- `crates/secrets/src/store.rs` `Secrets` + t43 envelope store — provider API keys are stored
  envelope-encrypted, resolved by handle, never logged (`crates/secrets/src/secret.rs` redaction).

## Implementation steps

1. Schema + state: define the `/sys/billing` node schema in `crates/core/src/ddl/server.rs`'s `/sys`
   analogue (single source of truth) and add the System-DB plan/subscription table via t42's
   migration runner. Read path returns plan/tier/status (no secrets). Tree green (free tier is the
   default; nothing gated yet).
2. Tier gate: express paid-tier feature gating as plan-state read through
   `crates/server/src/policy/enforce.rs` (or a thin capability check that reads `/sys/billing`),
   default-deny for team-tier features on the free tier. Pure tests: a free-tier plan denies a
   team-connection capability; a paid-tier plan allows it.
3. Provider seam: define a NEW `PaymentProvider` trait (binary leaf, behind the qfs Cloud managed
   mode) with a stub/in-memory impl for tests — checkout-session create, subscription status read.
   No vendor SDK in any pure core; the live impl rides `crates/qfs/src/transport.rs`'s
   `HttpExchange` pattern. FLAG the provider choice (see Considerations).
4. Provider webhooks: receive subscription-changed events through the existing
   `crates/watchtower` `WebhookBinding` (HMAC-SHA256 verify) and update `/sys/billing` plan state via
   the `/sys` runtime applier — at-least-once, idempotent (dedup on the provider event id, reusing
   watchtower's dedup ledger).
5. Admin view: render `/sys/billing` in the t53 admin surface (a dashboard view over the same path);
   a super-admin can also read/grant via a qfs statement (one-engine-three-faces). Plan changes go
   through preview→commit like any other `/sys` mutation.
6. Docs + version: document billing only after a live stub-provider smoke (free→paid plan change via
   a webhook updates the gate) passes; patch-bump `crates/qfs/Cargo.toml`;
   `cargo run -p xtask -- gen-docs --check`.

## Key files

- `crates/core/src/ddl/server.rs` (`/sys/billing` schema authority), t42 System-DB migration for the
  plan/subscription table.
- t53 `qfs-driver-sys` (`/sys/billing` read + applier), admin view over it.
- `crates/qfs/src/billing.rs` (new — `PaymentProvider` trait + binary-leaf live impl), webhook
  handler wired through `crates/watchtower` `WebhookBinding`.
- `crates/qfs/Cargo.toml` (patch bump).

## Considerations

- **Open product decision to FLAG, not guess (the headline of this ticket):** the payment provider is
  an external service (Stripe, Paddle, etc.) — its choice, fee model, and tax/invoicing scope are a
  business decision. Build the `PaymentProvider` seam + a stub; name the provider question in the PR
  and do not hard-code a vendor.
- **Plan state is data, gating is policy.** Keep the paid/free gate expressed as `/sys/billing`
  plan-state read through the policy/capability path (default-deny), not scattered `if paid` checks —
  this is what keeps billing "everything is a path" and auditable.
- **Secrets out of logs.** Provider API keys + webhook signing secrets are envelope-encrypted (t43),
  resolved by handle, verified via `crates/watchtower` HMAC, and never logged
  (`crates/secrets/src/secret.rs` redaction).
- **At-least-once webhooks.** Provider events arrive at-least-once; dedup on the provider event id
  via watchtower's ledger so a replayed "subscription cancelled" does not double-apply.
- **Safety floor.** A plan change is a `/sys` mutation: previewed, then committed; a super-admin sees
  the same preview→commit a dashboard click does.
- **Dep-direction & no new keyword.** Payment/provider edges land on `crates/qfs`; `/sys/billing` is
  a path, adding nothing to the frozen keyword set.
- **Versioning:** own PR + patch bump in `crates/qfs/Cargo.toml` (currently 0.0.7) + `v0.0.x` tag on
  ship.
