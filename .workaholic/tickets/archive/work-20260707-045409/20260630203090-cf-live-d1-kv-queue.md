# `/cf` live — Cloudflare D1 / KV / Queues read+commit (owner item #3)

## What's wanted

Make `/cf` actually read/write Cloudflare (D1 SQL, KV, Queues), not just be a reachable describe
mount. Owner will **paste a CF API token + account id** (and a D1 database id) for the live test.

## Current state

`/cf` is a reachable PLANNING + describe mount over a cred-free representative `CfRegistry`
(`crate::describe::cred_free_cf_registry`, commit `8cce093` / `040d13f`); `qfs-driver-cf` is a binary
dep + on the dep-direction allowlist. No live read/commit; the registry has no real D1/KV/queue.

## Plan

1. Build a **live** `CfRegistry` from a declared cf connection: `CREATE CONNECTION cfprod DRIVER cf
   AT '<accountId>' SECRET 'env:CF_API_TOKEN'`. The richer per-resource config (which D1 databases /
   KV namespaces / queues) needs more than `(driver, locator, secret)` — design it: either enumerate
   via the Cloudflare API at registry build, or extend the declaration. `CfBackend` is the
   `HttpApiBackend` (token = `qfs_secrets::Secret`).
2. Register the live cf read + apply facets over the connect-account fallback, gated by
   `cloud_bind_allowed` (the github/slack/google pattern), in `crate::shell` + `crate::commit`.
3. Live-verify against the owner's account (D1 `select`, KV get/put, queue tail) with the pasted
   token (export `CF_API_TOKEN`, set the account id).

## Key files

- `crates/driver-cf/src/{registry.rs,backend.rs,lib.rs}`, `crates/qfs/src/{shell.rs,commit.rs,
  describe.rs}`, `crates/qfs/src/secret_ref.rs` (token resolution).

## Considerations

- D1 is a full SQLite-dialect backend (reuses the t17 SQL path); KV/Queues push only a bounded scan.
- Credential pasted by the owner at implementation time (do not commit it). Mock tests
  (`MockCfBackend`) cover the logic hermetically; the live read is the acceptance proof.

## Final Report

Development completed as an environment-backed live `/cf` composition. qfs now builds a Cloudflare
driver from `CF_ACCOUNT_ID`, `CF_API_TOKEN`, and resource lists for D1 databases, KV namespaces, and
Queues; registers live read facets for D1/KV/Queue paths; and wires the same driver into commit so
missing configuration fails closed instead of reporting preview-only success.

### Discovered Insights

- **Insight**: The current connection declaration shape cannot faithfully carry Cloudflare's
  per-resource catalog yet, so the live path uses explicit environment resource lists as a narrow
  composition layer.
  **Context**: A later declarative `CREATE CONNECTION cf ...` design should model D1/KV/Queue
  resource handles directly rather than hiding that shape inside a generic locator string.
