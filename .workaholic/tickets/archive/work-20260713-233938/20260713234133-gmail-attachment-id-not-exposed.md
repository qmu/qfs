# Gmail attachment_id is not exposed, so the attachment node is unaddressable from a real read

## Overview

Found live (L67 Gmail→Drive round, owner-attended): the one-statement transfer
`INSERT INTO /drive/<folder> SELECT filename AS name, mime, content AS bytes FROM
/mail/<label>/<msg>/<att-id>` needs the real Gmail `attachment_id` in the source path, but no qfs
read surfaces it. The message-listing `attachments` column is `Array(Struct{filename, mime, size})`
(`MailMessage::attachment_schema`), dropping the `attachment_id` the internal `AttachmentMeta`
carries. The attachment node keys on that id (`read.rs`: `find(|a| a.attachment_id == attachment)`),
so a user cannot discover the id to address the node against a real message. The hermetic test passes
only because the mock's id (`att1`) is known out-of-band.

## Key Files

- `packages/qfs/crates/driver-gmail/src/schema.rs` — `MailMessage::attachment_schema()` (the array
  element struct) and `MailMessage::to_row` (the attachments `Value::Struct` builder).
- `docs/cookbook/gmail.md` — the "list a message's attachments, then download one" recipe (show the
  now-exposed `attachment_id`).

## Implementation Steps

1. Add `Column::new("attachment_id", ColumnType::Text, false)` to
   `MailMessage::attachment_schema()`.
2. In `MailMessage::to_row`, add `("attachment_id".to_string(), Value::Text(a.attachment_id.clone()))`
   to the attachments `Value::Struct` so a listing row carries the id.
3. Cookbook: update the attachment recipe to select `attachment_id` (e.g. `select attachments`, or
   `expand attachments |> select attachments.attachment_id`) and feed it into the
   `/mail/<label>/<msg>/<att-id>` address; regenerate skills.
4. Fix any golden/snapshot that pins the 3-field attachments struct.

## Considerations

- `/mail` is an `append_log` in `docs/drivers.md`; confirm whether the attachments struct fields are
  rendered there (if so, `gen-docs --check` will need the regen committed).
- This is a taught-surface change (cookbook recipe), so a **plugin version bump** is owed with the
  gen-skills regen.
- Column order: append `attachment_id` after `size` to avoid reordering the existing struct.

## Discussion

### Revision 1 — 2026-07-14 (mid-implementation pivot)

**Finding during live verification**: the first implementation (expose `attachment_id` in the
listing struct) was **wrong-premised**. Gmail's `attachmentId` is **ephemeral** — regenerated by
every `messages.get` (confirmed: three fetches of the same message returned three different ids).
So a literal id carried from a prior `select attachments` into a separate transfer statement is
already stale; the live commit failed `invalid_path` at the read.

**Direction change (owner decision)**: address an attachment by its **stable 0-based index**
`att<N>` instead. The driver resolves the index against the transfer's own `get_message` and fetches
with that fresh id, so the whole transfer is one plan with a live id. Reverted the `attachment_id`
column exposure; implemented `att<N>` addressing in `read.rs` (`resolve_attachment`) and taught it in
the cookbook.

## Final Report

Development completed with the pivot above. The attachment node `/mail/<label>/<msg>/att<N>` resolves
the Nth attachment (listing order) and fetches with the id freshly read, dodging the ephemeral-id
trap; a non-`att<N>` segment falls back to an explicit-id match (valid within a session). Verified:
`cargo test -p qfs-driver-gmail` (68 tests, incl. `att_index_resolves_a_fresh_id_and_out_of_range_fails_closed`),
clippy/fmt clean, gen-docs + gen-skills + cookbook ratchet in sync. **Live-proven (owner-attended,
v0.0.62):** `/mail/inbox/<msg>/att0 |> select filename as name, mime as mime_type, content as bytes
|> insert into /drive/my/qfs-extract-test` landed the 1579-byte `invite.ics` byte-exact (sha256
match on the Drive read-back). Ticks the L67 one-statement Gmail→Drive acceptance.

### Discovered Insights

- **Insight**: Gmail's `attachmentId` from `messages.get` is **not stable** — it regenerates per
  request. Any design that carries it across statements (or caches it) is broken by construction; a
  mock with a fixed id hides this (the hermetic test passed while the live transfer failed).
  **Context**: for ephemeral-handle resources, address by a stable property (index/name) and resolve
  the live handle inside the same read; never surface the handle as a user-carried value.
