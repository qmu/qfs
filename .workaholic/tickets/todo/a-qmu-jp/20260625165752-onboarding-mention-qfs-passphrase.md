# Onboarding: the install guide instructs `qfs account add` but it fails without `QFS_PASSPHRASE`

## Overview

The post-install **Next steps** guide (and every doc that shows `qfs account add`) tells a fresh
user to run, as step **2**:

```sh
qfs account add mail work
```

But that command fails **out of the box**:

```
qfs: error: QFS_PASSPHRASE is not set — export it to unlock the encrypted credential store
```

So the very first "connect a service" step the install advertises does not work as written. This is
a direct violation of the repo's standing **honesty principle** ("never document a capability before
it works / keep docs honest about exactly what works" — the same discipline that keeps the README
honest about the Cloudflare host being parked and release artifacts being CI-only).

**This is a docs/onboarding fix, not a code change** (a code follow-up is noted at the end).

## Root cause (a sequencing gap, verified)

- The Next-steps block was written + shipped in **592ff8c** ("install.sh: print a Next-steps
  welcome", bump 0.0.2→0.0.3) — when `qfs account add` was still a stub.
- ~2.5h later **8e86e1a** ("Implement qfs account add/list/use/remove (real encrypted store)") wired
  `account add` to the encrypted `LocalStore`, which requires the `QFS_PASSPHRASE` env var to derive
  the AEAD key (argon2id). `account.rs::open_store()` reads `QFS_PASSPHRASE`; unset → the error above
  (`account.rs:47-48`). The onboarding copy was honest at write time and silently went stale.
- **No user-facing doc anywhere mentions `QFS_PASSPHRASE`** — a repo-wide grep finds it only in Rust
  source/doc-comments and the error string. So the prerequisite was never surfaced.

## Two correctness facts any fix MUST get right

1. **`QFS_PASSPHRASE` is required** for `account add/list/remove` (not `use`, which only writes the
   plaintext `.active` sidecar). The per-store **salt sidecar is auto-created** — only the passphrase
   is a user prerequisite.
2. **The credential VALUE is read from STDIN, never argv** (`account.rs:9-11, 111-119`). So even WITH
   the passphrase exported, the bare `qfs account add mail work` shown in the docs fails on empty
   stdin (`no secret on stdin — pipe it, e.g. printf %s "$TOKEN" | qfs account add mail work`). The
   fully-correct documented invocation is:

   ```sh
   read -rs QFS_PASSPHRASE; export QFS_PASSPHRASE     # unlock the local vault (no shell-history leak)
   printf %s "$TOKEN" | qfs account add mail work     # the credential value via stdin, never argv
   ```

   A **secondary honesty gap** in the same files: `docs/guide/getting-started.md:114` comments
   `# prompts for the credential` and `install.sh` implies interactivity — but `account add` does NOT
   prompt; correct this to the real stdin-piped UX while touching these lines.

## Security framing (RFD §10 — must not regress)

- Frame `QFS_PASSPHRASE` as the **local-vault KDF master passphrase** (argon2id → AEAD over the
  `0600` blob at `~/.config/qfs/credentials`), **not** a service credential. Keep the two secrets
  distinct in the prose: the **passphrase** unlocks the vault (env); the **credential value** is
  supplied separately via **stdin** (never argv — argv leaks into the process table + shell history).
- Prefer history-safe guidance (`read -rs`, a `0600` sourced file, or a leading space with
  `HISTCONTROL=ignorespace`) over a bare `export QFS_PASSPHRASE=secret` inline.
- Do not overstate the guarantee: this is at-rest confidentiality only; a compromised host/process
  can read `QFS_PASSPHRASE` from the environment.

## Considerations

- **Versioning:** a shipped change → bump the patch in `packages/qfs/crates/qfs/Cargo.toml`
  (**0.0.7 → 0.0.8**) before the PR, with a matching `v0.0.8` tag on ship. The in-scope files
  (install.sh, README, `docs/guide/*.md`, SKILL.md) are hand-authored prose, not gen-docs goldens —
  no `gen-docs` regen required (unless `docs/server.md` is touched at its source).
- **Honesty:** the test-it step (step 1, `qfs describe` / preview) is correct and must stay — only
  the "connect a service" step is broken. Fix it so a fresh install can follow every advertised step.
- **Verify** after editing: with `QFS_PASSPHRASE` unset, the documented commands fail closed with the
  clear error; with it set, `printf %s "$TOKEN" | qfs account add mail work` succeeds (then
  `qfs account list` shows the account).

## Deferred follow-up (do NOT bundle here — separate ticket candidate)

A better default UX: when `QFS_PASSPHRASE` is unset, **prompt securely (no echo)** instead of
hard-erroring (`account.rs:47-51`), removing the onboarding cliff entirely. This is a code/UX change
(`rpassword`-style read), larger than this docs fix. The open ticket
`20260624214818-wire-real-execution-and-auth.md` (Slice 3) already names "an interactive no-echo
prompt" as the alternative to the env var — cross-reference it. Ship the docs fix first regardless.
