# t46 ‚Äî Session handling

## Overview
Delivers server-side sessions for the local web / dashboard face: opaque session tokens minted on
sign-in, persisted in the System DB, carried by a secure cookie over the existing `crates/http`
listener, with expiry and rotation. This is roadmap **M1** (Identity store, "session handling")
and continues **decision B** / **¬ß4.1** ‚Äî a session binds an HTTP request to a `users` row from
t45 (authentication state), still NOT authorization (policy/OAuth is M2). **What already exists:**
a real in-house HTTP/1.1 listener (`crates/http/src/serve.rs` over `tokio::net::TcpListener`, no
axum) and pure request/response DTOs (`crates/http-core/src/lib.rs` `HttpRequest`/`HttpResponse`
with `SENSITIVE_HEADERS` redaction); the identity store + sign-up from t45. **What is genuinely
new:** there is NO session concept, NO cookie handling, and NO `Set-Cookie` emission anywhere
today ‚Äî the HTTP listener currently serves read-only endpoints (`crates/http/src/handler.rs`
`dispatch`) and a `/status` table (`crates/qfs/src/serve_builtins.rs`), neither of which
authenticates a caller. This ticket adds a pure session core plus the binary wiring to issue and
validate sessions; it stops short of OAuth (M2) and of gating any data path on a session.

## Exact seams
- **New crate `qfs-session`** (pure-ish; no tokio): session domain ‚Äî `Session`, `SessionId`,
  `SessionToken` (opaque, high-entropy), `SessionStore` trait (`create/lookup/rotate/revoke`),
  and pure cookie-string formatting/parsing. Token generation uses OS entropy injected from the
  binary leaf (keep the core deterministic/testable).
- `crates/http-core/src/lib.rs` ‚Äî `HttpRequest`/`HttpResponse` DTOs + `SENSITIVE_HEADERS` /
  `is_sensitive_header` (the single redaction authority). `Cookie` and `Set-Cookie` are sensitive
  headers; ensure both are covered by `is_sensitive_header` so session tokens never appear in
  logs. Set-Cookie is emitted by building an `HttpResponse` header here (no new HTTP framework).
- `crates/http/src/handler.rs` `dispatch` and `crates/http/src/serve.rs` ‚Äî the request entry
  point. Add a session-extraction step (read the `Cookie` header ‚Üí `SessionStore::lookup` ‚Üí
  attach an optional authenticated `UserId` to the request context). This is read-side plumbing;
  it does NOT yet refuse unauthenticated requests (that gating is t50 for MCP, t51 for the SPA).
- `crates/http/src/policy.rs` `assert_read_only` ‚Äî unchanged contract; sessions are orthogonal to
  the read-only endpoint gate. Note the seam so a later write/login endpoint can coexist.
- `crates/store` (System DB, **t42**) ‚Äî add an `0003_sessions` migration: a `sessions` table
  keyed by a *hash* of the token (never the plaintext token), with `user_id`, `created_at`,
  `expires_at`, `rotated_from`. Reuses t42's embedded migration runner + `schema_version` bump.
- `crates/crypto-core/src/lib.rs` ‚Äî `sha256`/`sha256_hex` to hash the session token at rest (DB
  stores the hash; the cookie carries the plaintext token), and `constant_time_eq` for lookup
  comparison. Pure, zero-dep ‚Äî keep it that way.
- `crates/secrets/src/secret.rs` `Secret` ‚Äî wrap the plaintext token in transit so it is
  redacted/zeroized and never accidentally logged.
- `crates/qfs/src/serve.rs` (`run_serve` ‚Üí `qfs_http::serve_config_full`) ‚Äî composition root that
  injects the System-DB `SessionStore` into the listener; mirror how `CronBinding` /
  `WatchtowerBinding` are wired today.
- `crates/cmd/tests/dep_direction.rs` ‚Äî add `qfs-session` to allowlists; it must not depend on
  lang/plan/driver/codec/parser, and tokio stays out of it.

## Implementation steps
1. **Migration (tree green).** Add `0003_sessions` to t42's runner: `sessions (token_hash PK,
   user_id, created_at, expires_at, rotated_from NULLABLE)`. Idempotent apply on fresh +
   already-migrated DB; `schema_version` bump.
2. **`qfs-session` core.** Define `Session`/`SessionId`/`SessionToken` and a `SessionStore`
   trait. Implement pure cookie formatting: `Set-Cookie: qfs_session=<token>; HttpOnly; SameSite=
   Lax; Path=/; Max-Age=<ttl>` and `Secure` (gated on an injected "is-https/trusted" flag so
   localhost dev still works). Implement `Cookie:` header parsing to extract the token. Unit-test
   format/parse round-trips and attribute correctness.
3. **Token at rest.** Generate the opaque token from injected OS entropy in the binary; store
   `sha256_hex(token)` in the DB, return the plaintext token (as `Secret`) only in the response
   cookie. Lookup hashes the incoming cookie token and compares constant-time. Expiry checked on
   lookup; expired sessions are treated as absent and lazily reaped.
4. **Rotation + revoke.** On a privilege-relevant event (sign-in, and later consent in t49),
   rotate: mint a new token, set `rotated_from`, expire the old. `revoke` deletes the row
   (sign-out). Unit-test that a rotated/revoked token no longer resolves.
5. **HTTP wiring.** In `crates/http/src/handler.rs`/`serve.rs`, extract the cookie, look up the
   session, and attach an `Option<UserId>` to the request context ‚Äî without refusing
   unauthenticated requests yet. Confirm `Cookie`/`Set-Cookie` are in `SENSITIVE_HEADERS`.
6. **Compose + version.** Inject the System-DB `SessionStore` from `crates/qfs/src/serve.rs`. Add
   `qfs-session` to `dep_direction.rs`. Keep docs honest (no "login" advertised until M2 closes
   the loop). `cargo run -p xtask -- gen-docs --check`; patch-bump `crates/qfs/Cargo.toml`.

## Considerations
- **Safety floor.** Sessions are infrastructure, not a data effect ‚Äî minting one is reversible and
  needs no irreversible ack. Critically, attaching a `UserId` to a request grants NOTHING by
  itself this milestone; authorization is M2. Document that an authenticated session is inert
  until policy/OAuth land, so no path silently trusts it early.
- **Token hygiene (security-first).** The DB stores only `sha256_hex(token)` ‚Äî a System-DB leak
  does not yield usable session tokens. The plaintext token lives only in the `Secret`-wrapped
  cookie value; `Cookie`/`Set-Cookie` are in `SENSITIVE_HEADERS` so they are redacted in every
  log/trace/audit line. Cookies are `HttpOnly`, `SameSite=Lax`, and `Secure` when the listener is
  not plain-localhost. Lookups are constant-time.
- **Expiry & rotation.** Short idle TTL + absolute max lifetime; rotate on sign-in (and on the
  consent step in t49) to limit fixation. `rotated_from` preserves an audit chain without keeping
  the old token usable.
- **Dep-direction.** `qfs-session` is a pure-ish leaf (no tokio); SQLite I/O is the binary-injected
  `crates/store` impl. The listener stays in `crates/http`; live wiring lands only on `crates/qfs`.
- **Open product decisions to flag (do not guess).** (a) Cookie name and exact TTLs (idle vs.
  absolute). (b) Whether to also support a bearer-header session for non-browser local tools, or
  reserve bearer tokens entirely for the OAuth path (M2, t50) ‚Äî prefer the latter to avoid two
  token systems; flag it. (c) CSRF strategy for any future state-changing browser POST (likely a
  double-submit token) ‚Äî out of scope here, note the seam.
- **Versioning.** One PR, one patch bump, a `v0.0.x` tag on ship.
