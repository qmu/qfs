# t60 — `LET` binding

## Overview
Delivers the first piece of the **M6 — Language core** functional layer: a new `LET` keyword that
names an intermediate result so it can be referenced more than once (roadmap §1.2; decision H —
"a functional core … on top of the frozen closed-core vocabulary"). Today the grammar has no way to
name a sub-pipeline — every reuse means re-typing the subquery — so `LET products = FROM … |> …`
then `FROM products |> …` is genuinely new surface. Crucially, `LET` is one of only **two** genuine
new keywords the whole roadmap permits (the other is `TRANSACTION`, t62); it MUST be added to the
frozen keyword fixture **deliberately**, with the freeze tests updated as an intended vocabulary
change. The pipe/expression machinery (`Statement`, `PipeOp`, `Expr`, `Evaluator`, `PlanSource`)
already exists as a library; this ticket threads a binding environment through it. Pure — no I/O, no
new driver edge, stays wasm-buildable.

`LET` also forces the **statement model** the roadmap settled on (§1.2, decision O note): a program is
**multiple statements with no terminator** — `LET <name> = …` lines followed by the query that uses
them, one statement per line, `;`-free. Today's parser is strictly one-statement-then-EOF, so this
ticket adds multi-statement parsing with a newline as a soft boundary (a missing `|>` is a crisp
error, not a silent split). This is unambiguous because statements start with a closed keyword set and
pipelines only extend with `|>` — but it is real work that lands here, with `LET`.

## Exact seams
- `crates/lang/src/keywords.rs` — `Keyword` enum + `KEYWORDS` slice (currently **frozen at 38**) +
  `from_word`. Adding `LET` means: new `Keyword::Let` variant, new `"LET"` entry in `KEYWORDS`,
  a `from_word("LET")` arm, and a deliberate edit to the freeze tests
  `keyword_enum_matches_golden_fixture`, `keyword_count_is_frozen` (38 → 39), and the `ALL_KEYWORDS`
  fixture used by `from_word_recognizes_exactly_single_word_keywords`. `OPERATORS` (15) and
  `operator_count_is_frozen` are untouched — `LET` is a keyword, not an operator.
- `crates/parser/src/ast.rs` — `Statement` enum (Query/Effect/Ddl/Plan; **NOT** `#[non_exhaustive]`
  by design so the governance test locks the variant set). Add a `Statement::Let { name: Ident,
  value: Box<Statement> }` (or a top-level `Binding` form) — the binding's value is itself a
  statement (a query pipeline). The governance test that pins the `Statement` variant set is the
  tripwire to update deliberately, the same as the keyword freeze.
- `crates/parser/src/grammar.rs` — winnow-based `parse_statement()` (grammar locked per ADR-0001):
  add a `LET <ident> = <statement>` production. A bound name used as a source resolves through the
  existing `PathExpr`/`Source` path — decide (and document in the ticket) whether a bound name is a
  bare identifier source or a reserved sigil; prefer the bare-identifier source form (`FROM products`
  in today's grammar — note roadmap §1.2 now renders the post-decision-R target `products` with no
  `from`; the `from`-drop is the separate Resource/drop-`from` ticket, so t60 lands with `FROM`).
- `crates/core/src/eval.rs` — `Evaluator` (holds `&MountRegistry` + `Resolver` + `&StdlibRegistry`),
  `eval_statement() -> EvalValue` (`Relation(PlanSource)` | `Plan(Plan)`), and the `PlanSource`
  enum. A `LET` binds a name to a `PlanSource` in an environment so later references resolve to that
  relation source rather than re-evaluating a path mount. Add a binding environment (scope map) that
  `eval_statement` threads; a referenced bound name yields `EvalValue::Relation(PlanSource::…)`.
- `crates/core/src/resolve.rs` — `Resolver`, `resolve_statement`/`resolve_expr`. Name resolution
  must consult the `LET` environment **before** the mount registry so a bound name shadows nothing
  unexpected; an unbound name that is neither a path nor a binding stays the existing unrouted-path
  error. Reuse `EvalError` (`crates/core/src/eval.rs`) — add `EvalError::UnknownBinding { name }` (or
  fold into the existing resolve error) so a typo in a bound name is a structured, AI-consumable
  error, never a silent empty relation.

## Implementation steps
1. **Keyword (deliberate freeze edit).** Add `Keyword::Let`, the `"LET"` slice entry, and the
   `from_word` arm in `crates/lang/src/keywords.rs`; update `keyword_count_is_frozen` to 39, extend
   the golden fixture / `ALL_KEYWORDS`, and add a one-line comment marking this as an intended M6
   vocabulary change (decision H). `cargo test -p qfs-lang` green; `cargo fmt`.
2. **AST + governance.** Add the `Statement::Let { name, value }` variant in
   `crates/parser/src/ast.rs`; update the exhaustive-match governance test that locks the
   `Statement` variant set (intended). Derive the same traits the sibling variants derive
   (`Debug/Clone/PartialEq/Serialize`) so `-json` AST dumps and golden tests keep working.
3. **Grammar.** Add the `LET <ident> = <statement>` production to `crates/parser/src/grammar.rs` and
   allow a bound bare identifier as a `Source`. Add parser golden tests covering a two-`LET` binding
   example (the roadmap §1.2 example, in today's `LET`/`FROM`/uppercase rendering until the
   operator/`Resource`/lowercase tickets land); keep the existing corpus green. `cargo test -p qfs-parser`.
4. **Resolve + eval environment.** Thread a binding scope through `Resolver`
   (`crates/core/src/resolve.rs`) and `Evaluator::eval_statement` (`crates/core/src/eval.rs`):
   binding-name lookup precedes mount lookup; a referenced binding evaluates to its stored
   `PlanSource`; add the unknown-binding `EvalError` arm. Pure, in-process tests only.
5. **Exhaustive handling + docs.** Make every exhaustive `match` on `Statement` in `crates/core`
   handle the new arm (the compiler enforces this). Run the full gate:
   `cargo build/test/clippy --workspace`, `cargo fmt --all --check`,
   `cargo run -p xtask -- gen-docs --check` — the generated `docs/language.md` must regenerate to
   include `LET` (do not hand-edit it; run `gen-docs`). Bump the patch in `crates/qfs/Cargo.toml`.

## Considerations
- **Safety floor is preserved trivially.** `LET` binds *queries* (pure, read/transform); it adds no
  effect node, so describe-is-pure / preview-touches-nothing / commit-is-explicit /
  irreversible-needs-ack are all untouched. A `LET` whose value contains an effect statement should
  be rejected (or only the relational form allowed) — flag and prefer: `LET` binds a relation
  source only, not an effect, so it can never smuggle a write into a pure context.
- **Governance — this is the area where it matters most.** `LET` is a *genuine* new keyword and must
  pass through both tripwires (the `KEYWORDS`/count freeze in `crates/lang/src/keywords.rs` and the
  `Statement` variant-set governance test in `crates/parser`). Updating them is intended here and
  must be called out in the PR as a deliberate closed-core vocabulary change (decision H) — contrast
  with t61 (lambdas/`map`/`filter`/`reduce`) which add **zero** keywords.
- **Dep-direction (`crates/cmd/tests/dep_direction.rs`).** All changes live in the pure cores
  (lang/parser/core); no new crate, no driver/runtime edge, so `qfs-cmd` and the terminal leaves are
  untouched and tokio stays out. The cores remain wasm-buildable.
- **Scoping is an open design choice to flag, not guess.** Decide and document: are bindings
  top-level (visible to all following statements in a script) or block-scoped; is shadowing allowed;
  is the value evaluated lazily at reference or once at bind. Roadmap §1.2 shows top-level,
  reference-by-name reuse — prefer that, and keep semantics conservative (no recursive/forward
  references) so a later relaxation is non-breaking.
- **Docs honesty.** Do not advertise `LET` in `docs/guide/*` or the skill until this slice ships and
  `gen-docs --check` is green; the roadmap already tags it 🧭 proposed, which is correct until merge.
- **Versioning.** Own PR + patch bump in `crates/qfs/Cargo.toml` (currently 0.0.7) + a `v0.0.x` tag
  on ship.
