# t64 — `/hosts/<host>/claude/...` driver (AI sessions under the host realm)

## Overview

Delivers the **`/hosts/<host>/claude/...` driver** that turns each machine's Claude Code sessions into queryable
paths (M7, roadmap §3.3, decision L): `FROM /hosts/<host>/claude/sessions |> WHERE status='running'`
reads what an agent is doing, and `INSERT INTO /hosts/<host>/claude/sessions/current/instructions VALUES
(...)` steers it. The host realm `/hosts/<host>/...` comes from **decision P / §1.3** (it supersedes
the old top-level `/claude/...`): Claude sessions are a service *under* the host, not a top-level
mount. Reachable across the t63 tunnel, it makes a fleet of machines one queryable
surface. The driver itself is genuinely new; what exists to build on is the full driver contract
(`crates/driver/src/lib.rs` `pub trait Driver` with a pure introspective half proven by
`fixture_driver_introspection_is_pure`) and a complete template driver (`crates/driver-local`)
whose read + append shapes map directly onto sessions (read) and instructions (append-log). The
driver exposes exactly two archetypes — `RelationalTable` for `/sessions` (read) and `AppendLog`
for `.../instructions` (append) — and adds ZERO keywords (a new path family is a registry edit, not
a grammar change).

## Exact seams

- `crates/driver/src/lib.rs` — `pub trait Driver` (`mount`/`id`/`describe`/`capabilities`/
  `procedures`/`pushdown`/`prelude`/`version_support`/`plan_write`/`applier`), `Archetype`
  (`RelationalTable` for sessions, `AppendLog` for instructions), `Capabilities`, `Verb`,
  `NodeDesc`, `ProcSig`. The introspective half MUST stay pure (proof test
  `fixture_driver_introspection_is_pure`); only `applier()` is impure.
- `crates/driver-local/` — the template: `lib.rs` `LocalFsDriver`, `applier.rs` `LocalApplier`,
  `effect.rs`, `read.rs` `scan_rows`, `row.rs`, and the apply bridge `local_apply_driver()` +
  `qfs_runtime::PlanApplierBridge`. Mirror this structure for `qfs-driver-claude`.
- `crates/driver/src/lib.rs` `plan_write` — the per-driver write-planning seam (the same hook
  `GitDriver` uses): `INSERT INTO .../instructions VALUES (...)` lowers to an append effect through
  this seam, not the generic node.
- `crates/plan/src/node.rs` `EffectKind` (`Insert` for the append; sessions are read-only —
  `is_inherently_irreversible()` stays false for an append), `EffectNode`.
- `crates/core/src/registry.rs` `MountRegistry::{register,resolve,resolve_path}` — register
  the host realm's `claude/...` session service (mounted under `/hosts/<host>/`) here. Describe
  registry (cred-free): `crates/qfs/src/describe.rs` `describe_registry()`.
  Live apply registry: `crates/qfs/src/commit.rs` `live_registry()`.
- t63 tunnel: `<host>` in `/hosts/<host>/claude/...` resolves either locally or over the relay; the
  cross-machine call rides the t63 seam and is gated by `crates/server/src/policy/enforce.rs`
  `evaluate`.
- t47 MCP tools (`describe`/`preview`/`commit`) — the agent reaches `/hosts/<host>/claude/...` through the same
  MCP surface as any other path; no special-casing.
- `crates/cmd/tests/dep_direction.rs` — add `qfs-driver-claude` to the `qfs-driver-*`
  runtime-consumer allowlist; live wiring lands on `crates/qfs` only.

## Implementation steps

1. New crate `qfs-driver-claude`: implement the pure introspective half — `describe` returns the
   `/hosts/<host>/claude/sessions` schema (`RelationalTable`: `id, task, status, progress, last_message`)
   and `.../instructions` (`AppendLog`). Add a fixture-introspection test mirroring
   `fixture_driver_introspection_is_pure`. Tree green (pure tests only).
2. Read facet: implement `read.rs` `scan_rows` over a local session source — read Claude Code's
   on-disk session state (a `SessionSource` trait injected from the binary, so the pure driver names
   no filesystem path). `WHERE`/`SELECT` pushdown profile mirrors `driver-local`.
3. Append facet: implement `plan_write` so `INSERT INTO .../instructions VALUES (...)` lowers to a
   single append `EffectNode` (reversible — appending an instruction is not a `Remove`/irreversible
   `Call`); implement the `LocalApplier`-style applier behind a `qfs_runtime::PlanApplierBridge`.
4. Binary wiring: register the host realm's `claude/...` session service (under `/hosts/<host>/`) in `crates/qfs/src/describe.rs` `describe_registry()` (pure,
   always available) and in `crates/qfs/src/commit.rs` `live_registry()` (apply, gated). Inject the
   real `SessionSource` from `crates/qfs`. Add to `crates/cmd/tests/dep_direction.rs`.
5. Cross-machine: resolve `/hosts/<host>/claude/...` where `<host> != local` through the t63 tunnel; the
   destination node serves it via its own local `claude/...` session service. Add an audit entry per remote
   steer (`crates/server/src/audit.rs`).
6. Docs + version: document `/hosts/<host>/claude/...` only after a live local read + append smoke passes;
   regenerate `docs/drivers.md` via `cargo run -p xtask -- gen-docs` (never hand-edit). Patch-bump
   `crates/qfs/Cargo.toml`.

## Considerations

- **Safety floor.** `FROM /hosts/<host>/claude/sessions` is a pure read (describe/preview touch nothing);
  steering an agent is an `INSERT` (reversible append) that commits explicitly. Removing or aborting
  a session, if ever added, would be `EffectKind::Remove` → irreversible → extra acknowledgement —
  do NOT model "stop the agent" as a silent reversible op.
- **Purity discipline.** The driver's introspective half names no filesystem path and does no I/O —
  the real session location is injected from `crates/qfs` via a `SessionSource` trait, keeping the
  driver wasm-buildable and the proof test green.
- **Zero new keywords.** A new path family adds nothing to `crates/lang/src/keywords.rs` (frozen at
  38); this is a registry + driver edit only.
- **Cross-machine = same policy.** Reaching `/hosts/<remote>/claude/...` rides t63 and re-checks `POLICY`
  at the destination (default-deny); a teammate cannot steer an agent on a host their policy does not
  grant.
- **Dep-direction.** New driver edges land on `crates/qfs`; add the crate to the allowlist in
  `crates/cmd/tests/dep_direction.rs`.
- **Open product decision to FLAG:** the exact session-state source (Claude Code's on-disk format
  vs. a local IPC/API) and how "current" session is identified are implementation-dependent and may
  shift across Claude Code versions — name the coupling in the PR rather than hard-coding a format.
- **Honesty-first + versioning:** keep `/hosts/<host>/claude/...` out of docs/skill until the smoke passes; own
  PR + patch bump + `v0.0.x` tag on ship.
