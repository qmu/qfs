# Architect Review — t33 JOB scheduler (`qfs-cron`)

**Reviewer**: Architect (Neutral / structural bridge)
**Scope**: analytical review only (no test/build/clippy execution — disk at 99%)
**Commit**: `73feb82`
**Artifacts read**: ticket t33; `crates/cron/{Cargo.toml,lib.rs,scheduler.rs,schedule.rs,lastrun.rs,policy.rs,hash.rs,store.rs,commit.rs,binding.rs,daemon.rs,clock.rs,tests.rs}`; `crates/qfs/src/{cron.rs,serve.rs,main.rs}`; `crates/http/src/serve.rs`; `crates/cmd/tests/dep_direction.rs`.

## Decision

**Approve with observations.**

The structural translation is faithful and consistent with the established E7 leaf pattern. The three headline questions all resolve in favor of the implementation: the leaf topology + wasm split is the correct, consistent E7 pattern; the idempotency design is sound under its stated model; and the `LAST_RUN()` advance ordering is correct incremental-window semantics, immune to the t32 param-shadow concern by node-type construction. The observations below are non-blocking (one correctness edge in first-run scheduling, two minor dead-code/doc items, and the standing crypto-core carry-over which I now rule is ripe).

---

## Headline 1 — `qfs-cron` leaf topology + wasm split: RIGHT and CONSISTENT

The crate mirrors `qfs-http` exactly: a leaf consuming `qfs-server` (Binding/registry seam) + `qfs-exec` (the `build_plan` evaluator), consumed only by the terminal `qfs` binary, off `qfs-runtime`. This is the correct E7 pattern and the placement rationale holds: it cannot live in `qfs-server` (must stay off `qfs-exec`, runtime-free, CO-t30-1) nor in `qfs-cmd` (off `qfs-exec`/binding crates).

**Pure/native split is clean.** The fence is the absence of `native` (the t25/slack lesson applied correctly), not a marker:
- `native = ["dep:qfs-exec", "dep:qfs-server", "dep:tokio"]` — all three optional, all three the things that genuinely cannot be wasm-clean (qfs-exec/qfs-server both carry non-optional tokio; tokio is the daemon loop).
- The pure core (`schedule`, `policy`, `scheduler`, `lastrun`, `hash`, `store` trait + `MemJobStore`, `clock` trait + `MockClock`, `commit` trait) depends only on `qfs-core` + `qfs-parser` (both tokio-free spine crates). No tokio/runtime type leaks into the pure surface: `Scheduler::tick`/`dispatch` are **synchronous** and return owned `Vec<Dispatched>`; the only async is `daemon::run_daemon`, correctly `#![cfg(feature = "native")]`-gated. `SystemClock` and `RecordingCommitter` (which pull `SystemTime`/`qfs-exec`) are `native`-gated; `MockClock` (uses `Cell`, wasm-safe) and the trait stay in the core. This is structurally correct.

**Guard coverage is complete.** The new `cron_binding_is_a_leaf_serve_consumer` pins all four invariants: (a) qfs-cron → {qfs-server, qfs-exec} present under default features; (b) leaf — only `qfs` may depend on it; (c) NOT → qfs-runtime. The allowlist extensions are consistent: `exec_is_confined_above_the_spine_and_off_the_runtime` admits qfs-cron as the fourth exec consumer; `binary_is_the_thin_entrypoint...` admits `qfs-cron` to the binary's allowed workspace-dep set. Critically, **qfs-cron is correctly absent from `runtime_consumers_allowed`** — because it does not depend on qfs-runtime, the runtime-leaf-confinement guard is genuinely untouched, and qfs-server stays runtime-free. Tokio dead-ends in the binary (the daemon's `tokio::spawn` in `serve.rs` is in the terminal sink). No spine inversion.

> **Observation 1 (structural, non-blocking).** The guard `cron_binding_is_a_leaf_serve_consumer` asserts the `qfs-server`/`qfs-exec` edges are present *under default features*. That is correct for the workspace build, but the load-bearing wasm property — "with `--no-default-features` the core has neither edge" — is asserted only by the doc comment and (implicitly) by a wasm build that this trip cannot run (disk constraint). The t25/slack reviews accepted the same shape, so this is consistent, not a regression. **Proposal**: when disk permits (post-trip), add a CI lane that runs `cargo tree -p qfs-cron --no-default-features` and greps that `tokio`/`qfs-exec`/`qfs-server` are absent — a mechanical analogue of the dep_direction guard for the *negative* (wasm) direction. Carry this as a shared follow-up for qfs-http and qfs-cron together (both rely on the same unverified-in-this-env property).

## Headline 2 — Idempotency / effectively-once: SOUND under the stated model

The design genuinely composes at-least-once delivery (failed/missed windows re-cover) with idempotent effects to reach effectively-once. Adjudicating the three sub-questions:

**(a) `run_id = hash(job, scheduled_for)` dedups a retried fire while allowing distinct windows.** Correct. `run_id_for` hashes `"<job>\0<scheduled_for>"` via the vendored SHA-256, truncated to 16 hex chars. Same `(job, scheduled_for)` → same run-id → `is_committed` short-circuits at step (2). Distinct `scheduled_for` (or distinct job) → distinct run-id → runs. The NUL separator prevents the `("ab", 1)` vs `("a", "b1")` ambiguity class. The determinism is the right call over a random UUID precisely because it makes the dedup key recomputable on retry without consulting the store. Truncation to 64 bits is fine for a non-adversarial dedup key (the material is a job name + an epoch second, both controlled; this is not a security boundary — see Headline crypto ruling).

**(b) Two concurrent dispatch for the same due time → exactly one commit.** Correct *in the pure model*. The ordering in `dispatch` is: `is_committed` check → `acquire_lease` → (under lease) re-read `run_state` → rewrite → commit → `record_run` → `release_lease`. The single-flight property rests entirely on `acquire_lease` being atomic-compare-and-set in the `JobStore` impl. The `MemJobStore`/`LedgerJobStore` impls satisfy this via a `Mutex`-guarded `contains_key`-then-`insert` (atomic under the lock). The doc and the `concurrent_dispatch_same_due_commits_once` test correctly acknowledge that real cross-replica leasing (a CAS over D1/a DO) is a JobStore-impl concern, not the scheduler's — the scheduler's contract (`acquired: bool`) is the right seam. This is honestly scoped.

**(c) UPSERT/@version reliance given the JOB DO plan is arbitrary.** This is the soft spot, and it is honestly documented rather than overclaimed. The scheduler itself guarantees only *single-flight + ledger dedup* (the lease + run-id). Effect-level idempotency (so a torn retry after a crash *between* commit-apply and `record_run` is a safe no-op rather than a double-apply) is delegated to the DO body using `UPSERT`/`@version`, which the scheduler cannot enforce — an arbitrary `INSERT`-bodied JOB that crashes post-apply, pre-`record_run` would re-fire the same window (the run-id is NOT yet in `committed`) and double-apply. The ticket §Considerations states exactly this ("effects themselves stay idempotent … so the worst case is a safe no-op"), and the run-id dedup narrows the window to the apply↔record gap. **This is the correct boundary**: the scheduler delivers at-least-once + a dedup key; idempotent-effect authorship is the JOB author's contract. I rule the design sound *as a scheduler*; the residual double-apply window is inherent to at-least-once and is correctly pushed to the effect layer, not hidden.

> **Observation 2 (structural fidelity, non-blocking).** The `committed` set is written inside `record_run(Success)` (store.rs:329, qfs/cron.rs:125), i.e. AFTER the commit's apply already happened. So `is_committed` protects against a *retried dispatch after a fully-recorded success*, not against a crash in the apply↔record window. That is the correct and only achievable guarantee for an at-least-once scheduler over a non-transactional commit+ledger, and it matches the ticket. **Proposal**: add one sentence to the `commit.rs`/`scheduler.rs` doc making the boundary explicit for future readers — "the run-id dedup covers retried *completed* dispatches; a crash between apply and `record_run` re-covers the window and relies on effect idempotency (UPSERT/@version)" — so the apply↔record gap is documented at the code, not only in the ticket. No behavior change.

## Headline 3 — `LAST_RUN()` advance ordering + correctness: CORRECT, and immune to the t32 param-shadow concern

**Advance ordering is correct incremental-window semantics.** `record_run(Success)` sets `last_run_at = Some(scheduled_for)` (the boundary, NOT `now`) and the `Failed`/`Disabled` arms leave `last_run_at` untouched (store.rs:322-340, qfs/cron.rs:119-134). So:
- A successful run advances the high-water mark to the window boundary it covered → the next due-set computation enumerates `(scheduled_for, now]` with no gap and no overlap. No skipped window.
- A failed run leaves `last_run_at` unmoved → the next `tick` re-enumerates the same window → at-least-once re-cover. No double-count of a *committed* window (the failed one never advanced).
- Storing `scheduled_for` rather than `now` is the load-bearing detail: on a slow run that finishes at `now > scheduled_for + interval`, advancing to `now` would silently skip `(scheduled_for, now]`; advancing to `scheduled_for` preserves the incremental contract. Verified by `dispatch_previews_plan_with_last_run_resolved` (advances to 7200, not 10000) and `failed_run_leaves_last_run_at_unmoved_and_recovers`. Correct.

**`LAST_RUN()` is injection-safe and does NOT inherit the t32 param-shadow concern.** This is the crux and it resolves cleanly. The t32 `bind_params` shadow concern was that a `$param` placeholder could be confused with a same-named *column reference* (a `Col`/`Path` node). `LAST_RUN()` is structurally distinct: `is_last_run` matches *only* an `Expr::Fn` with `name.eq_ignore_ascii_case("LAST_RUN")` **and `args.is_empty()`** (lastrun.rs:187-189). A column named `last_run`, a path segment `last_run`, or a one-arg `LAST_RUN(x)` user function are all different AST node variants (`Expr::Col`/`Expr::Path`/`Expr::Fn` with non-empty args) and are *not* rewritten. There is no column-collision surface because the binding is a function-call node, not a name lookup against the row scope. The rewrite replaces the whole `Expr::Fn` node with one `Expr::Lit(Literal::Int(boundary))` leaf — zero re-parse, structurally identical for any boundary — so it is the same injection-safe AST-rewrite discipline as `bind_params` but immune to the shadow class by construction. The rewrite walks every Expr-bearing position (verified against the full `PipeOp`/`Source`/`EffectBody`/`Statement` match — including `Join.on`, set operations, `Call` args, `Between`/`Like`/`In`/`AnyOp`, and DDL `do_plan`/`as_query` recursion), and the parallel `detect_stmt`/`detect_expr` walk used by `references_last_run` covers the same surface. The scoping claim holds: `LAST_RUN` is never registered into the stdlib namespace — substitution runs only here on the DO body at dispatch, so it cannot leak into arbitrary statements. **I rule this distinct from and stronger than the t32 concern.**

## Other surfaces

**1. Dep discipline (no chrono/uuid).** Sound. `Instant = i64` epoch seconds matches `JobDef.last_run`/`Value::Timestamp`; `civil_from_epoch_secs` uses Hinnant's `civil_from_days` (the well-known exact integer algorithm) with `div_euclid`/`rem_euclid` so negative epochs would still behave (the scheduler only feeds non-negative). The cron goldens I traced by hand confirm correctness: `*/15 * * * *` → 900 (00:15); `0 */2 * * *` → 7200 (02:00); `30 9 * * 1-5` from epoch 0 (Thu, dow=4) → 34200 (09:30 same Thu) — the dow computation `((days % 7) + 4).rem_euclid(7)` gives 4 (Thu) at days=0, which is in `1-5`, so the same-day 09:30 is correctly the next match. The 4-year bounded look-ahead (`MAX_MINUTES`) correctly terminates a never-matching expression (e.g. `* * 30 2 *` — Feb 30) with `None` rather than looping. `next_after` for `Every` is the right `anchor + n·interval` with `n = delta/interval + 1` for `delta >= 0` and `n = 0` for `delta < 0` (fires at the anchor when `from` precedes it).

**2. SHA-256 vendored a THIRD time — RULING: the crypto-core extraction carry-over is now RIPE; do it as part of t37/t38, before the trip ends.** This is now the strongest case in the trip. SHA-256 is independently vendored in `qfs-driver-objstore` (t22, SigV4 signing), `qfs-driver-slack` (t25, HMAC signing), and now `qfs-cron` (t33, idempotency-key + plan fingerprint). Three copies of a security-relevant primitive is a divergence hazard: a fix or hardening to one (e.g. a constant-time comparison helper, a streaming API, a correctness fix to the padding) does not propagate, and the objstore/slack copies are on the *signing* path where a divergence is a security bug, not merely a wrong hash. **Urgency ruling: yes, worth doing before the trip ends.** The natural home is a new pure leaf `qfs-crypto-core` (no workspace deps, wasm-clean, the same shape as `qfs-http-core` single-sourcing the redaction set in `http_core_is_a_pure_leaf...`) consumed by all three driver/cron crates, with a dep_direction guard asserting (a) it is a pure leaf carrying no `tokio`/`ring`/`openssl`, and (b) the three consumers depend on it rather than hand-rolling. **The cron copy is genuinely lower-stakes** than objstore/slack: it derives a *public* run-id and a plan fingerprint from non-secret material (job name + epoch + plan structure) and never compares secrets — the hash.rs doc states this correctly and there is no timing-attack surface. So if t37/t38 capacity is tight, the *signing* copies (objstore/slack) are the priority to converge; cron can follow. But the extraction itself should land this trip — the third copy is the signal that "we'll deduplicate later" has accrued enough debt. **Proposal**: file/elevate the `qfs-crypto-core` carry-over as a t37-security or t38 sub-item with the guard described above; converge the two signing copies first, then cron.

**3. MissedPolicy due-set folding — correct + bounded.** `Skip`/`Coalesce` → newest boundary only (one run); `CatchUp{max}` → oldest `min(max, len)` in order; default = `Coalesce` (no thundering catch-up). The `MAX_ENUMERATED = 10_000` cap bounds the enumeration before folding so a years-behind JOB cannot allocate unbounded — and since `Skip`/`Coalesce` collapse to one and `CatchUp` is capped at `max`, the materialized vector is the only bounded cost. Verified by the four `missed_policy_*` tests. Correct and bounded.

> **Observation 3 (correctness edge, low-severity, non-blocking).** First-run scheduling uses a **fixed 24h LOOKBACK** (`policy.rs:100`, `first_run_due`): on a JOB's first eligibility it picks the latest boundary `<= now` within the last 24 hours. For an `Every` interval **longer than 24h** (e.g. `EVERY 7d` / a weekly cron `0 0 * * 0`), the most recent boundary may be >24h before `now`, so `first_run_due` returns empty and the JOB does **not** fire on its first eligible tick — it only fires once `now` reaches the *next* boundary (correct cadence thereafter, but the first window is silently deferred up to one full interval). This is a missed-first-fire, not a correctness violation of the incremental contract (once `last_run_at` is set the `(from, now]` enumeration is exact), and the goldens/tests all use sub-24h intervals so it is untested. **Proposal**: derive the look-back from the schedule rather than a constant — for `Every { interval, .. }` look back `interval` seconds (or `interval.max(LOOKBACK)`); for `Cron` keep the 24h (or 1-interval-equivalent) window. Alternatively, document that first-run for intervals >24h defers to the next boundary as intended behavior. Either resolves the surprise; the code change is small and local to `first_run_due`.

**4. Cron parse — correct, structured, no panic.** Restricted 5-field, validated at load via `CronExpr::parse` → `CronField::parse` per field with explicit domains (min 0-59, hour 0-23, dom 1-31, month 1-12, dow 0-6). Supports `*`, `a,b`, `a-b`, `*/step`, bare `n`; out-of-range/malformed → structured `ScheduleError::{FieldCount,OutOfRange,BadField}` (never a panic), the same discipline as the parser's other load-time errors. Fields pre-expand to a sorted/deduped allowed-set so matching is a `binary_search` membership test (no per-tick re-parse). Verified by `invalid_cron_is_structured_error_not_panic`.

**5. Purity — upheld.** The scheduler constructs no effects: `dispatch` clones the pre-parsed `PlanSpec` statement, applies the `LAST_RUN()` literal rewrite, and hands it to the injected `Committer::commit` — it never builds a `Plan` itself nor performs I/O. The DO body is the t31 canonical spec rehydrated via `PlanSpec::from_canonical` (binding.rs:123) — never stored as a live Plan. The `JobStore`/`Clock`/`Committer` traits are the only I/O surface and they are all injected. Structurally clean against the §3 purity invariant.

**6. Secret hygiene — clean.** `RunRecord`/`RunState`/ledger carry only `job`, `run_id`, `scheduled_for`, `status`, `applied_count` (a u64), `plan_hash` (a SHA-256 hex of the preview projection), and a `failure_note` sourced from `CommitError` (which is documented secret-free). No DO payload, no credentials. `RunRecord::log_line` projects only those scalar fields. `commit.rs` derives `plan_hash` from `qfs_core::preview(&plan)`'s debug form (a structural projection), never the source. The `reconcile` refusal path logs the JOB name + reason class only, never the body/interval payload (binding.rs:80-86). Verified by `audit_record_carries_no_plan_payload` (asserts both `log_line` and the full `Debug` exclude a token-like literal). Correct.

**7. PREVIEW-grade committer park — honestly documented, not an overclaim.** `qfs/src/cron.rs` `PreviewCommitter` wraps `qfs-cron`'s `RecordingCommitter`, which builds the plan via `qfs_exec::build_plan` and applies over a `qfs_core::RecordingApplier` — the same no-creds PREVIEW state the t32 HTTP *read* drivers are in. The module doc (cron.rs:9-19) states explicitly that routing a JOB's real-effect COMMIT through the runtime `Interpreter` with live drivers is deferred to t34/t35/t38, and that durable run-state persistence (the in-memory `LedgerJobStore` vs. `/server`/DO) is parked. This is consistent with the rest of the serve composition and does NOT overclaim that JOBs run real effects today. The honesty is appropriate; the seam (the injected `Committer`) is exactly what lets t38 swap the real applier in without touching the scheduler. Consistent.

**8. Circuit-breaker + PolicyRef seam — correct, no widening.** `on_failure` records `Failed` (which bumps `consecutive_failures` in `record_run`), then re-reads the counter and, at `>= MAX_CONSECUTIVE_FAILURES` (5), records a `Disabled` note with a secret-free message. The ordering is correct: the counter is incremented by the `Failed` record *before* the threshold read, so the 5th consecutive failure (counter=5) trips; `circuit_breaker_auto_disables_after_repeated_failures` fires 6 boundaries and asserts a `Disabled` note appears. The `Disabled` `last_status` is also belt-and-suspenders-checked at the top of `tick` (scheduler.rs:108). `PolicyRef` is threaded into `commit(&stmt, &job.policy)` and never widened (binding currently defaults it to empty pending the sibling POLICY ticket — correctly noted as not-yet-wired rather than silently dropped). The seam does not widen the binding's surface. Correct.

> **Observation 4 (dead code / minor, non-blocking).** Two trivial items: (i) `policy.rs:83` `vec![*boundaries.last().unwrap_or(&from)]` — the `unwrap_or(&from)` branch is unreachable because `boundaries.is_empty()` returns early at line 76; harmless but slightly misleading. (ii) `qfs/src/cron.rs:133` has both an explicit `RunStatus::Never => {}` arm and a trailing `_ => {}` catch-all in the same `match` — the catch-all is redundant given the explicit arms cover the (non-`#[non_exhaustive]`-at-use) variants; it will mask any *future* `RunStatus` variant silently (whereas `MemJobStore::record_run` in store.rs:323-340 has no catch-all and would force a compile error on a new variant — the safer shape). **Proposal**: drop the `unwrap_or(&from)` (use `boundaries[boundaries.len()-1]` or keep the early-return and `expect`), and remove the `_ => {}` in `qfs/src/cron.rs` so a new `RunStatus` variant is a compile error in both stores symmetrically. Cosmetic; no behavior change.

## Cross-cutting coherence

The change is structurally coherent with the E7 line: `qfs-cron` is the cron sibling of `qfs-http`, wired by the same `serve_config_with(... extra_bindings)` seam (a clean, generic `Vec<Box<dyn Binding>>` extension so qfs-http gains no qfs-cron dependency — the two leaves cross only through the `Binding` trait). The dep_direction guard suite is extended in lockstep (four invariants pinned, allowlists extended with documented rationale, runtime-consumer allowlist correctly NOT touched). Translation fidelity from the ticket to the implementation is high: every hard-part and acceptance criterion in the ticket maps to a named module + test. The two genuine residuals — the >24h first-run defer (Observation 3) and the standing crypto-core triplication (Other-surface 2) — are the only items I would want tracked; neither blocks acceptance of t33.
