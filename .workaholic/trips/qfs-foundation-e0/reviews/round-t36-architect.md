- **Reviewer**: Architect (neutral / structural)
- **Commit**: `f9eb0fe` (Constructor work, recovered + committed by Lead after a transient API 500 on the Constructor's final return — no Constructor self-report; this review derives from code + ADR-0005)

- **QA mode**: analytical review only (no test/build/clippy execution — disk at ~99%; the Lead is running the build verifications separately and reports native build + `cargo test --workspace` 1128 passed + `clippy --workspace --all-targets -D warnings` green).

## Decision

**Approve with observations.**

All three headline invariants hold structurally: (1) the `qfs-host` seam is a genuine thin formalization over the existing t32/t33/t34 serve composition — no serve logic is duplicated, and the two new daemon primitives are crash-safe; (2) the parked `host-workers` decision is proportionate, reversible through one mapping module, and — critically — the ADR framing is **honest**, not an overclaim that the CF host runs today; (3) the no-vendor-in-core deny-test genuinely resolves the real dep graph, names the right forbidden crates, and would fail if a vendor edge were added to core. The feature-gating keeps tokio dead-ending in the terminal binary with no spine inversion. Observations below are refinements and one honest-scoping nuance on the daemon "attach", not blockers.

## Headline ruling 1 — The `qfs-host` seam + the daemon-reuses-existing-composition claim

**Ruling: the daemon is genuinely a thin formalization; the two new primitives are crash-safe; the seam invents no new runtime semantics.**

### Seam fidelity (no new runtime semantics)
- `RuntimeHost` (`crates/host/src/host.rs`) carries exactly the cause-attachment contract the ticket specifies — `now`/`serve_endpoints`/`schedule_jobs`/`consume_events`/`durable`/`native_store` — over **owned DTOs only**. No `tokio::*`/`worker::*`/AWS/CF type appears on the trait surface; the only return shapes are `BindingSet`, `HostError`, `&dyn DurableStore`, `Option<NativeStoreHandle>`. The `BindingSet` (`dto.rs`) is the single deployment contract both hosts consume identically — adding a deployment adds zero keywords, exactly as the closed-core invariant requires.
- The async-fn-in-trait choice on `RuntimeHost` (non-dyn, used as a concrete type per host) vs the hand-rolled `HostFuture = Pin<Box<dyn Future>>` on `DurableStore` (dyn-safe because `durable()` returns `&dyn DurableStore`) is the correct structural split, and it is what lets the wasm-clean core avoid pulling `async-trait`. The doc comment (`host.rs` lines 9–19) states this precisely. `HostFuture` is intentionally `!Send` so a CF Durable Object's single-threaded `!Send` futures are expressible — a forward-looking correctness call, not an accident.

### Daemon reuse (no duplicated serve logic)
- `crates/qfs/src/host.rs` `TokioHost` is the trait projection of the **already-wired** composition. `serve_endpoints`/`schedule_jobs`/`consume_events` do NOT spawn listeners/intervals/buses — they record the cause-attachment counts to the on-disk ledger and return `Ok(())`. The actual listener (`qfs-http`), interval (`qfs-cron`), and bus + `/hooks` ingest (`qfs-watchtower`) remain wired in `serve.rs` lines 68–130, untouched. This is a faithful "formalize, don't rebuild" — verified by reading both files: there is no second `serve_config`, no second `Scheduler`, no second bus in the host crate.
- `serve.rs::attach_daemon_host` boots the config once more (in-memory, idempotent) ONLY to project the `BindingSet`, builds `TokioHost` over a project-local state dir, and drives the three non-suspending attach futures with `qfs_host::block_on` so the attach stays off the tokio runtime built below. A host-setup failure is explicitly **non-fatal** (line 48–50: degrade to the legacy in-memory composition with a `tracing::warn!`) — an honest additive-observability posture, not a hard dependency.

### Crash-safety of the two new primitives — sound
- **`FileDurableStore`** (`daemon.rs`): `write_sync` does write-temp → `f.sync_all()` (fsync the data) → `fs::rename` (atomic replace) → directory `sync_all()` (make the rename durable). This is the textbook crash-safe atomic-file-update sequence; a crash mid-write leaves either the old file intact or the fully-written new file, never a torn cursor. The dir-fsync is best-effort (`let _ = dir.sync_all()`) which is the correct degradation for platforms that reject a dir fsync — the rename is already ordered. `cas` takes a per-store `Mutex` so the read-modify-write is serialized (single-flight) — the at-least-once primitive is real, not advisory.
- **`AuditLedger`** (`daemon.rs`): append-only, `writeln!` + `sync_all()` before returning, under a `Mutex<File>`. Each line is fsync'd before the call returns, so a crash never loses a recorded fire. Lines are NAMES+OPS only (the caller passes the already-secret-free summary). This correctly replaces the in-memory `drain` for a long-lived daemon.
- `path_for` sanitizes `/`→`_` so `watcher/notify/cursor` maps to one flat file with **no directory traversal** — a `..` or absolute key cannot escape the state dir. Good.

**Observation 1 (the "attach" is a ledger-record, not a literal re-binding — honest-scoping nuance).** `TokioHost::serve_endpoints/schedule_jobs/consume_events` record attachment counts to the ledger; they do not themselves install the causes (those are wired in `serve.rs`). This is correct and intentional, but it means the `RuntimeHost` impl on the daemon is currently an **observability shim over the real composition**, not the literal owner of the listener/interval/bus. The structural risk is drift: a future change to `serve.rs`'s wiring would not be reflected by the ledger counts unless the binding-set projection stays in sync. **Proposal**: leave the t36 shape as-is (it is the honest "formalize first, migrate the wiring under the trait later" step the ADR describes), but record a carry-over note that the *next* step is to move the actual listener/interval/bus construction behind `TokioHost`'s methods so the trait becomes the literal composition root rather than a parallel record. The ADR's "REUSES, does not rebuild" wording is accurate for today; the carry-over keeps the eventual full migration visible.

## Headline ruling 2 — The parked `host-workers` decision (ADR-0005): proportionate, reversible, and honestly framed

**Ruling: right call; reversibility seam is genuine; the ADR framing is HONEST (not an overclaim that the CF host runs today).**

- **Proportionate + consistent with precedent.** ADR-0005's reasoning explicitly mirrors ADR-0002/0003/0004 (winnow over chumsky, in-house engine over DuckDB, in-house HTTP over axum): an uncached heavy dependency tree on a ~99%-full disk is exactly the footprint risk those ADRs codified. The `worker` crate is confirmed unresolvable offline (the ADR and `workers.rs` doc both state `cargo` reports "no matching package named `worker`" under `--offline`). Parking is the consistent, disk-safe call.
- **Reversibility seam is genuine (one mapping module).** `crates/host/src/workers.rs` is the single boundary: `endpoint_event`→`Fetch`, `job_event`→`Scheduled`, `webhook_event`→`Queue`, `watcher_event`→`(DurableObject, "WatchtowerState")`, `native_store_handle`/`resolve_native_store`→name-only handle. The `CfEvent` enum pins the archetype→primitive mapping the real `#[event]`/`#[durable_object]` handlers will mechanically fill in. The parked `DurableObjectStore` returns a **structured `HostError::Durable`** on every `get`/`put`/`cas` (never a panic) — a caller reaching it under `host-workers` fails cleanly. The drop-in is real: the eventual `worker`-backed handlers replace only this module's bodies; the DTO wiring and mapping table are already pinned and tested (`archetype_mapping_is_pinned`).
- **The acceptance criterion is honestly met by the non-`worker` scaffold.** The criterion is `cargo build --target wasm32-unknown-unknown --features host-workers` *succeeds*. The scaffold takes NO `worker` dependency (owned DTOs + mapping table only), so it genuinely builds for wasm32 — the build the criterion names truly passes (the Lead is verifying it). The ADR does NOT claim the CF host runs end-to-end; "Scope of what built vs. parked" explicitly lists the `worker`-backed `WorkersHost`, the four `#[event]` entrypoints, the `#[durable_object]` class, and the `qfs.wasm` cdylib as **parked**, and the Consequences section states plainly "the CF Workers host is not end-to-end runnable today." This is honest framing: the *build* passes, the *runtime* is parked, and both are recorded. No overclaim.

**Observation 2 (a `webhook_event` is exported asymmetrically vs the others — minor surface tidiness).** `lib.rs`'s `host-workers` re-export list (lines 78–82) exports `endpoint_event`, `job_event`, `watcher_event`, `native_store_handle`, `resolve_native_store`, `CfEvent`, `DurableObjectStore` — but **not** `webhook_event`, even though it is `pub` in `workers.rs` and is part of the same archetype mapping (`WebhookBinding`→`Queue`). It is reachable as `workers::webhook_event` but not re-exported at the crate root like its three siblings. This is cosmetic (it does not affect the build or the parked semantics), but it is an inconsistency a reader will trip on. **Proposal**: add `webhook_event` to the `host-workers` re-export block so the four archetype mappers present a uniform crate-root surface, OR (if the omission was deliberate because the webhook path also flows through `endpoint_event`'s `/hooks` fetch) add a one-line doc note in `workers.rs` explaining why webhook is not re-exported. Either resolves the asymmetry.

## Headline ruling 3 — The no-vendor-in-core deny-test (the load-bearing invariant)

**Ruling: the deny-test genuinely enforces the vendor-free core; the feature-gating keeps tokio dead-ending in the terminal binary with no spine inversion.**

### The deny-test resolves the real graph and would actually fail on a vendor edge
- `no_vendor_in_core.rs::load_resolved_graph` runs `cargo metadata --format-version 1` **WITHOUT `--no-deps`** and walks `resolve.nodes[].dependencies` — the FULL transitive resolved graph, not the manifest's direct edges. `reachable_names` does a real transitive closure (DFS over resolved node ids). So a vendor crate pulled in **transitively** (not just as a direct edge) is caught — this is strictly stronger than the `dep_direction.rs` `--no-deps` manifest guards.
- `FORBIDDEN_IN_CORE` = `worker`, `worker-sys`, `hyper`, `axum`, `aws-sdk-s3`, `rusoto_s3` — the right set: the CF SDK + its sys crate, the two HTTP-server frameworks the in-house `qfs-http` replaces, and the two AWS storage SDKs the thin HTTP clients replace. The test asserts NONE is reachable from BOTH `qfs-core` and `qfs-server`. If someone added e.g. an `axum` edge to `qfs-server`, `reachable_names("qfs-server")` would contain `axum` and the assertion fires with a precise message. The guard is load-bearing and genuine.
- The deliberate omission of `tokio` from the forbidden set is **correctly reasoned and documented** (lines 11–15): `qfs-server` legitimately carries tokio `signal` for the run loop, and `qfs-core`'s tokio-freedom is already pinned by `qfs-plan`'s purity dep-closure test + the runtime-leaf-confinement guard. This test adds the NEW t36 confinement (the `worker` SDK + HTTP-server vendors), not a duplicate of the existing tokio fences. No gap: tokio-in-core is covered elsewhere; this test owns the vendor-SDK surface.
- The second test, `host_wasm_clean_core_is_free_of_worker_and_tokio`, is honestly scoped. Its comment candidly acknowledges that `cargo metadata` resolves the feature UNION across the workspace, so a sibling enabling `host-daemon` could pull `qfs-server` into `qfs-host`'s resolved deps — and rather than make a false claim, it asserts the **narrower robust property**: `worker` is absent from the ENTIRE workspace resolved graph. That is true (parked/uncached) and is the strongest claim that survives the feature-union caveat. This is exactly the kind of honest test-scoping I want to see — it does not pretend to prove a per-feature wasm closure that `cargo metadata` cannot express.

### Feature-gating + spine confinement
- `host-daemon = ["dep:qfs-server"]` — qfs-server (tokio-bearing) is an **optional** dep; the load-bearing wasm fence is the ABSENCE of the feature, not a marker (the ADR and Cargo.toml both call out "the t25/cron lesson"). `host-workers = []` pulls nothing. The `compile_error!` in `lib.rs` (lines 38–42) fires if both are set — one deployment target per binary, mechanically enforced.
- `dep_direction.rs::host_is_the_only_deployment_seam_and_a_leaf` (lines 989–1091) pins: (a) qfs-host's only workspace dep is the optional `qfs-server`; (b) qfs-host is a LEAF — only the terminal `qfs` binary may depend on it (`assert_eq!(consumer, "qfs")`); (c) qfs-host does NOT depend on qfs-runtime (it causes injected committers to run, it does not drive COMMIT); (d) qfs-host carries no `worker`/`hyper`/`axum`. The binary's allowlist (`binary_is_the_thin_entrypoint...`) admits `qfs-host` with the same composition-root rationale as the t32/t33/t34 leaves. So `host-daemon`'s tokio (via `qfs-server`'s `signal`) dead-ends in the terminal binary — no spine inversion. The runtime-leaf-confinement guard is untouched because qfs-host never appears in its consumer allowlist and never edges to qfs-runtime. Structurally sound.

**Observation 3 (the deny-test names the SDK crates, but a vendor pulled under a NEW name would slip — maintenance surface).** `FORBIDDEN_IN_CORE` is a fixed allowlist of crate names. It correctly catches the six named vendors, but it is a denylist, not a structural rule — if a future driver pulled, say, `worker-kv` or `aws-config` (a sibling of the named crates) into core, the test would not fire until someone added the name. This is an inherent limitation of name-based deny-tests, not a t36 defect. **Proposal**: keep the explicit denylist (it gives precise, actionable failure messages), but add a short comment block listing the *families* the denylist is meant to fence (CF Workers SDK family, HTTP-server frameworks, cloud storage SDKs) so a future maintainer adding a sibling vendor knows to extend the list. Optionally, a complementary positive assertion — "qfs-core's transitive vendor set ⊆ {the known-pure allowlist}" — would convert the denylist into an allowlist and catch unknown vendors, but that is a larger change appropriate for a later hardening ticket, not t36.

## Other surfaces

**1. wrangler.toml generator — host-agnostic, pure, name-only.** `wrangler.rs::generate_wrangler_toml` derives Cron/Queue/DO/d1·r2·kv bindings from the `BindingSet` by NAME only. The golden (`fixtures/wrangler.golden.toml`) matches the generated output (`deployment.rs::wrangler_toml_matches_the_checked_in_golden`, with `QFS_BLESS_GOLDEN=1` re-bless). Verified by reading the golden against the generator: `crons = ["0 */6 * * *"]` (from `EVERY 6h`), `class_name = "WatchtowerState"`, `queue = "inbound-events"`, `D1_ANALYTICS`/`R2_BACKUPS`/`KV_SESSIONS` binding names, `database_id = ""` left blank for the deployer. Output is deterministic (sorted crons, sorted queues, native stores pre-sorted+deduped in the `BindingSet`). The binding-set derivation (`derive.rs`) is pure over owned strings — the `from_server.rs` projection is the only `qfs-server`-coupled step, correctly gated behind `host-daemon`. **No token or `database_id` value is ever emitted** — confirmed in both the generator and the golden.

**2. Least-privilege / secrets — clean.** The daemon reads creds from the E5 store (the systemd unit comment + `serve.rs` use `qfs_secrets::Secrets`; no env-dump, no logging of secret values). `HostError` variants carry NAME/reason only (`host.rs` lines 28–40) — safe to log. The wrangler template references binding NAMES only and documents signing-secret HANDLES as comments (`wrangler.rs` lines 93–99: `# signing-secret handle (run wrangler secret put ...)`), never the value. `WebhookBinding::secret_handle` is a `qfs-secrets` handle, never an inline token (`dto.rs` lines 165–175). wasm embeds no token (the scaffold has no secret material at all). Consistent with RFD §10.

**3. At-least-once idempotency — model is sound, with one rigor nuance.** `mock.rs::deliver` drives a `(cause, run_key)` through the `cas`-guarded cursor: read prior → if already at `run_key` short-circuit no-op → else `cas(prior → run_key)`, recording the effect only on a successful swap. The test (`at_least_once_redelivery_is_idempotent`) drives a JOB twice and a WEBHOOK twice (same keys) and asserts exactly 2 committed effects. The `cas` itself is real (`MemDurableStore::cas` compares current vs expect under a `Mutex`). **Nuance**: the `if prior_key.as_deref() == Some(run_key) { return Ok(false) }` short-circuit (lines 121–124) means the redelivery test path returns early *before* exercising the `cas`. The `cas` is what guards a genuine concurrent race, but the test's redelivery is caught by the cheaper pre-check, so the test proves the dedup *result* but not that the `cas` arm itself rejects a redelivery. This is acceptable (the model is correct and the `cas` is independently unit-tested in `daemon.rs::file_store_put_get_roundtrip_and_cas`), but see the proposal. **Proposal**: add a one-line variant to the mock test that drives a redelivery whose `prior` is stale (simulating two deliveries racing from the same prior cursor) so the `cas`-arm's `false` return is directly asserted — closing the gap between "the pre-check dedups" and "the `cas` is the real idempotency primitive under concurrency."

**4. Build profiles — honest, no faked musl.** `[profile.release]` (opt-level 3, thin LTO, codegen-units 1, strip) and `[profile.wasm-release]` (inherits release, opt-level "z", fat lto, strip, panic="abort") are present and correctly scoped. The ADR + `release.sh` + `Cargo.toml` all state plainly that the musl static cross-link is **CI-only** (no x86_64 musl linker on the trip host — the t01/A2 ground truth), and the NATIVE release build is the local proxy. `release.sh` does NOT fake a musl artifact — it builds via the real `cargo build --release --target ...-musl` (which only succeeds with the toolchain CI installs) and, for the parked `qfs.wasm` cdylib, writes a `qfs.wasm.PARKED` note rather than a fake binary (lines 40–47). Honest.

**5. Confinement — green, no spine inversion.** Covered under ruling 3: qfs-host is a leaf consumed only by the binary; the binary allowlist admits it; qfs-host never edges to qfs-runtime; the wasm fence is the absence of the optional `host-daemon` feature.

**6. Parks honest — all recorded, none overclaimed.** The ADR's "Parked (recorded honestly)" section lists the three carry-overs: the `worker`-backed `WorkersHost` (structured-error scaffold), the `qfs.wasm` cdylib (qfs-host is a lib today; the `crate-type=["cdylib"]` Workers entrypoint is parked with `worker`), and the static musl binaries (CI-only). CI (`ci.yml`) builds the wasm-clean core + the parked scaffold on wasm32 (the build the acceptance criterion names), and the `release-artifacts` job is tag-gated and installs the musl toolchain. The systemd unit is a project-local TEMPLATE (not installed system-wide — system-safety honored). Nothing claims the CF host runs end-to-end today.

## Cross-cutting coherence

The t36 layer sits cleanly atop t30 (server registry), t32/t33/t34 (the three fire-path leaves), and t35 (POLICY at fire time). It adds zero grammar keywords, projects the registry into one owned `BindingSet`, and attaches causes through one trait — the "two causes for one plan interpreter" shape RFD §2/§8 describes. The deny-test promotes "vendor-free core" from convention to mechanically-enforced invariant, which strengthens the whole epic, not just t36. The parked-CF posture is consistent with the four prior footprint ADRs, so the carry-over is a known, reviewed pattern rather than a surprise.

## Summary of observations + proposals (all non-blocking)

1. **Daemon `attach` is a ledger-record shim, not the literal owner of listener/interval/bus** — honest for t36; record a carry-over to migrate the actual wiring behind `TokioHost`'s methods so the trait becomes the literal composition root.
2. **`webhook_event` not re-exported at the crate root** like its three archetype siblings — add it to the `host-workers` re-export block (or doc why).
3. **Deny-test is a name-denylist** — add a comment naming the vendor *families* it fences; consider an allowlist-style positive assertion in a later hardening ticket.
4. **MockHost redelivery is caught by the cheap pre-check before the `cas` arm** — add a stale-`prior` variant so the `cas`-as-idempotency-primitive is directly asserted under a simulated race.

None block acceptance. The seam, the daemon reuse, the parked-CF honesty, and the deny-test all hold.
