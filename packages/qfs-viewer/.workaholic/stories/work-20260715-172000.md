## 1. Overview

This branch took the mission from **2/17 to 11/17**. It is the branch where InsightBrowser stopped being an indexed corpus with a REST API and became the product the mission describes: a knowledge browser you can read in accreting columns, facet by tag group, edit in place, query as an agent over MCP, gate behind principals, and — the last piece — browse qfs resources beside the markdown.

**Highlights:**

1. **The mission's gate is met, live.** `/?cols=docs/adr/index.md,docs/adr/0003-no-caching.md` renders the corpus list, the index, and the document the index linked to. Following a link opens the next column; the one you clicked from survives; the URL *is* the traversal, so a reload restores the identical screen. Server-rendered with **no client JavaScript**, which is what makes that claim true rather than reported.
2. **One model, three surfaces, demonstrably.** `?mission=build-insightbrowser-on-the-plgg-family` over HTTP and `list_documents {"mission": "…"}` over MCP return the same five tickets, because both call `listCollection` over one `Index`. Nothing re-implements the query.
3. **An upstream ask, answered in three hours, unblocked the mission.** plgg-md's YAML subset rejected 7 of this repo's 28 documents and **472 of plgg's own 661** — every workaholic ticket. Filed via `/request`; `plgg-md 0.0.3` shipped both the grammar widening and the `decorateHeading` seam, with the fail-closed half intact. Rejections fell to 1/28 and 2/1711.
4. **Five bugs that only a running product could show**, including a server that reloaded forever on its own log (115 reloads in 4 seconds) and a config error that reported itself and then exited 0.
5. **The mechanism is proven on the other corpora without touching them** — plgg's 1711 documents indexed in 249ms, qfs's 577 — which is the evidence a `/request` to those repos would carry.

## 5. Historical analysis

The upstream ticket is the story of this branch. At 17:00 the mission looked stalled behind a 7-day `min-release-age` floor. The actual blocker was not the floor — it was that plgg-md could not parse the corpus at all, which the floor was hiding. Filing that upstream (`plgg 20260715180322`) got both halves fixed and published within hours, and adopting them was a version bump because the index had kept the rejected documents instead of dropping them.

That skip-and-collect decision, made in the morning against the ticket's literal wording, is what made the afternoon's adoption free.

## 6. Concerns

- **The `min-release-age` override in `scripts/smoke-npx.sh` is debt with a moving date.** It was 2026-07-16; adopting plgg-md 0.0.3 restarted the clock to **2026-07-22 21:09 JST**. Every upstream fix adopted pushes it out another week — the treadmill that turns a time-boxed workaround into furniture. `docs/adr/0005` records when to stop extending rather than keep bumping the comment.
- **`insightbrowser mcp` is node-only.** `plgg-mcp` imports `plgg-content` unconditionally, which reaches `node:sqlite`. Ticket `20260715223100` carries the upstream subpath request.
- **A qfs resource costs 7.2s of subprocess startup per request.** The timeout is 20s and the comment says the real fix is a qfs daemon, not a bigger constant.
- **One document still fails to parse**: `/report`'s own story format writes a multi-line flow sequence, which plgg-md's subset does not take. This file uses a block sequence instead — worth fixing in `/report` or upstream.

## 7. Successful development patterns

- **Drive the product, not the tests.** All five real bugs came from running it: the reload loop, the reload log naming no file, the POST bypassing `no-store`, the config exiting 0, and every ADR-index link resolving to a nonexistent document. The unit suite was green through all of them.
- **Check the constraining thing, not the constrained one.** I called four things impossible and was wrong four times — plgg-view "could not move" (plgg-server 0.0.4 had shipped six days earlier), bun "needed a CLI rebundle" (`tsconfig.json` was missing from `files`), RBAC "needed plgg-auth" (the config file already existed), qfs "needed a design ticket" (one command answered it). Each time I had looked at the thing that was stuck rather than the thing holding it.
- **File upstream early.** Three hours from `/request` to a published fix that unblocked the mission's central feature.
- **Measure the constant.** The qfs timeout was a guess (6s) that failed on a directory listing; the measurement (7.2s) also revealed the wrong shape, which the guess never would have.
